What tool use actually is

"Tool use" is a phrase that misleads beginners because it implies the model is reaching out and doing something. It isn't. The model emits a structured tool call — a JSON-shaped payload with a tool name and a dictionary of arguments — as part of its normal token stream. Your code, the harness, sees that call, runs the actual function, and feeds the result back as a tool_result message on the next turn. The model then continues generating, now conditioned on whatever your tool returned. The model decides; your code executes. The pattern is the same across every major SDK — names differ, the shape doesn't.

Three consequences fall out of this that builders frequently miss:

The model doesn't actually call anything — it asks. Nothing in the model's runtime touches your database, your filesystem, or the open internet. The model produces tokens that the harness parses into "I would like you to run screen_sanctions with these arguments." The harness is what calls. This is also where every authorization decision lives: the model can ask for anything; the harness decides what gets executed. Primer 12 will return to this — it is the single highest-leverage place to install guardrails.

The tool's behavior is invisible to the model except through what comes back. The model has no introspection into your code, no sense of whether the tool hit a cache or a cold database, no way to know that one customer ID returned in 40ms and another in 4 seconds. The only signal it gets is the tool_result string. If that string is rich and self-describing, the model can act on it intelligently. If it's a bare ID or an opaque error code, the model is left guessing.

If the tool result is ambiguous, the model will guess. Returning {"status": "ok"} on a screening hit and {"status": "ok"} on a clean record will produce confidently wrong downstream behavior — the model has no choice but to fill in the missing distinction. Make tool output structured and self-describing: include the matched fields, the confidence score, the source list, the timestamp. Verbose, parseable results are cheaper than retries.

"Tool use" is the agent's superpower not because the model gains new capabilities — it is still the same next-token predictor it was a turn ago — but because the model gains eyes (search, retrieval, file reads) and hands (writes, queries, transactions, side-effecting actions) inside your system. Everything else in this course — the agent loop, sub-agents, RAG, evaluation harnesses — depends on getting this primitive right. If the tool call protocol is sloppy at this layer, no amount of orchestration above it will rescue the system.

Defining a tool

A tool definition is small: a name, a description, an input schema. That is the entire surface. Each of those three fields does a different job, and getting the proportions right is most of the craft. The model reads all three of them on every turn, alongside the system prompt — they are prompt material, not configuration. A sloppy tool description is a sloppy prompt, paid for on every call.

The name. Imperative verb, snake_case, narrow scope. screen_sanctions, not do_screening. fetch_customer_record, not customer_api. send_escalation_email, not notify. The model picks tools the way a developer picks library functions — by reading the name first and the description second. A vague name forces the model to lean on the description to disambiguate; a precise name lets the description do narrower work. When you find yourself wanting to suffix a tool with _v2 or _advanced, the underlying name is wrong — rename, don't decorate.

The description. One or two sentences. Three things in priority order: what it does, when to use it, and any constraints worth flagging (cost, latency, side effects, prerequisites). "Screen a customer against a sanctions list. Returns hits with match-confidence scores. Use for any new customer or any payment above the screening threshold." That is enough. What the description should not contain: instructions about other tools, lore about the system, motivation, or hedges. The model needs a contract, not a paragraph.

The schema. JSON Schema is the lingua franca — every SDK accepts a subset of it. Five things matter: the property types, a one-line description on every property, an enum wherever the values are bounded, a required list, and the avoidance of free-form blobs. If a property could in principle hold "anything," that property is a future bug report. Either narrow it or split it into the cases you actually care about.

PYTHON 
tools = [{
    "name": "screen_sanctions",
    "description": "Screen a customer against a sanctions list. Returns hits with match-confidence scores. Use for any new customer or any payment above the screening threshold.",
    "input_schema": {
        "type": "object",
        "properties": {
            "customer_id": {
                "type": "string",
                "description": "The internal customer identifier."
            },
            "list": {
                "type": "string",
                "enum": ["OFAC_SDN", "UN", "EU", "HMT", "ALL"],
                "description": "Which list to screen against. Use ALL for new customers."
            }
        },
        "required": ["customer_id", "list"]
    }
}]

Notice the shape. The name is a verb. The description tells the model when to reach for this tool, not just what it does. Every property has a description. The list field is an enum, not a string — there is no universe in which "OFAC" and "OFAC_SDN" should both be accepted, and the enum makes the model's choice deterministic. Both properties are required: there is no sensible default for either, and "optional with a default" is how you end up screening against the wrong list on the second Tuesday of the month.

A tool definition that looks like this on day one keeps working on day ninety. A tool definition with three free-form strings and no descriptions starts failing the moment the agent encounters a customer record it hasn't seen before. The schema is where reliability is bought.

Schema discipline

If §2 was the shape of a good tool, this section is the disciplined habit that produces one. Of every craft technique in this course, schema tightening has the highest leverage-to-effort ratio. Ten minutes per tool, paid once, removes an entire category of failures that would otherwise show up as flaky behavior in production and burn days of debugging.

Four common sins account for most schema-induced failures:

  • Missing description on properties. The model has to guess what the field means. Sometimes it guesses right. The cases where it guesses wrong are the ones you remember.
  • Free-form string where an enum belongs. Anywhere the valid values are bounded — risk tiers, country codes, status names, severity levels — an enum makes the model's output predictable. A string field invites the model to compose something plausible and wrong.
  • Optional everything. If nothing is required, the model will omit half the fields on half the calls, especially the ones it considers "obvious." Required fields force the model to fill them; their presence in the schema is also a signal to the model that they are load-bearing.
  • Nested objects with no descriptions. A bare type: object blob tells the model "put a dictionary here" and nothing else. Sub-schemas with their own property descriptions turn the nested shape from a hint into a contract.

Show the same field in both registers and the difference becomes obvious. The same model, given a tight schema, will produce parseable output on inputs it would have fumbled with a loose one.

PatternReliableBrittle
Bounded valuesenum: ["LOW", "MID", "HIGH"]type: string with hopes
Required keysExplicitly listed in requiredAll optional, model guesses
Property descriptionsOne sentence per propertyNone — model infers intent
Nested shapesSub-schema with own descriptionstype: object blob

A working heuristic: read your schema aloud to a colleague who has never seen the tool. If, for any property, they would ask "what does that mean?" or "what values can it take?" — the schema is too loose. Add the description; add the enum. The five minutes it costs you pays back the first time the model encounters a customer whose record uses a slightly different vocabulary than the one in your prompt.

One more piece of discipline: avoid optional fields with defaults baked into the schema. If a field has a sensible default, set the default in your tool implementation, not in the schema. The schema is what the model sees; the implementation is what the harness runs. Putting defaults in the schema teaches the model it can skip the field, which it then does in cases where you really wanted it to think about the value.

A tight schema is the difference between a tool that works 95% of the time and one that works 75%. Every property description is a prompt, paid for in tokens once per turn and earning its keep on every call. Treat schemas with the same care you give prose — they are prose, just typed.

Structured output: schemas as guardrails

The JSON Schema mechanism you learned for tool inputs has a second, equally important use: constraining the model's output when you want a parseable response rather than free text. Anywhere you would have written a regex to extract a field from a model's prose, the schema does it cleanly and at the source. The model emits structured output directly; your code parses it as a typed object.

There are two ways to get this in practice. The first is the one you can use today on any model with tool calling: force a tool call by setting tool_choice to a specific tool name. The model must emit a call matching that tool's input schema, even if you have no intention of actually "using" the tool. You define a tool called return_classification with the schema of the output you want, force the model to call it, and read the arguments. It feels like a hack — it is a hack — but it is the most reliable structured-output mechanism available across providers today.

The second is a model-native structured-output feature. Several providers now expose first-class JSON-schema-constrained generation that doesn't pretend to be a tool call. The output is cleaner — no fake tool wrapper — but the feature is newer, the schema-coverage is narrower, and the edge cases are still being shaken out. For production work today, forced tool use is the conservative choice; native structured output is the direction to migrate toward as the feature matures across the providers you target.

The trade-off is real. Forced tool calls always work, but the output is tool-shaped: you parse block.input rather than a clean response object. Native structured output is cleaner code, but you accept whatever the provider's schema support actually covers — which today is "most things, sometimes." Pick the option whose failure mode you can live with, and write the wrapper that lets you swap them out without rewriting the call sites.

RELIABILITY

Constrain the output, then constrain less.

Start with a tight schema and a forced tool. Once your eval suite shows reliable parsing across 50+ varied inputs, you can experiment with loosening — letting the model choose tools, falling back to free text when no tool fits. Loosen one constraint at a time; eval after each step.

Validation, retries, and the parse loop

Even with a tight schema and a forced tool, the model will occasionally hand you something that doesn't validate. An enum value just outside the allowed set. A nested object missing a required key. A confidence score expressed as the string "high" when the schema asked for a float between 0 and 1. These are rare in good prompts and unavoidable at scale, and the difference between an agent that survives them and one that doesn't is a small amount of harness code applied consistently.

The pattern is three steps:

Validate the result against the schema before treating it as truth. Pydantic, a JSON Schema validator, hand-written type guards — pick a tool and use it on every tool call's arguments. A successfully parsed tool call is not a validated one; the SDK only checks the shape, not the semantics. list: "ALL" is a valid string, but if your real list of accepted values is ["OFAC_SDN", "UN", "EU", "HMT", "ALL"], only the validator will catch the typo before it hits the database.

On validation failure, feed the error back to the model as a tool_result with is_error: true and the validation message as the content. The model treats this as exactly what it is: a signal that its previous output did not satisfy the contract, with a specific reason. In practice the model corrects itself on the next turn for the vast majority of these cases. Two retries is enough — any more and you are papering over a schema problem you should fix at its source.

After 2 failed retries, log and escalate. Do not fall back to a default value or a "best effort" guess. The model has demonstrated, three times, that it does not know what the right answer looks like. Quietly substituting a default turns a visible failure into a silent one, which is the worst class of bug in production agents. Surface the escalation, capture the trace, and look at it during your next eval review.

PYTHON 
from anthropic import Anthropic
import json

MAX_RETRIES = 2

def call_with_validation(client, system, user_msg, tools, validator):
    history = [{"role": "user", "content": user_msg}]
    for attempt in range(MAX_RETRIES + 1):
        resp = client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=1024,
            system=system,
            tools=tools,
            tool_choice={"type": "tool", "name": tools[0]["name"]},
            messages=history,
        )
        for block in resp.content:
            if block.type == "tool_use":
                try:
                    return validator(block.input)
                except ValueError as e:
                    history.append({"role": "assistant", "content": resp.content})
                    history.append({"role": "user", "content": [{
                        "type": "tool_result",
                        "tool_use_id": block.id,
                        "content": f"Validation failed: {e}",
                        "is_error": True,
                    }]})
                    break
    raise RuntimeError(f"Validation failed after {MAX_RETRIES} retries")

The shape is small, deliberately. MAX_RETRIES = 2 is a constant, not a parameter — when you find yourself wanting to raise it, the right move is to revisit the schema, not the retry budget. The error message gets propagated verbatim because the model is good at correcting against a specific complaint and bad at correcting against "something is wrong." The final raise is the escalation; this is where your tracing and alerting should hook in, so the failure shows up where humans will see it.

When to force a tool vs let the model choose

The tool_choice parameter is small — three modes — and the right setting is decided by what the agent is being asked to do, not by personal preference. Most production reliability problems on tool-using agents come down to either picking the wrong mode for the task, or leaving the choice unset and letting the model drift between modes turn-to-turn.

Forced tool (tool_choice: {type: "tool", name: "X"}) — the model must emit a call to tool X. Use it whenever you know exactly what shape the next response should take: a single-step classification, a structured extraction from a document, a "return-this-shape" pseudo-tool you defined just to constrain the output. This is the safest mode for a non-interactive pipeline step.

Any tool (tool_choice: {type: "any"}) — the model must call some tool, but it picks which one. Use it for agents whose job is to act, not chat: the user has asked for something to happen, and a text-only response would mean the agent gave up. The model still has to choose well, but it cannot escape into "let me explain why I won't do this."

Auto (tool_choice: "auto" or unset) — the model decides whether to call a tool at all, and which one. Use it for conversational agents that mix dialogue and action: the user might be asking a question, asking for help, or asking the agent to do something concrete, and the model needs the discretion to pick.

tool_choiceUse forFailure mode
forced toolSingle-step extraction, structured outputWrong tool used as a hammer
anyMulti-step agents that must act, not chatModel picks the wrong tool
autoConversational mixed text + toolsRefuses to call when it should

The default for new agent code should be more constrained than feels comfortable. Start with forced tool for any step where you know what shape you want; move to any when the agent has multiple legitimate next steps; reach for auto only when the agent must hold a conversation. It is far easier to loosen a constraint after the eval suite proves the next step is safe than to chase down a phantom "the model just stopped calling the tool" bug in week three.

What this connects to

Tool use and structured output is the primitive every later module builds on. The pointers worth carrying forward:

  • Primer 2 — Prompting & Context Engineering — tool descriptions and property descriptions are prompts. Every word in them is read every turn, and the same discipline that produces a good system prompt produces a good tool description: imperative, specific, no lore.
  • Primer 5 — The Agent Loop — tools are how the agent's loop iterates. A loop without tools is a chat; a loop with tools is an agent. This primer is the static view of a single tool call; Primer 5 is the dynamic view of many of them composed in sequence.
  • Primer 9 — Sub-Agents, Skills & Harnesses — when one tool isn't enough, you compose tools into skills, and skills into sub-agents. The schemas you wrote here are the contracts that hold those compositions together.
  • Primer 12 — Safety, Guardrails & Prompt Injection — tool authorization is where most agent security incidents live. The harness decides what gets executed; that decision is your last clean line of defense.

Tight schemas, forced tool choice where you know the shape, and a small parse-retry loop on every call together solve perhaps 95% of agent reliability problems before they appear. Build them in from the first tool, not after the first incident. The cost is small and front-loaded; the benefit compounds every turn the agent runs.
