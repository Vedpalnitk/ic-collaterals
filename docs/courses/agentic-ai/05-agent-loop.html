The Agent Loop

A while-loop around an LLM, and what makes that simple idea harden into something real.

What an agent loop is

An agent loop is the smallest piece of code that turns a single-shot LLM call into something that can take action. It is, almost literally, a while loop. On each iteration, the loop (a) presents the model with the current conversation state and the list of tools it may call, (b) parses the model's response for any tool calls, (c) executes those tools, (d) appends the tool results to the message history, and (e) repeats — until the model emits a terminal response with no tool calls.

Two roles fall out of that structure, and they are the most important distinction in the whole primer. The model is the decider. It looks at the state, picks the next action, and emits a structured call. The harness — your code — is the executor. It runs the tool, captures the result, and feeds it back in. The model never touches a network socket, a file, or a database. The harness never decides what to do next. Each one does the thing it is good at, and they take turns.

That clean division is the reason the canonical loop is the right default. It is small enough to fit on one page, debuggable end-to-end, and survives most real-world tasks long before you need anything fancier. When people show you a complex multi-agent diagram, ask where the decider/executor split lives. If the answer is fuzzy, the system probably has a bug it has not noticed yet.

Mechanically, here is what one iteration looks like in pseudocode:

PSEUDOCODE 
messages = [system_prompt, user_request]
while True:
    response = model.call(messages, tools=TOOLS)
    messages.append(response)            # assistant turn
    tool_calls = extract_tool_calls(response)
    if not tool_calls:
        return response                  # terminal
    for call in tool_calls:
        result = TOOLS[call.name](**call.input)
        messages.append(tool_result(call.id, result))

Everything else in this primer is a refinement of that ten-line shape. Termination conditions, history management, error handling, failure modes — they are all answers to questions that arise the first time you actually run this loop against a real task. You will not appreciate why the answers matter until you have felt the loop misbehave at least once. Build it. Watch it fail. Then come back.

The canonical tick

One tick of the loop is five small acts, in order. They blur together in production logs, so it is worth slowing them down once. The animation below holds each act for about a second and a half, then cycles. Pause it whenever you want to dwell.

READ. The harness assembles the current state — system prompt, tool definitions, the running message history — and hands it to the model. Nothing fancy; just a list of messages and a list of tools.

DECIDE. The model reads the state and produces a response. That response is a sequence of content blocks: maybe some thinking, maybe some natural-language text, maybe one or more tool calls. The model is not doing anything yet — it is just emitting structured output.

EMIT. Inside that response, the harness finds the tool calls. Each one is a small object: the tool's name and a JSON dict of arguments. If there are no tool calls, the loop ends here and the response is returned. If there are, the harness validates the arguments against the tool's schema and prepares to execute.

EXECUTE. The harness runs the tool. This is where the system actually touches the world — an API request goes out, a database query runs, a file is read. The result comes back as text, JSON, or a stream of bytes the harness has to serialize back into something the model can read.

APPEND. The tool's result is wrapped in a tool_result message and appended to the history. The loop iterates: the next READ sees a longer history, with the model's last tool call and the result it produced, and the model picks its next action with full knowledge of what just happened.

That is the whole mechanic. Read, decide, emit, execute, append. Watch a few ticks pass below before moving on.

Tool definitions and parsing

A tool, from the model's point of view, is three pieces of metadata: a name, a description (in natural language, written for the model to read), and an input schema (JSON Schema describing the arguments). You send the full list of tools on every turn. The model returns structured tool-call blocks; your harness pulls them out and executes them.

Here is the shape with the Anthropic SDK. The pattern is essentially identical across providers — different field names, same idea.

PYTHON 
from anthropic import Anthropic

tools = [{
    "name": "get_weather",
    "description": "Get current weather for a city.",
    "input_schema": {
        "type": "object",
        "properties": {"city": {"type": "string"}},
        "required": ["city"],
    },
}]

resp = Anthropic().messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    tools=tools,
    messages=[{"role": "user", "content": "What's the weather in Bengaluru?"}],
)

for block in resp.content:
    if block.type == "tool_use":
        print(block.name, block.input)
        # → get_weather {'city': 'Bengaluru'}

Two responsibilities the harness owns and the model does not.

Schema validation. Providers do a reasonable job of constraining the model's output to match the schema, but they do not guarantee it. The model can still emit a call with a missing required field, a string where a number was expected, or an enum value you did not list. Validate every tool call against the schema before you execute it. When validation fails, the right move is almost never to crash — instead, append a tool-result message that explains the failure to the model in plain text ("city is required") and let it try again. The model is surprisingly good at fixing its own malformed calls when you tell it what was wrong.

Actually running the tool. The model produces the call; your code runs it. That sounds trivial until you realize how much hides in "runs it." A real tool runner needs a timeout (so a hung API does not freeze the loop), a serializer (the model can only consume text, so binary results need to be encoded), a sandbox boundary when the tool can touch the user's machine, and a retry policy for transient failures. None of this is the model's concern. All of it is yours.

Treat the tool layer like any other I/O boundary in your system. Every call has an input contract and an output contract. Every failure has a defined mode. The model only sees the strict, structured surface; the messy reality lives behind the harness.

Termination

A loop without a defined exit is a bug. The canonical loop has three reasonable termination conditions, and a well-built harness checks all of them.

1. The model emits no tool calls. This is the natural endpoint. The model has decided it is done — it has gathered what it needs and produced a final answer. Return that answer to the caller and exit. This is the case the loop is designed to reach.

2. The max-iteration safeguard fires. You count iterations and stop when you hit the cap. Twenty iterations is a reasonable default for most tasks; some need more, very few legitimately need fewer. The cap is not a feature — it is a circuit breaker. It exists so a misbehaving loop costs you bounded dollars instead of unbounded ones.

3. The model emits an explicit "finish" signal. Some harnesses give the model a tool called finish or respond_to_user whose only job is to signal completion (and optionally carry the final answer as its argument). This is a stronger guarantee than "no tool calls" — it forces the model to make termination an explicit, structured decision rather than an implicit one. Worth considering when the cost of running an extra turn is high or when you want a clean audit trail of why each loop ended.

You also want to detect a fourth case, which is not termination but the thing that makes you wish for it: the stuck loop. The model keeps calling tools but is making no progress — same call, same result, same call again. The simplest detector is a small ring buffer of the last N tool calls; if you see exact-repeat calls more than two or three times, log it, surface it to the operator, and either break the loop or inject a system message telling the model to summarize and finish.

SAFETY

Always cap iterations.

A loop without a cap is a loop that will sometimes spin forever — running up cost, holding open tool connections, and producing nothing. 20 iterations is a reasonable default for most tasks. If you legitimately need more, surface it as a config knob, not a removed cap.

One nuance worth flagging. The model's "I'm done" signal and your max-iteration cap are not interchangeable. The first is the model declaring success; the second is your harness preventing damage. When the cap fires, you are not done — you have run out of patience. Your application code should treat these two exits differently, even if the user-facing message is the same.

State and history

The state of the loop is the message history. Every assistant response, every tool call, every tool result, in order. That list is the only thing the model sees on each iteration — it has no memory between calls, no hidden scratchpad, no implicit context. If something is not in the history, the model does not know it happened.

This is clean but expensive. The history grows linearly with every turn, and you re-send all of it on every call. By turn 10 of a session, you are paying for the system prompt, the tool definitions, and nine previous turns of conversation each time the loop ticks. Two pressures bite at once.

Cost. Input tokens add up. A 20K-token prompt running ten turns is a quarter of a million input tokens — without caching, that is real money per session. Prompt caching (see Primer 1) flattens most of this curve, but only for the stable prefix. The growing history part still costs full price.

Signal-to-noise. Even when cost is fine, an overlong history hurts performance. Old tool results from two tasks ago compete for the model's attention with what you actually need it to do now. The model can lose the thread, mis-remember earlier facts, or fixate on stale context. More history is not always more grounding.

You have four serious strategies for history management. They sit on a spectrum from "keep everything" to "keep almost nothing," and you usually compose them rather than pick one.

  • Full history. Fine for short sessions (≤5 turns) or when every detail might matter. The default until proven otherwise.
  • Sliding window. Keep the last N turns; drop older ones. Cheap and simple. Works for tasks where local context is what matters and the goal lives in the current few turns.
  • Summarized prefix + recent window. Run a periodic summarization pass over the older turns and replace them with a compact summary. The recent window stays verbatim. This is the most common production pattern.
  • Tool-result compression. Replace verbose tool outputs (a 5,000-token API response, a giant file dump) with a structured summary the moment they arrive. Often the highest-leverage move because tool results dominate token counts in tool-heavy agents.

Compression has a failure mode worth naming: the summary throws away the exact piece of information the model needed three turns later. Hedge by keeping recent turns verbatim, by summarizing in a structured (rather than prose) format, and by giving the model a way to ask for the original — for example, a recall_full_result tool that fetches the unsummarized version from a side store.

For the deeper memory treatment — persistent state across sessions, vector recall, structured agent memory — see Primer 6 — Memory & State. The strategies here cover within-session history; that primer covers cross-session memory.

Failure modes

The canonical loop is robust enough to be a default, but every team that runs one in production trips on a small set of recurring failures. The patterns are recognisable. Naming them is half the cure.

Loop won't terminate

The model keeps calling tools, often the same one with slightly different arguments, and never emits a terminal response. You watch your iteration counter climb toward the cap.

The common cause is an ambiguous tool output. The tool returns something the model cannot interpret as "this is the answer" — say, a paginated API response without a clear end-of-results marker, or an error message that looks like it might be a transient problem worth retrying. The model retries. The tool returns the same ambiguous thing. The cycle continues.

The fix lives in the tool, not the prompt. Give every tool output a structured shape that says, in effect, "here is the data, here is the status, here is whether you should call me again." A status: complete field, an explicit more_results: false, a clean error code. The model is good at following these signals when they exist and bad at guessing when they do not.

Model won't use tools

Tools are defined and available; the model answers from its own knowledge anyway. Often the answer is plausible-looking but wrong, because the model is hallucinating instead of calling the lookup tool you wrote specifically to prevent hallucination.

Two causes. First, weak prompting — your system prompt does not make it clear that tool use is expected, so the model treats tools as optional sugar rather than the right path. Second, overlap with prior knowledge — the model "knows" Bengaluru's typical weather, so it answers from memory rather than calling get_weather. The second case is more insidious because it works most of the time, until the day the model's memory is stale or wrong and there is no fallback.

Two fixes. Be explicit in the system prompt: "Always use the available tools to look up real-time information; do not answer from prior knowledge." If that is not enough, use the provider's tool_choice control — set it to "any" or "required" for the turns where you need tool use to happen no matter what.

Cascade failures

A tool errors out. The harness raises an exception. The loop dies. Or worse, the harness catches the exception, returns a malformed tool result, and the model spirals into a recovery loop calling the broken tool over and over.

The fix is a single principle that takes a few iterations to internalize: tool errors are messages to the model, not exceptions in your program. When a tool fails — schema validation, timeout, API 500, anything — wrap the error in a structured tool_result with is_error: true and a human-readable message, then append it to the history exactly like a successful result. The model sees the failure, reasons about it, and usually picks a different action.

Exceptions should be reserved for things the model genuinely cannot recover from — the harness itself misconfigured, the LLM provider down, the iteration cap hit. Everything in between flows through the message history. Once you make this distinction sharp, a whole category of cascade failures disappears.

When to outgrow the canonical loop

The canonical loop is the default for a reason: it is simple, observable, and sufficient for most agent tasks. But it has a shape, and some tasks have a different shape. Three signals tell you it is time to reach for something else.

The task wants parallel exploration. If the work decomposes into branches that can be explored independently — three candidate database queries, five competing hypotheses, a research task with multiple sources to investigate at once — a single sequential loop turns that natural parallelism into wasted wall-clock time. This is the canonical case for multi-agent, where a coordinator spins up sibling agents that work in parallel and aggregate their findings. See Primer 8 — Multi-Agent Patterns.

The task has a clear plan-then-execute structure. Some tasks benefit from explicitly separating planning from execution — first lay out the full sequence of steps, then run them in order. A canonical loop blurs this distinction; the model plans implicitly, one step at a time, with no separate artifact for the plan as a whole. When you need plan visibility, plan editing, or human approval of the plan before any action runs, you want a planner + executor pattern. See Primer 7 — Planning & Decomposition.

One agent cannot hold all the context. Long tasks blow through the context window. Complex tasks blow through the model's ability to attend to everything at once. The fix is sub-agents: spawn a child agent with its own clean context window for a focused sub-task, then return only the distilled result to the parent. The parent keeps a tidy history; the child does the heavy lifting and disappears. See Primer 9 — Sub-Agents, Skills, Harnesses.

Notice what is not on this list: "we want to feel sophisticated." The single most common bad call in agentic systems is reaching for multi-agent or planner architectures before the canonical loop has actually failed. Multi-agent multiplies the number of debugging surfaces by N. Planners introduce a new failure mode where the plan is wrong before the first action runs. Sub-agents make state harder to reason about.

DEFAULT

Outgrow the loop only after it has hurt you.

Premature multi-agent is the most common bad call in agentic systems. If you cannot point at a specific failure your canonical loop is committing — wasted parallelism, opaque planning, context exhaustion — the right answer is to stay with the canonical loop and add better tools, better caching, or better termination. Reach for something fancier when the simple shape demonstrably is not enough.

The canonical loop will get you further than you expect. Build one. Run it against a real task. Watch where it strains. When it strains in a way that matches one of the three signals above, then — and only then — pick the right next pattern from the primers that follow.
