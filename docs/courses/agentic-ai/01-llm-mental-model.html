The model as a sampler

An LLM is a function. It takes a sequence of tokens as input and returns a probability distribution over the next token in the vocabulary. That is the whole interface. Everything you have ever seen a model do — answer a question, write code, plan a trip, refuse a request — is the output of repeatedly sampling from that distribution and feeding the sampled token back in.

This framing matters because most builder mistakes come from thinking of the model as a database, an oracle, or a person. It is none of those. It is a sampler over a learned distribution. Holding that picture in mind changes how you debug.

Three consequences fall out directly.

Same prompt, different answers. If the temperature is above zero, the model picks among likely next tokens stochastically. Run the same prompt twice and you will get two different trajectories. This is not a bug in your code. It is the system working as designed. If you want determinism, you have to ask for it — temperature near zero, sometimes a fixed seed where the provider exposes one — and even then "deterministic" means "very nearly the same," not "byte-identical across runs."

Self-consistency is cheap and effective. Because every generation is a sample, you can run the same prompt N times and take the majority answer. This works surprisingly well for tasks with a clean "right answer" — arithmetic with a calculator tool, multiple-choice, structured extraction. It is the simplest reliability technique you have. The cost is N× the tokens; the payoff is fewer failures from single bad samples. Use it when a task is important and the answer space is narrow.

Hallucinations are samples, not bugs. When a model invents a function name, a citation, or a fact, it is not malfunctioning. It is doing exactly what it was trained to do: sampling a plausible continuation from its learned distribution. The continuation just happens to be wrong. You cannot debug this by adding more instructions to "be accurate." You fix it by narrowing the distribution the model is sampling from — give it the document with the answer (retrieval), force it through a schema (structured output), or constrain it to a tool call. Better prompts help at the margin. Better inputs help more.

Tokens, not words

The model does not see words. It sees tokens — integer ids drawn from a fixed vocabulary, typically 50K to 200K entries. Tokenization happens before the model and is shared with the cost meter: you pay per token of input and per token of output, not per word, character, or message.

Modern tokenizers use a flavor of byte-pair encoding (BPE). The vocabulary is learned from a large training corpus by repeatedly merging the most frequent adjacent byte pairs into new tokens. Frequent strings end up as a single token; rare strings get broken into several. The word the is one token. A long medical term might be four. An emoji might be three. A piece of base64 might be one token per character.

The practical rule of thumb for English prose is the only number you need to carry around in your head:

~4 chars/token
ENGLISH PROSE, GPT-CLASS TOKENIZER

So 1,000 tokens is roughly 750 words, or two pages of prose. A 200K-token context window holds about 150,000 words — a long novel. This ratio is stable enough to plan with.

It is also fragile in predictable ways. The same idea written as JSON or code costs more than as prose. Punctuation, indentation, and quoted keys all eat tokens that an English sentence would compress past. As a rough multiplier, structured formats run 2–3× more tokens per unit of meaning. A 500-word API response shaped as nested JSON might tokenize like 1,500 words of prose.

Non-Latin scripts are worse. Older tokenizers were trained primarily on English and assign one token per character (or worse, per byte) for Hindi, Arabic, Chinese, Korean, and Thai. The same paragraph can cost 3–5× as many tokens in those languages as in English. Newer tokenizers narrow the gap but do not close it. If your product serves multilingual users, model this in your cost forecast — it is not a rounding error.

When you care about the exact count, ask the provider. Every major SDK ships a token-counting endpoint that uses the same tokenizer the model uses, which means it agrees with what you will be billed for.

PYTHON 
from anthropic import Anthropic
client = Anthropic()
client.messages.count_tokens(
    model="claude-sonnet-4-6",
    messages=[{"role": "user", "content": "Hello world"}],
)
# → InputTokens(input_tokens=10)

Get into the habit of counting before you ship. A prompt that "looks short" can hide a 2,000-token few-shot example. A "small" JSON payload can be the largest line item in your bill. Tokens are the unit of cost and the unit of context — measure them, do not estimate from word counts.

The context window is a budget

Every model has a maximum number of tokens it can attend to in a single call. This is the context window. Frontier models today sit at 200K to 1M tokens. That sounds enormous. It runs out faster than you think.

The window is a shared budget. Everything you send draws from the same pool:

  • System prompt — instructions, role, formatting rules. Often 500–5,000 tokens once you take it seriously.
  • Tool definitions — names, descriptions, JSON schemas for every tool the model can call. A useful agent with 8–12 tools often spends 3,000–8,000 tokens here alone.
  • Few-shot examples — sample inputs and expected outputs included in the prompt. Easy to grow without noticing.
  • Retrieved context — documents, code, search results pulled in at runtime. Bounded only by what you put a limit on.
  • Conversation history — every prior user message and every prior assistant response, replayed in full on every turn.
  • The current user message — usually the smallest item on this list.
  • Output — the response also lives in the window, and providers cap output tokens separately.

For a multi-turn agent, the dominant term is usually the system prompt + tool definitions + accumulated history. On turn 10 of a session, you are re-sending the same system prompt, the same tool defs, and nine prior turns of conversation — and paying for all of them, every time.

This is where prompt caching earns its keep. Both Anthropic and most major providers let you mark a prefix of the prompt as cacheable. The next request that shares that prefix reads the cached tokens at a steep discount instead of re-processing them. The exact mechanics vary, but the shape is consistent: cached input is much cheaper than fresh input, and the savings compound across turns.

CACHING

Cache the parts that don't change.

System prompts, tool definitions, retrieved docs that repeat across turns — all of these should be cache-marked. With Anthropic's prompt caching, cached tokens cost ~10% of normal at read time, with a 5-minute TTL. For agents that loop, this is a 5–10× cost reduction.

Two principles fall out of the budget framing. First, order things from stable to volatile. Put the static system prompt and tool definitions at the top; put the volatile retrieved context and the current user message at the bottom. Caches key off a prefix match, so any change near the top invalidates everything after it. Second, budget the response. If you have a hard 1,000-token output limit, set it; do not leave it open. A model that has room to write 4,000 tokens often will, even when 200 would do.

Treat the window like a memory budget in an embedded system. You measure, you cap, you cache, and you do not assume the next model will give you more headroom than you have today.

Sampling: temperature, top-p, schema

The model emits a probability distribution. The sampler decides which token to actually pick. That choice is governed by three knobs that you, the caller, control.

Temperature scales the distribution before sampling. At temperature 1.0, you sample from the model's raw distribution. Above 1.0, the distribution flattens — unlikely tokens get more chance, and outputs feel "creative" and unstable. Below 1.0, the distribution sharpens, and the most likely token becomes overwhelmingly likely. At temperature 0, you take the argmax — the single highest-probability token, every time.

Temperature 0 is not the same as deterministic. Floating-point ties get broken differently across hardware, batching can shuffle results, and providers sometimes inject small amounts of noise. Treat temperature 0 as "very nearly the same answer," not as "byte-identical." If you need true reproducibility, log inputs and outputs and verify, do not trust the seed.

Top-p (also called nucleus sampling) is an alternative truncation. Instead of scaling the whole distribution, you keep only the smallest set of tokens whose cumulative probability adds up to p (e.g. 0.9), and sample from that set. The benefit: long-tail garbage tokens are dropped no matter what the temperature is. The cost: another knob to reason about. Most builders pick temperature or top-p, not both.

Structured output is the strongest constraint of the three, and it is in a different category. Instead of nudging the distribution, you tell the model that the next tokens must match a schema — a JSON object with these fields and types, or a specific tool call with these arguments. Providers implement this by masking the distribution at each step to only include tokens consistent with the schema. The result is reliable, parseable output without prompt-engineering acrobatics.

The decision rule for builders is short:

SettingUse forFailure mode
Temp 0.0Structured extraction, classificationRepetitive on long generations
Temp 0.7Drafting, summaries, proseOccasional drift
Forced schemaAnything you'll JSON-parseSchema validation failures (retry)

The shorter rule: free text for prose, schema for anything you will parse. If your downstream code does json.loads on the response, you should be using structured output. The cost of a regex over a freeform reply is a class of bugs you do not need.

The cost/latency frontier

Per-call cost is straightforward. You pay for input tokens at one rate and for output tokens at a higher rate. Frontier-class models today price output tokens at roughly 5× input. Cost per call is:

cost = input_tokens × in_rate + output_tokens × out_rate

For an agent, multiply that by the number of turns in a session, and multiply that by the number of sessions per day. Three multipliers stack: tokens, turns, traffic. Each one is easy to underestimate by 2–3×.

Work an example. Use round, illustrative numbers — Sonnet-class pricing in the neighborhood of $3 per million input tokens and $15 per million output tokens. (Check current rates; the point here is the ratio and where the leverage lives, not the exact dollars.)

An agent has a 20,000-token system prompt and tool definitions. Each turn, the user sends a 1,000-token message and the model emits a 500-token response. The loop runs 10 turns per session.

On turn 1, the model sees 21,000 input tokens (system + tools + user) and emits 500 output tokens. On turn 2, it sees 22,500 (the prior turn's user + assistant are now history) and emits another 500. By turn 10, the input has grown to roughly 33,000 tokens. The session total is approximately 275,000 input tokens and 5,000 output tokens.

At illustrative rates: 275,000 × $3/M + 5,000 × $15/M = $0.825 + $0.075 ≈ $0.90 per session. Run that 10,000 times a day and you are at ~$9,000/day, ~$270,000/month.

Now turn caching on. The first 20,000 tokens of every turn — the system prompt and tool definitions — are identical. You mark them cacheable. After the first turn warms the cache, subsequent turns within the 5-minute TTL pay for those tokens at roughly 10% of the normal rate.

The 275,000 input tokens split into roughly 200,000 cached and 75,000 fresh. Cost becomes 200,000 × $0.30/M + 75,000 × $3/M + 5,000 × $15/M = $0.06 + $0.225 + $0.075 ≈ $0.36 per session. The same 10,000 sessions/day now costs ~$3,600/day, ~$108,000/month. Same agent, same behavior, ~60% cheaper. In production agents the gap is often wider.

In agents, the system prompt is read once per turn but rarely changes. Cache it.

Latency is a separate axis with its own levers. Time-to-first-token is dominated by input size — bigger prompts mean longer prefill. Total response time is dominated by output size — every output token is generated serially. Caching helps prefill (cached tokens skip most of the work) but does nothing for output speed.

The cheapest perceived-latency win is streaming. Instead of waiting for the full response, stream tokens to the user as they generate. A 2-second response that starts producing at 300ms feels dramatically faster than a 2-second response that arrives all at once. Use streaming wherever a human is reading the output. Skip it when a downstream system is parsing structured output and needs the full payload anyway.

What models can and can't do

An LLM is shockingly good at some things and reliably bad at others. The pattern is not random. It follows from what the model is: a high-capacity pattern-matcher trained on text, with no internal memory between calls and no native ability to do arithmetic, run code, or fetch live data. Anything that fits those constraints, it does well. Anything that breaks them, it does poorly until you give it a tool.

Strong at

  • Language: paraphrase, summarize, translate, style transfer
  • Structure: extract, classify, route, format
  • Short-horizon planning and tool selection
  • Generating code in well-represented languages

Weak at

  • Arithmetic past a few digits (use a tool)
  • Long-horizon consistency across many turns
  • Knowledge after the cutoff date
  • Tasks requiring persistent state without memory

The strengths share a shape. They are all transformations of text where the answer is a function of the input plus learned conventions. Summarization, translation, classification, extraction, routing — all of these have well-defined inputs and outputs and a large training corpus showing the model what good looks like. Short-horizon planning is in this category too: "given this user request and these tools, which tool do I call next?" is a one-step decision the model is very good at.

The weaknesses share an opposite shape. They are tasks where the answer requires either a precise mechanism the model does not have (multi-digit arithmetic, exact lookups, deterministic execution) or state that lives across many steps (long-horizon goals, multi-turn consistency, knowledge the training run never saw). The model will happily produce confident-looking output for these tasks. The output will often be wrong.

The fix in every case is the same: do not ask the model to do the thing it cannot do. Give it a tool that can. Calculators for arithmetic. Code execution for anything that needs to be precise. Retrieval for knowledge after the cutoff. External memory — a scratchpad, a database, a vector store — for state that has to persist. The model's job is to decide which tool to call and with what arguments, not to be the calculator itself.

One more failure mode worth naming: long-horizon consistency. On a single short generation, the model holds the thread. Across 50 turns of a conversation, or 30 steps of a plan, it drifts. It forgets constraints it agreed to earlier. It contradicts itself. It loses the goal. This is not a flaw you can prompt away. It is the cost of having all working memory live in the context window, where it competes with everything else for space.

This is the bridge to the rest of the course. A single model call is short-horizon by nature. An agent loop is the structure you build around a model to give it long-horizon behavior — letting a strong-at-short-horizons model accumulate work across many turns, with tools to handle what it cannot, and external memory to remember what the window cannot hold. That is where we go next.
