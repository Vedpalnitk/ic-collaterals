The two memories

Talk about agent memory long enough and the word starts doing too much work. "The agent remembers" can mean four different things to four different people. Before any of the engineering makes sense, pull the concept apart into the two pieces that actually matter.

Short-term memory is the message stack. It is everything in the current context window: the system prompt, the tool definitions, the conversation history so far, the retrieved passages, the current user message. It is what the model sees on this call. It lives in RAM, exists only for the duration of the session, and disappears when the process ends. It is what Primer 5 called the loop's state — every turn reads it, every turn writes to it.

Long-term memory is everything else. A database, a vector store, a key-value cache, a file on disk — anywhere durable that survives past the session and can be queried back into the context on a future call. The user closes the chat on Tuesday; on Friday they come back and the agent "remembers" their preferences. That recall is a database lookup, dressed up.

The distinction matters because the engineering for each is completely different. Short-term memory is a budgeting problem: you have a fixed token window and you need to decide what fits. Long-term memory is a storage and retrieval problem: you need a schema, a write path, a query path, and a story for what happens when the store grows. Conflating them produces the worst of both — a context full of stale facts and a database full of conversational noise.

Here is the load-bearing claim of this primer: most "memory" problems builders bring to a design review are short-term ones in disguise. The agent forgot a constraint from earlier in the same session. The agent kept making the same tool call because the prior result aged out of the window. The agent gave inconsistent answers on turn 14 versus turn 4. None of those are long-term-memory failures. They are short-term-memory failures — failures to manage the message stack — and adding a vector store to fix them only adds a second thing to debug.

Start with short-term. Get it right. Reach for long-term only when the short-term machinery genuinely cannot carry the load.

The message stack is your default memory

The conversation history is short-term memory. There is no separate "memory module" hiding behind the model — the model has no state between calls except what you put back into the context. Every turn, the loop reads the full message stack and rewrites whatever portion you chose to keep. Memory, at this level, is just deciding what survives the rewrite.

Primer 1 framed the context window as a budget; Primer 2 made that operational. Short-term memory engineering is the same discipline applied turn by turn: every token you keep is a token you cannot use for something else, and the budget is fixed. Three strategies dominate, and most production agents end up using all three, layered.

Sliding window. Keep the last N turns; drop everything older. Simple, predictable, and works surprisingly well for agents whose useful context is local — a tool-using loop where the last three turns capture the active task. The failure mode is obvious: anything important that happened on turn 1 disappears by turn N+1, including constraints the user set up front.

Summarized prefix. Keep the last N turns verbatim; replace everything older with a single summary message that compresses the dropped turns into a few sentences. This is the dominant pattern in long-running assistants. The summary preserves decisions and commitments while the recent window preserves the live working state. Done well, an eight-thousand-token history collapses into a five-hundred-token summary plus the last few exchanges.

Tool-result compression. Tool outputs are the largest line item in most agent histories — a single API response can be ten thousand tokens of JSON. Replace old tool results with a short stand-in once the model has acted on them. "Result of search_customer(88312): returned 14 records; agent selected #3 for case pack." That sentence is everything later turns need; the original payload can go.

None of these strategies require a database. They are all decisions about what the loop writes back into the next turn's context. That is the discipline: treat the message stack as a budget you actively manage, not as an append-only log you hope will fit.

One more move worth naming. If a constraint absolutely must hold across the whole session — a hard policy, a sensitive customer identifier, an output-format requirement — do not rely on it surviving in the rolling history. Lift it into the system prompt, where it is read every turn at the top of context. The system prompt is the one slot the model attends to without fail. Use it for the standing rules that the rolling memory cannot be trusted to carry.

When to reach for long-term

Long-term memory exists for one reason: continuity across sessions. The user closes the chat, the process ends, the in-memory state evaporates — and yet, when the user comes back tomorrow, you want the agent to start somewhere other than zero. If your agent never has to do that, you do not need long-term memory. Most agents never have to do that.

Three patterns genuinely require it. Cross-session continuity for a single user — a coding assistant that should remember last week's project structure, a customer-service agent that should know this caller already had their refund processed. Recurring users with stable preferences — dark mode, preferred language, timezone, notification settings, the kind of facts that should not be re-derived on every conversation. Personalization grounded in history — recommendation systems, where the model's outputs should reflect what this specific user has done before.

Notice what is not on that list. One-shot tool-use agents. Stateless API endpoints. Anything where the session is the unit of work and what happened last week is irrelevant. For those, long-term memory is dead weight: an extra store to provision, a query path to debug, a write path to keep clean, and a privacy surface to defend.

Don't add long-term memory until short-term memory hurts.

The cost of adding it prematurely is not just the infrastructure. It is the operational complexity. A long-term store is a second source of truth. Every turn, you now have to decide whether to query it, what to retrieve, how to merge what you found with the current context, and what to write back. The retrieval logic becomes a system to maintain. The write logic becomes a system to debug. The privacy story — what we stored, who can read it, when we delete it — becomes a compliance question. And the model now has two places it can be wrong: the live context, and the recall from storage.

The honest test is this. Run the agent without long-term memory for a week. Watch what users actually do. If they routinely start their second session by re-stating what they told you in the first, you have a long-term-memory case. If they don't — if every session is naturally self-contained — you do not, and adding a memory store will create more bugs than it prevents.

Episodic vs semantic

Once you have decided long-term memory is warranted, the next decision is what kind. The cognitive-science distinction between episodic and semantic memory is a useful frame here — not because the model literally has either, but because the two map cleanly onto two very different storage problems.

Episodic memory is "what happened." Time-stamped records of events: the user said X on Tuesday at 3pm; the agent called refund_order(778) last Friday; the previous session ended with an unresolved escalation. Episodic memory is sequential, append-only, and queried by recency or by event type. The storage is closer to a log than a database — a stream of facts, ordered by time, that you scan or filter.

Semantic memory is "what's true." Atemporal facts about the user or the world: the user prefers dark mode; this customer is on the enterprise plan; the product catalog includes these SKUs. Semantic memory is a key-value or document store, queried by lookup. There is no concept of "when" — only the current state of what we believe.

The two require different writes. Episodic writes are append-only: a new event arrives, you append it to the log, you never modify the old ones. Semantic writes are upserts: a new fact about the user arrives and it replaces the old fact. "User prefers dark mode" written on Tuesday is overwritten by "user prefers light mode" written on Friday. Get this wrong — store a preference as an episodic event and you end up with a contradictory pile of "user prefers X" records, with the model having no way to know which one is current.

They also require different reads. Episodic reads are usually filtered: "what happened in this user's last three sessions involving refunds." Semantic reads are direct lookups: "what is this user's preferred language." Episodic memory benefits from vector embeddings and similarity search because the query is fuzzy ("anything about refunds"); semantic memory benefits from structured key-value access because the query is exact ("preferred_language"). Mixing the two access patterns into one store is how memory systems become unmaintainable.

A working rule. Ask, for each fact you want to remember: is this an event or a setting? Events go into episodic storage, queried by recency and topic. Settings go into semantic storage, queried by key and always reflecting the latest value. If you can't tell which it is, you don't yet understand the fact well enough to store it.

Summarization and compaction

The most common short-term-memory failure is overflow: the message stack grows past the context window and the loop either crashes or starts silently dropping turns. Sliding windows postpone the problem; they do not solve it. The actual solution is to keep summarizing — compressing the older parts of the conversation into a shorter representation that preserves the load-bearing facts.

The pattern is called the rolling summary. The agent's history is partitioned into three regions, in order: a single summary message at the top, the last N verbatim turns at the bottom, and nothing in between. Each time the verbatim region grows past a threshold, the oldest few turns get summarized and merged into the summary message, then dropped from the verbatim region. The summary message itself is overwritten — it always reflects everything that has been compressed so far.

The summarization step is itself an LLM call, usually to a cheaper model than the one driving the main loop. The prompt is narrow: take this rolling summary plus these new turns; produce an updated summary that preserves user requests, decisions made, key facts established, and any open commitments. Discard small talk and tool-call mechanics.

PYTHON 
from anthropic import Anthropic

client = Anthropic()
MODEL = "claude-sonnet-4-6"
KEEP_VERBATIM = 8     # last 8 turns kept as-is
COMPACT_AT    = 14    # trigger compaction once history exceeds this

def compact(summary, old_turns):
    """Fold old turns into the rolling summary. Returns a new summary string."""
    rendered = "\n".join(f"{t['role']}: {t['content']}" for t in old_turns)
    prompt = f"""Existing summary:
{summary or '(none)'}

New turns to fold in:
{rendered}

Produce an updated summary. Preserve: user requests, decisions made,
key facts, open commitments. Drop: pleasantries, redundant tool chatter."""
    resp = client.messages.create(
        model=MODEL, max_tokens=512,
        messages=[{"role": "user", "content": prompt}],
    )
    return resp.content[0].text.strip()

def step(state, user_msg):
    """One turn of an agent loop with rolling-summary memory."""
    state["history"].append({"role": "user", "content": user_msg})

    # Compact if the verbatim region has grown too long.
    if len(state["history"]) > COMPACT_AT:
        cut = len(state["history"]) - KEEP_VERBATIM
        state["summary"] = compact(state["summary"], state["history"][:cut])
        state["history"] = state["history"][cut:]

    # Build the message stack: summary as a system-level prefix, then verbatim.
    system = state["base_system"]
    if state["summary"]:
        system += f"\n\nPrior conversation summary:\n{state['summary']}"

    resp = client.messages.create(
        model=MODEL, max_tokens=1024,
        system=system, messages=state["history"],
    )
    reply = resp.content[0].text
    state["history"].append({"role": "assistant", "content": reply})
    return reply

Three things to notice. The summary lives in the system prompt, not as a fake user turn — it is standing context about the session so far, not something the user said. The verbatim region is sized in turns, not tokens, because that is what the loop manages naturally; if you need a token-aware version, swap the length check for a tokenizer call. And the summary is rebuilt cumulatively from the previous summary plus the new turns, so the cost of each compaction is bounded — you are never re-summarizing the whole history from scratch.

The failure mode to watch for is summary drift: each pass loses a little detail, and after ten compactions the summary has quietly lost something the model now needs. The fix is structural — if certain fields must survive forever (the customer ID, the case number, the original request), extract them into a separate structured-state object that the loop carries verbatim and never summarizes. Use the rolling summary for narrative; use a small structured state for facts you cannot afford to lose.

When memory is the wrong abstraction

"The agent should remember" is one of those phrases that sounds like a design and is actually a punt. Before you build memory of any kind, check whether the thing you want is really memory at all. Most of the time it isn't, and three other abstractions fit better.

Is it a tool call? If the fact you want the agent to "remember" lives in a system of record — the customer record, the order database, the support-ticket queue — the right answer is almost never to copy it into memory. The right answer is to call the system of record. A tool call is fresh, authoritative, and audited. A memory entry is stale the moment something else updates the underlying fact. "The agent remembered the customer is on the enterprise plan" is usually a worse implementation of "the agent called get_customer_plan(id) when it needed to know."

Is it a re-retrieval? If the agent needs information that exists in a document, a knowledge base, or a vector store, the right answer is to retrieve it again on the turn it is needed, not to remember it from an earlier turn. RAG (Primer 4) is the dominant pattern here. Memory of a document is a cache; retrieval is the source. Caches go stale; sources don't.

Is it a state machine? Many things builders call "memory" are really workflow state — which step of a process the user is in, which approvals are pending, which fields of a form have been filled. That is not memory; that is application state, and it belongs in a structured object the orchestrator carries explicitly, not in the model's recall. State machines are debuggable; remembered state is not.

DISCIPLINE

Memory is the last resort, not the first.

Before adding a memory store, prove the alternatives don't fit. If the fact has a source of truth, call it as a tool. If it lives in a corpus, re-retrieve it. If it represents workflow state, model it as state. Reach for memory only when none of those work — which is rarer than the design conversations suggest.

The reason this matters is not aesthetic. Every abstraction has a debugging story. Tool calls have logs and audit trails. Retrieval has scores and reranks. State machines have transitions you can trace. Memory has none of these — when an agent's memory produces a wrong fact, the only thing you can do is read the entire memory store and guess what went wrong. That is a poor debugging story for production, and it is the reason memory should carry less of the load than first instinct suggests.

The discipline is to keep memory thin. The smaller the memory layer, the less of the agent's behavior depends on something you cannot easily inspect. Tools, retrieval, and explicit state should carry the weight. Memory carries only the residue — the things that genuinely cannot be re-derived and genuinely matter across sessions.

What this connects to

Memory is one of the load-bearing decisions in agent design, and almost every later primer touches it. The pointers worth carrying forward:

  • Primer 5 — The Agent Loop — back-reference. The loop is what reads and writes the message stack on every turn; this primer is the engineering discipline on top of that loop. Short-term memory is the loop's state; long-term memory is what the loop optionally reads from and writes to a durable store.
  • Primer 9 — Sub-agents, Skills, Harnesses — once a memory layer becomes complex enough to need its own logic — what to write, when to summarize, what to retrieve — it often becomes a sub-agent in its own right. A "memory manager" sub-agent that owns the rolling summary, the episodic log, and the semantic store is a clean architectural move when memory grows past a few hundred lines of orchestration code.
  • Primer 11 — Observability & Traces — memory bugs are notoriously hard to diagnose because they manifest several turns after the bad write. Traces are the only honest debugging tool: capture every memory read and write with the turn that triggered it, and the bug becomes visible. If you cannot reconstruct, from a trace, why the agent recalled what it recalled, you do not yet have a memory system you can run in production.

The short version of this primer fits on a single line: the message stack is your default memory; everything else is an exception that must justify itself. Most agents need less memory than their designers think. The ones that genuinely need more pay for it with a real storage system, a clear schema, and a debugging story — not with a vector store sprinkled in as a fix for a context-engineering problem.
