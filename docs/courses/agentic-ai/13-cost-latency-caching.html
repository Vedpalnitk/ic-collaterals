The components of cost

The bill for an LLM-powered system is not a single number you negotiate with a vendor. It is a product. Per-token rate, times tokens per call, times turns per session, times sessions per user, times users. Five factors. Each one is independently a knob. A change in any one of them moves the bill — and because they multiply rather than add, the moves compound.

Consider the numbers that show up in practice. A premium model is roughly an order of magnitude more expensive per token than a small one. A conversational agent that re-sends a 20K-token system prompt on every turn carries ten to twenty times the input tokens of a single-turn classifier. A user session that runs 10 turns costs ten times what a single-shot session does. And the user count is whatever your product reaches.

The arithmetic is unforgiving. A pilot that costs eight cents per user per day looks free. The same architecture at a hundred thousand active users costs eight thousand dollars a day, and at a million it is six figures a month. The pilot did not warn you. The factors did.

The temptation, when the bill arrives, is to find the one factor that fits in a slide and squeeze it. That's a mistake. Because the factors multiply, a 30% cut to any one of them yields the same headline reduction — but the engineering effort to get there varies by an order of magnitude. Prompt caching is usually the cheapest 5–10× you will ever find. Switching a routing decision to a smaller model is a one-line change. Trimming the system prompt by 40% is a half-day of editing. Negotiating a 15% discount with a vendor is a quarter of meetings.

The rest of this primer is the ranked menu of moves. Caching first, because it pays for almost any agent. Routing second, because most turns do not need your best model. Batching third, for the work that does not need to happen now. Streaming for the latency users actually feel. And cost regressions, because none of this stays fixed if you don't watch it.

Prompt caching: the biggest lever

Prompt caching is the single highest-leverage cost move in agent design, and most teams discover it three months later than they should. The mechanism is simple: the provider caches the key/value tensors for a stable prefix of your input, and on subsequent calls that hit the same prefix, you pay roughly 10% of the normal input-token rate for the cached portion instead of the full rate. The write costs slightly more than a normal read, but you pay that once and amortize it over every subsequent call.

For an agent loop, this is transformative. The system prompt — your role, mandate, constraints, tool definitions, possibly few-shot examples — is read on every turn but never changes within a session. Same for the early conversation history once a few turns are committed. Cache the stable prefix; pay full rate only for the volatile tail.

The worked example below makes the ratio concrete. Take a 10-turn agent loop. System prompt: 20K tokens (a realistic agent with tool definitions and policy). User message: 1K tokens per turn. Output: 500 tokens per turn. Illustrative rates: $3 per million input tokens, $15 per million output tokens, cached input at 10% (so $0.30 per million).

Without caching, every turn re-pays for the full system prompt. Input per turn: 20K + 1K = 21K. Across 10 turns: 210K input tokens. Output: 5K tokens. Cost: 210K × $3/M + 5K × $15/M = $0.63 + $0.075 ≈ ~$0.71 per session. If you also re-send the growing conversation history each turn (which a naive loop does), the input climbs into the 250–300K range and you land near ~$0.90 per session.

With caching, the 20K system prompt is paid at full rate once (cache write), then at 10% on the nine subsequent turns. System-prompt cost: 20K × $3/M + 9 × 20K × $0.30/M = $0.060 + $0.054 = $0.114. Plus user inputs: 10K × $3/M = $0.030. Plus output: 5K × $15/M = $0.075. Total: ~$0.22 per session — and if history is also cached as it grows, you land near ~$0.36 per session in the realistic case.

The ratio is the point. $0.90 vs $0.36 is a 2.5× reduction with the system prompt cached. A more system-prompt-heavy agent — say a 50K prompt with rich tool definitions — pushes the ratio to 5× or more. The numbers above are illustrative; your real rates depend on the model and provider tier. The shape of the saving does not.

PYTHON 
from anthropic import Anthropic

# Cache-controlled system prompt. The cache_control marker tells
# the provider this prefix is stable and should be cached for reuse.
system = [
    {
        "type": "text",
        "text": LARGE_SYSTEM_PROMPT,  # 20K tokens of role + mandate + tools
        "cache_control": {"type": "ephemeral"},
    }
]

resp = Anthropic().messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    system=system,
    messages=[{"role": "user", "content": "Assemble case pack for 88312."}],
)

# Inspect cache behavior on the response.
print(resp.usage.cache_creation_input_tokens)  # paid once
print(resp.usage.cache_read_input_tokens)      # paid at ~10% rate

Three operational notes. First, the cache is short-lived — typically minutes — so it serves an active session well and a long-idle one poorly. Second, the cached prefix must match exactly; a single token of drift invalidates the cache and you pay the write cost again. Treat the cached portion as immutable per session. Third, your monitoring needs to track cache hit rate as a first-class metric. A drop in hit rate is a regression that looks normal in latency dashboards but doubles your input bill quietly.

Model routing

Not every turn of an agent needs your best model. A premium model is built to handle the hardest case in your distribution. Most turns are not the hardest case. Treating every call as if it were is paying premium pricing for routine work — the API equivalent of taking a taxi to the mailbox.

The discipline is to identify which calls actually need the top model and route the rest down. The distinction usually splits along a few axes. Routing decisions — classify this request, pick a tool, decide which agent to hand off to — are short, structured, and forgiving; a small model handles them well. Drafting and reasoning — write the analysis, plan the sequence of tool calls, reconcile contradictory evidence — is where the top model earns its rate. Mechanical transforms — re-format JSON, summarize an output for a UI card, translate a snippet — do not need reasoning at all; the smallest model in the family is enough.

Hard reasoning, ambiguous evidence, high stakesLarge (Opus-class)~10–15×

Wire the routing decision itself with a small model. A common pattern: every incoming request first hits a small-model router that returns a structured verdict — {"complexity": "simple|moderate|hard", "tier": "small|mid|large"} — and the orchestrator dispatches to the matching backend model. The router is cheap enough that even if it's wrong 5% of the time, the savings on the other 95% dominate. Add an escape valve: if the chosen backend produces a low-confidence answer or fails a quick self-check, retry once with the next tier up.

The trap to avoid is using your top model for the router. It defeats the point. The router needs to be fast and cheap, not perfect. A good small-model router is sufficient if you've given it a clear prompt and a few examples; it does not need to reason about the underlying request, only to classify it.

Batching

The Batch API exists for the half of your workload that doesn't have to happen right now. You hand the provider a file of requests; they process them within a 24-hour window; you get the results back at roughly half the cost of synchronous calls. The discount is meaningful — 50% off the line item — and the operational cost is just a queue and a callback.

The decision rule is simple. If a request is on the user-facing critical path — a chat turn, a real-time form completion, a tool call inside an agent loop — it must run synchronously and pay full rate. If a request is anywhere else, batching is on the table. The candidates are usually obvious once you look:

  • Eval runs. You run hundreds or thousands of test cases against a new prompt or model. None of them is time-sensitive. Batch.
  • Backfills. You add a new field to a record and need to populate it across the existing corpus. Batch.
  • Nightly enrichment. Tag, classify, or summarize the day's incoming data for tomorrow's dashboards. Batch.
  • Offline data pipelines. Turning unstructured documents into structured records for a downstream warehouse. Batch.
  • A/B comparisons across models. Run the same hundred prompts against two model versions for an offline comparison. Batch both.

The trap is using batch for anything a user is waiting on. The Batch API does not commit to a fast turnaround inside the 24-hour window — it might come back in five minutes or twenty hours, and you cannot tell which in advance. If your dashboard widget waits on a batch result, your dashboard widget will be empty for stretches. Keep batch on the offline side of the system and the question of when answers it for you.

One adjacent point: the eval discipline from Primer 10 pairs naturally with batch. The biggest single user of the Batch API in mature agent shops is the evals pipeline — every prompt change kicks off a batch run of the test set, which lands by morning at half cost. The economics of doing serious eval work are noticeably better once batching is wired in.

Streaming for perceived latency

Streaming does not make the model faster. The model takes the same wall-clock time to produce the same number of tokens whether you stream or not. What streaming changes is when the user sees the tokens — and for any human-facing surface, that perceived latency is what gets called "speed."

A 1,500-token response at 60 tokens per second takes about 25 seconds to finish. Without streaming, the user stares at a spinner for 25 seconds and then the whole response appears at once. With streaming, the first word appears in roughly 800 milliseconds — the time-to-first-token — and the rest scrolls in as it generates. The total time is identical. The experience is not.

The decision matrix is short. Chat-style interfaces should always stream. The user is reading as the tokens arrive; the streaming pace is matched to reading speed; the first-token latency dominates the perceived experience. Tool-loop interiors usually should not stream. When the model's output is consumed by another piece of code — to be parsed for a tool call, validated against a schema, fed into the next step — streaming buys you nothing and complicates the parsing logic. Wait for the full message and act on it cleanly. Long-form artifacts — a generated report, a draft document — are a judgment call. If the user is actively watching, stream. If you're rendering into a saved artifact they'll review later, batch the write.

Two implementation notes. First, the time-to-first-token (TTFT) metric is more useful than total latency for any streaming surface. Track it explicitly. A regression in TTFT — caused by a larger system prompt, a slower cache hit, a model swap — degrades the chat experience even if total latency is unchanged. Second, design the UI to make the streaming legible. A response that streams smoothly feels alive; a response that streams in jerky bursts feels broken. Pick a rendering cadence that smooths the bursts and shows a cursor at the leading edge.

Cost regressions

Cost is an eval. It deserves the same treatment as accuracy, latency, and safety: tracked per release, compared across versions, alarmed on drift. Most teams don't think of it this way, and most teams are surprised by their bill at the end of the quarter. The pattern is recognizable. Someone adds a few sentences to the system prompt to fix a behavior. Someone else adds a new tool with a verbose schema. A third change inadvertently breaks the cache prefix. None of these show up in accuracy metrics. All of them show up in the bill.

The discipline is to run a small, fixed set of representative tasks against each release, measure tokens-per-task — input tokens, output tokens, cache hit rate, all broken out — and compare to the previous release. A 30% increase in tokens-per-task with no measurable quality gain is a regression. It might be intentional and worth it. More often, it's an accidental side effect of a change that shipped without anyone looking.

The metrics worth tracking are concrete. Tokens per task — split input and output. Cache hit rate on the cacheable portion of the prompt. Cost per session at p50 and p95. Latency per call and time-to-first-token for streaming surfaces. Plot them over time. The drift will be obvious if you're watching.

DISCIPLINE

Cost is an eval. Track it with the others.

Tokens-per-task and cache hit rate belong on the same dashboard as accuracy. A 30% increase with no quality bump is a regression — find which change caused it and decide deliberately whether to keep paying for it.

One organizational note. Cost regressions are almost always solvable, but they get solved only when someone owns the number. If no one is reviewing the cost dashboard, the dashboard does not exist. Pick someone — usually the same engineer who owns the eval set — and put "weekly review of cost-per-task" on their calendar. The discipline is not subtle. It's just consistent.

What this connects to

Cost, latency, and caching sit at the operational floor of the whole course. They turn the design choices upstream into a number on a bill, and they constrain what is buildable at scale. The pointers worth carrying forward:

  • Primer 1 — LLM Mental Model — the budget framing for the context window. This primer is the operational consequence: a token spent is paid for every turn it's in scope, by every user, across every session. The budget in Primer 1 is also a literal budget in dollars.
  • Primer 5 — The Agent Loop — the loop is where the multiplicative cost effects compound. Every architectural decision in the loop — what to keep in context, how often to call, when to summarize — is a cost decision dressed as a design decision.
  • Primer 10 — Evals & Measurement — the eval discipline extends to cost. Tokens-per-task is an eval metric. The same pipeline that catches accuracy regressions should catch cost regressions, and the Batch API makes the eval run itself cheaper.
  • Primer 18 — Agent Platforms & Internal Tooling — model routing is most naturally a platform capability. A model gateway that handles routing, caching, batching, and cost reporting in one place is the right home for everything in this primer. Most production agent shops build this gateway before they build their third agent.

Cost is not glamorous. It is also not optional. The architectures that survive contact with production traffic are the ones that took these numbers seriously from the pilot — not the ones that got a surprise invoice in month three.
