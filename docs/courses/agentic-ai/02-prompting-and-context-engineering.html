Prompt vs context

The word "prompt" gets used for two different things, and conflating them is the source of most of the bad advice on the internet. A prompt, strictly, is the single message you send — the user-role string the application hands the model right now. The context is everything the model actually sees on that call: the system prompt, the tool definitions, the conversation history, the retrieved documents, and the current user message, all concatenated into one token sequence that the model attends to.

The model does not know which part you "wrote" and which part came from a retrieval step or a previous turn. It sees one sequence. Its next-token distribution is conditioned on the whole thing.

This matters because most failures builders blame on "the prompt" are actually failures of the context. The instruction was clear. The user's request was reasonable. But the model also saw four irrelevant prior turns, a 2,000-token tool schema for a tool it didn't need, and a retrieved passage about a different customer. With that on the table, the next-token distribution drifted somewhere unhelpful. Tightening the user message would not have rescued it. Tightening the context would.

A bad prompt is rarely the problem. A bad context is.

Prompt engineering is the craft of writing the message — clear instructions, the right examples, a structured request. It is necessary and it has a ceiling. Context engineering is the craft of deciding what the model sees at all — which prior turns survive, which documents get retrieved, which tools get defined, which examples make it into the system prompt this run. It is the larger discipline, and it is where the leverage sits when you move from a single call to an agent.

The shift from "I am writing a prompt" to "I am composing a context" is the single most important upgrade a builder makes in their first month with these systems. Once you make it, you stop tuning sentences and start designing pipelines: how the context gets assembled, how it gets trimmed, what gets cached, what gets pulled fresh. The rest of this primer is about that discipline.

System prompts: role, mandate, constraints

The system role exists for a reason. It is the slot reserved for instructions that apply to the whole interaction — not this turn, every turn. The model is trained to weight the system prompt as standing orders. Use that. Put the things that must always be true here, and only the things that must always be true.

A well-formed system prompt does three jobs, in this order:

Role. Who the model is being asked to be. "You are a compliance analyst." "You are a senior backend engineer reviewing a pull request." "You are a customer-service triage assistant for a retail bank." One sentence. Pick the role that matches the work, not a flattering archetype. "Expert PhD in finance" is worse than "compliance analyst" because the model has seen far more examples of analysts doing analyst work than of expert PhDs writing anything.

Mandate. What the work is, in one or two sentences. "Your job is to assemble a KYC case pack from the customer record and screening results." This anchors the model to a goal. Without a mandate, the role floats — a "compliance analyst" might answer questions, write essays, or refuse on policy grounds. With a mandate, the role narrows to a job: case packs, produced from these inputs.

Constraints. What not to do, where the limits are, what to escalate. This is where structure beats prose. The model reads bullets more reliably than embedded clauses. A constraint buried mid-paragraph ("...and of course never make claims without evidence...") gets weighted alongside the surrounding sentences. The same constraint on its own line, prefixed with a dash, gets weighted as a standing rule.

PYTHON 
from anthropic import Anthropic

system = """You are a KYC analyst.
Your job is to assemble a case pack from the customer record and screening results.

Constraints:
- Never claim a customer is clean without an attached evidence record.
- Escalate any PEP hit or sanctions adjacency to a senior reviewer.
- Cite the source for every risk factor."""

resp = Anthropic().messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    system=system,
    messages=[{"role": "user", "content": "Assemble case pack for customer 88312."}],
)

Notice the shape. Role on line one. Mandate in the next paragraph. Constraints as a bulleted list with imperatives. No lore, no flattery, no "please." The model does not need to be motivated. It needs to know what to do and what not to do.

Two things to avoid. First, do not stuff turn-specific facts into the system prompt. "The customer is 88312" belongs in the user message, not the system block — it will change next turn, and a stale system prompt is a slow-moving lie. Second, do not let the system prompt grow without pruning. Every line you add is read every turn and paid for every turn. A constraint that fired once during testing and never since is a candidate for deletion, not preservation.

A good rule for sizing: the system prompt should be the shortest string that produces the behavior you want across a varied test set. Not the longest you can write before the model gets confused. Compression is a feature.

Few-shot: when, when not, and how

Few-shot prompting means showing the model worked examples of the input/output pattern before asking it to do the same on a new input. The technique predates instruction-tuned models — it used to be the only way to get a base model to do anything specific — and it still works, just for a narrower set of cases than people assume.

Zero-shot is the default. Modern instruction-tuned models follow clear instructions on standard tasks without examples. Reach for few-shot only when there is something the instruction alone cannot convey.

Three reasons to add examples:

  • The output format is non-obvious. A custom JSON shape your downstream code expects, a domain-specific log format, a structured handoff between agents. Examples lock the shape better than schema descriptions.
  • The reasoning pattern is non-standard. You want the model to follow a specific decision tree — check A first, then B only if A passed, then escalate if B is ambiguous. Show two or three traces of that pattern and the model will imitate it.
  • Edge-case coverage matters. Pick the two hardest cases in your test set and show the desired behavior on them explicitly. This is cheap insurance.

Three reasons not to:

  • The format is standard. Free text, simple JSON, a bullet list — the model already knows how to produce these. Examples add tokens without changing behavior.
  • The examples bloat the context. If you are reaching for ten or more examples, you are doing in-context learning. Consider fine-tuning instead — same outcome, none of the per-turn cost.
  • You'd be tempted to write "and so on…" That phrase means you don't really know the pattern yet. Go back to the spec; do not paper over it with examples the model will overfit to.

How to do it well: pick examples that span the input distribution, not three near-duplicates. Put them inside the user message or, for the strongest effect, as separate user/assistant turns in the history. Never paraphrase a real customer's record verbatim — your examples become part of the training signal for whatever future model your provider distills, and PII in examples is PII in the prompt.

Structuring the message stack

The model's view at any given turn is not a single string. It is the message stack: the system prompt at the top, the tool definitions alongside it, the conversation history in order, and the current user message at the bottom. The SDK serializes this into one token sequence; the model reads it as a whole. But where you place a piece of information in the stack matters more than the words you choose for it.

Two empirical patterns dominate. First, the model attends most strongly to the start and the end of context. Instructions placed at the very top of the system prompt and information placed at the bottom of the last user message both get strong attention weight. Material in between competes harder for it.

Second, the so-called "lost in the middle" effect: as conversation history grows, the middle of the stack — turns 3 through N-2, say — gets progressively under-weighted. Important constraints that you set on turn 1 and never repeated will quietly stop firing by turn 20. The model still sees them. It just no longer treats them as load-bearing. If a constraint matters across a long session, restate it in the system prompt (which always sits at the top) or re-inject it into the latest user message.

Roles in the stack are not cosmetic. The system / user / assistant / tool labels are how the model distinguishes who said what. They affect attention patterns and influence what the model treats as authoritative versus what it treats as data. Two common mistakes follow from forgetting this.

One: pasting raw user input into the system prompt to "give the model context." This collapses the boundary between standing orders and untrusted data. The model will treat injected instructions in the user input as if they came from you. This is the basic shape of prompt injection. Keep the system role for things you wrote and trust.

Two: hand-rolling fake assistant turns to "show the model what it said." If the model didn't actually generate those tokens, you are lying to it about its own history. The behavior gets harder to predict — the model may try to reconcile the fake turn with its current parameters and produce something neither of you wanted. If you need to demonstrate an output shape, do it as a few-shot example clearly labeled as an example, not as a fabricated history.

A working rule pulls it together: if it must always be true, put it in the system prompt. If it depends on this turn, put it in the user message. Standing orders go up top. Turn-specific facts go at the bottom. The middle is for history, and you should expect the model to under-weight it.

Prompting an answer vs prompting a behavior

This is the single most important distinction in this primer, and it is the one most builders only learn by getting burned. There are two fundamentally different things you can ask a prompt to do, and they require different craft.

Prompting an answer is the case most people start with. You have one question; you want one response. "Summarize this document." "What is the weather in Bengaluru?" "Classify this ticket as billing, technical, or other." The prompt is a one-shot specification, the model produces one output, the interaction ends. The craft is in writing the question well, picking examples that nail the format, and tuning until the model's response on your test case looks right.

Prompting a behavior is the case agents force you into. You are no longer specifying one answer; you are specifying a pattern the model will apply across many turns and many inputs you have not seen yet. "When the user asks for weather, call get_weather with the city, then format the result as a brief summary." "When you receive a ticket, check the customer's plan tier first, then route based on the matrix in the constraints, and escalate to a human if any field is missing." The prompt is no longer a question. It is a policy.

The shift happens whether builders notice it or not. The first time someone wires an LLM into a tool-using loop, they are no longer prompting an answer. They are writing a policy the model will execute against tomorrow's user input, which they have not seen. Most of the pain in early agent projects comes from builders applying answer-prompting craft to a behavior-prompting problem.

The differences in craft are concrete:

  • Stop writing "here's an example, copy this." Write "here's how to think about cases like this." Examples teach format; rules teach generalization. Behavior prompts need both, weighted toward rules.
  • Stop chaining if-then conditions in prose. A paragraph with five embedded conditionals is a lottery for the model. Structure the conditions as a decision tree or a bulleted policy — material the model can read deterministically.
  • Stop testing against one representative case. A behavior prompt needs to survive twenty varied inputs, including the ones designed to break it. If you've only seen it work on the happy path, you've shipped a fragile policy and labeled it a feature.
DISCIPLINE

Prompt the behavior, not the example.

If you tune the prompt to nail one example, you'll find a second example that breaks it. Tune the prompt to a generalizable rule, then verify it across a varied set. Treat any prompt that "only works for this one case" as broken.

The mental shift is also a tooling shift. Answer-prompting can be done in a chat window. Behavior-prompting needs an eval set — twenty to a hundred varied inputs, expected outputs or rubrics, a way to run the prompt across all of them and see the failures. Once you are prompting a behavior, the eval set is the prompt's specification; the prompt text is the implementation. Tuning the prompt without re-running the evals is editing code without compiling it.

Context as a budget

Primer 1 framed the context window as a budget. This section is the operational version of that idea: how, when you are doing context engineering, you actually spend the budget.

Every turn, everything in scope competes for the same fixed pool of tokens — 200K, 1M, whatever the model gives you. The competitors are the system prompt, the tool definitions, the conversation history, the retrieved passages, the current user message, and the room left for the response. The window is shared. You cannot grow one item without shrinking another or paying more.

The items split cleanly into expensive and cheap. Tool definitions are expensive — a useful agent's eight to twelve tools easily run 3,000–8,000 tokens together. Retrieved passages are expensive and unbounded if you don't cap them. Conversation history is expensive and grows monotonically unless you summarize. The system prompt is expensive only on the first turn; with caching, it's nearly free thereafter. Instructions and short examples are cheap. The current user message is usually the smallest line item.

Once you see the items clearly, the engineering choices become obvious. Pay for the things that earn their cost.

Spend on

  • The system prompt — read every turn, cache it.
  • 2–5 well-chosen few-shot examples
  • The retrieved passages the answer actually depends on
  • Tool definitions for tools the agent will call this turn

Trim first

  • Old tool results that aren't relevant anymore
  • Conversation turns from 5+ exchanges back — summarize instead
  • RAG passages that scored low on the retrieval rerank
  • Tool definitions for tools the agent never uses

When the budget gets tight in production — typically around turn 15 of a long session, or when a RAG step pulled in a long document — you have three levers, in order of preference. Summarize history: replace turns 1–10 with a 500-token summary that preserves the decisions and drops the chatter. Retrieve less: tighten the top-k on your retriever, rerank harder, return fewer passages. Prune tools: if the agent has 12 tools but this user request will only touch 3, narrow the tool set per turn rather than declaring all 12 every time.

The discipline is to think of every token in the context as a choice you made — either deliberately because it earns its cost, or accidentally because no one trimmed it. By the time you have an agent in production, the context is your most important configuration surface. Treat it like one.

What this connects to

Prompting and context engineering touch nearly every later primer in the course. The pointers worth carrying forward:

  • Primer 1 — LLM Mental Model — the context-as-budget framing started there. This primer is the operational version: how to spend the budget once you are composing real contexts.
  • Primer 3 — Tool Use & Structured Output — tool definitions are themselves a form of prompt; the model reads each tool's name, description, and schema as instructions about when and how to call it. Structured-output schemas are how you turn "prompting an answer" into "prompting a parseable answer" — the schema does the work that a fragile post-hoc regex would have done.
  • Primer 4 — Retrieval-Augmented Generation — RAG is the dominant strategy for getting the right context into the budget. Most of context engineering at scale is retrieval engineering: what to retrieve, how many passages, how to rerank, when to skip.
  • Primer 5 — The Agent Loop — the system prompt is read once per turn but rarely changes; cache it. Context engineering at agent scale is where prompting stops being a knack and becomes a discipline — the loop forces every decision about what to include, trim, and summarize to be made dozens of times per session.

Prompting is a habit, not a trick. The prompts that ship are the ones you've tested across a varied set, structured for legibility, and stripped down to the load-bearing instructions. Everything else is decoration — and decoration costs tokens every turn.
