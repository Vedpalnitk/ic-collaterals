Why agent evals are different

A unit test is a simple promise: function(input) == expected. You run the function, you check the result, the build is green or red. The discipline that makes software shippable rests on that promise. Agents break it on the first call.

An agent is a model that decides, a harness that executes, and a loop that lets the two of them take turns. None of those three pieces is deterministic in the way a function is. The model samples from a distribution. The harness invokes tools that hit real systems. The loop chains them together for an unknown number of steps. You are not checking a value; you are watching a process unfold and asking whether enough of its endings are the ones you wanted.

The non-determinism. Send the same prompt to the same model twice and you will get two different trajectories. Different reasoning, sometimes different tool calls, occasionally a different final answer. Temperature near zero narrows the distribution but does not collapse it. A test that asserts a single output is a test that flakes the first time the wind changes. The only honest unit of measurement is a rate — a pass count over a sample size — with a confidence interval attached.

The multi-step nature. Even when the agent gets the final answer right, the path mattered. Did it call the screening tool before the decision tool? Did it stop at the first confident lead, or did it over-search and burn ten extra turns? Success is not a single assertion; it is a sequence of decisions ending in the right state, on a reasonable budget. Evals have to capture both the destination and, where it matters, the route.

The tool mediation. Agent outputs are joint products of the model and the tools. If the screening API returns slightly different data on Tuesday than on Monday, your agent's answer changes — and that's the system working correctly. The tool's state at call time is part of the input, but it is not part of the prompt you wrote. Evals that don't account for this either freeze the world (mocks, fixtures) and lose realism, or accept the world as it is and reason about variance.

Traditional unit tests struggle on all three counts. They are binary where the system is probabilistic. They are single-shot where the system is multi-step. They are synthetic where the system is plumbed into real tools. You can write them; they will not protect you. What you need is a different posture: a population of tasks, scored on a rubric, summarized as a rate, and watched for movement over time.

Don't ask "did it answer correctly once." Ask "what fraction of the time does it answer correctly, and is that fraction stable?"

The task suite

An agent eval starts with a corpus of tasks — small, well-specified problems with a way to score the outcome. The corpus is the load-bearing artifact. Models change; harnesses change; prompts change. The task suite is the constant the rest of the system is measured against. A good suite has three properties.

Representative. The tasks are sampled from real user queries, not synthesized by a developer in a notebook. Notebook tasks tend to be too clean, too neutral, and too short. They miss the messy phrasing, the ambiguous intent, the half-typed referent ("the one from last week"), and the long-tail edge cases that production traffic surfaces every day. If your agent runs in production, sample from production logs. If it doesn't yet, sample from whatever real user input you have — pilot transcripts, support tickets, internal dogfood — and re-sample when real traffic starts. The closer the suite tracks the live distribution, the more your CI score predicts the user-facing one.

Balanced. A suite that's 80% easy queries will tell you the model is great long after it has stopped being great on the cases that matter. Bin tasks by difficulty (easy, medium, hard), by tool path (which tool sequence the answer requires), and by intent class (including refusals — a healthy agent says "I can't" sometimes, and that needs its own coverage). Track pass rates per bin, not just the headline. Regressions usually show up first in the smallest bin, and they are invisible if you only look at the average.

Growing. Every production bug becomes a fixture. The first time a user reports the agent screening the wrong customer, you reproduce the failure, add it to the suite, and from that day on the CI gate ensures the bug stays fixed. The suite is the institutional memory of every mistake the system has made. Treat additions like you treat regression tests in a mature codebase: cheap to write, expensive not to.

A task itself is small. It has an identifier, a prompt (the user input), and a rubric (the set of conditions the trajectory and result must satisfy). Optional fields capture seed state, expected tool calls, ceiling on iterations, and tags for the bins above. The whole thing is code, kept in the repo, versioned with everything else.

PYTHON 
EVAL_TASKS = [
    {
        "id": "kyc-001-clean-pep-match",
        "prompt": "Run KYC on customer ID 88312. PEP screening only.",
        "rubric": {
            "must_call_tool": "screen_pep",
            "must_attach_evidence": True,
            "max_iterations": 6,
        },
    },
    # ...
]

Where do the tasks come from? In a mature setup, three sources. Sampled traffic — a fixed fraction of real sessions, de-identified, hand-labeled with the correct outcome, then frozen into the suite. Regression fixtures — every bug ticket the team closes ships with the failing case attached. Adversarial cases — written deliberately to probe known weak spots: ambiguous phrasing, multi-language input, customers that match multiple records, requests that should be refused.

How do they grow? They grow with the system. A useful target for an agent in production is 100–300 active tasks in the main suite, with another 20–30% held back for the shadow set (see §5). New tasks land via PR like any other code change, reviewed by a peer who confirms the rubric is right and the case isn't a duplicate. Old tasks get retired when they no longer reflect anything the agent is asked to do. The suite is a living document of what "working" means for this agent, today.

Judges: rules, models, humans

A task is only as useful as the thing that scores it. Three classes of judge are in use, and a serious eval system uses all three, in different ratios, for different reasons.

Rule-based judges

Rule-based judges check things you can write down as code. Did the agent call screen_pep? Did its final response validate against the JSON schema? Did it attach an evidence object with a non-empty source field? Did it stay under the iteration cap? These checks are cheap, deterministic, and run in milliseconds. They are also narrow — they can tell you whether the structural contract was met, but they cannot tell you whether the answer was right. Use them as the floor: a task that fails a rule-based check is failing before you even ask a model judge whether the substance was correct.

Model-as-judge

Model-as-judge is the workhorse. A second LLM, given the task, the agent's trajectory, the agent's final answer, and a written rubric, produces a graded score and a short explanation. Cheap-ish — one extra call per task — and broad: it can rate correctness, completeness, citation quality, tone, and a dozen other axes that no rule could express. It is also wrong sometimes, in ways the rule-based judge never is: it can be inconsistent across runs, biased toward verbose answers, susceptible to prompt-injection from the agent's own output, and prone to grade-inflation when the rubric is vague. The fix is structural: short, specific rubrics; structured output; lower temperature; and a separate judge model from the one being evaluated (so a model isn't grading itself).

Human judges

Humans are the gold standard and the bottleneck. A subject-matter expert can score a trajectory with judgment no rule and no LLM has — was this the right call given what a compliance officer would actually do? Is the evidence cited the right evidence, or just plausible-looking? Humans do not scale to grading every PR, but they do something more important: they grade the graders. A small, carefully labeled set of human-scored tasks is what lets you trust the model judge on everything else.

Which brings up the recurring question: how do you know your model judge is right? You hold out 50–100 tasks that humans have scored, you run the model judge on the same set, and you compute agreement. Cohen's kappa κ over about 0.7 is the rough threshold for "the model is grading well enough to deploy as the primary judge"; below that, you go back and tighten the rubric or change the judge model. Re-run the calibration on a fresh human-scored sample every quarter, or whenever you change the judge prompt — because the judge drifts too, and a judge you stopped checking is a judge you stopped trusting.

Scoring rubrics

How a task is scored decides what the team optimizes for. Pass/fail is the simplest rubric — the trajectory either met every condition or it didn't, and the score is a single bit. It has the virtue of being unambiguous: CI is green or red, the gate is binary, no one argues about partial credit. It has the vice of hiding everything else. Between two releases, you cannot tell whether the failing cases got closer or further away. A run that's "99 fails, 1 pass" looks identical to a run that's "100 fails, 0 pass," even though one is on the verge of green and the other is in the wilderness.

Graded rubrics fix the hiding problem. Score each axis you care about on a small ordinal scale — 0 to 4 is a common choice — and let the judge produce a per-axis grade. A typical agent rubric has four axes: correctness (did the final answer get the substance right), completeness (did it address everything the user asked), citation (did it ground its claims in the right evidence), and latency (did it finish within the iteration and wall-clock budget). The scores are noisier than a binary, but they show movement. A regression from a 3.8 average correctness to a 3.5 is something you can act on a week before the binary pass rate ever flips.

Composite scores roll the per-axis grades into a single number — usually a weighted mean. They make great dashboard tiles and terrible debugging tools. A 3.4 composite that's "3.0 correctness, 3.8 everything else" tells a very different story from a 3.4 composite that's "4.0 correctness, 2.8 citation," and the composite alone tells you neither. Report the components on the dashboard, even when leadership wants the headline number.

The recommendation: gate on a binary derived from a graded rubric. The CI rule is something like "pass if every axis ≥ 3, fail otherwise." The team gets the unambiguous gate they need, and the dashboard gets the direction-of-travel signal it needs, from the same run. You pay the cost of a graded judge once and get both products out the other side.

Eval-driven development

Test-driven development has a rhythm: write the failing test, watch it go red, write the smallest change that makes it green, refactor. Eval-driven development is the same rhythm at a different cadence. Write the failing eval — a new task representing a behavior the agent doesn't yet do, or a bug it currently has. Run the suite, see the new case fail in a sea of green. Change a prompt, a tool, a guardrail. Run again. Green. Commit.

This sounds straightforward and is not, because evals are slow and noisy in ways unit tests are not. A 200-task suite scored with a model judge takes minutes to run and costs real money. A single task with temperature above zero can flip pass and fail across consecutive runs. The discipline of writing the eval first is what protects you from optimizing toward a moving target with no record of where you started. Without it, "the agent is better now" becomes a vibe call. With it, it becomes a diff.

Three things make this work in practice.

Evals run on every PR. Not nightly, not weekly — on the PR itself, as a required check. The cost is non-trivial (a few dollars per PR is typical) and worth paying every time. A reviewer should never have to ask "did you run the evals." The CI ran them; the result is on the PR; the conversation starts from there.

The team holds a shadow set. If every developer can see every task in the suite, the team will (consciously or not) overfit prompts to those tasks. A prompt that mentions "always cite the customer ID" because two of the visible tasks failed for not citing it might pass those two and quietly degrade five others nobody is looking at. Hold back 20–30% of tasks that only run in CI and that only the eval owner can read. The shadow score is the honest one.

Results are diffs, not points. A PR doesn't fail because the overall pass rate dropped a point. It fails because these three specific tasks regressed. Show the per-task delta — what was passing on main and is failing on this PR, what was failing on main and is passing on this PR. The diff makes the conversation specific. "You broke kyc-014" is a different review than "the score went down."

DISCIPLINE

The shadow set is what keeps you honest.

If every developer sees every eval task, the team will (consciously or not) overfit prompts to them. Hold out 20–30% of tasks as a shadow set that only runs in CI and only management sees pass rates for. Treat divergence between visible and shadow scores as a leading indicator of overfitting.

Regression detection

Once evals run on every PR, the next problem is figuring out what the numbers mean. Agent eval suites are small by software-testing standards — 50 to 300 tasks is typical, and the cost and runtime of going much larger is real. At those sample sizes, single-point comparisons are noisy enough to mislead, and the team that flies by them ends up chasing variance instead of regressions.

Put confidence intervals on the pass rate. A pass rate is a proportion, and proportions have well-understood error bars. The Wilson interval is the standard choice for small N and proportions near 0 or 1; it gives sensible bounds where naive normal-approximation intervals do not. You don't need to derive it — every stats library has a one-liner — but you do need to report it. A score of "84% ± 7%" is a different artifact from a score of "84%," and only the first one stops the team from arguing about a 3-point delta that is inside the noise floor.

Diff per task, not just per suite. The headline pass rate can move because one critical task regressed or because five low-value ones flipped. Those situations call for different responses, and the headline doesn't distinguish them. Surface, for every PR: which tasks passed on main and failed on the PR, which tasks failed on main and passed on the PR, and which tasks flipped between runs of the same code. A focused regression in one tool path is a fix; a broad sweep across the suite is a model-behavior shift; a single high-value case is a P1 ticket. The diff is what makes those calls obvious.

Re-run before you declare. Temperature above zero means a fraction of tasks will flip on any given run, with no underlying cause. Before reporting a regression, re-run the failing tasks two or three times. If they pass on the rerun, you have flake, not a regression — note it, mark the task as variance-prone, and move on. If they fail consistently, you have a real signal. This rule alone removes most of the noise from the team's day.

Watch the "looks like a regression" trap. A 2-point pass-rate drop on a 100-task suite has a confidence interval roughly ±8 points around it — comfortably consistent with no change at all. Teams that don't internalize this end up chasing every wiggle. The remedy is partly statistical (intervals, reruns) and partly cultural (don't act on a single-PR number; trend over a week of runs). Regression detection is its own discipline. Don't fly by single numbers.

What this connects to

This primer is the methodology layer. The other half — the runtime plumbing that captures trajectories, ships them to the eval store, and makes the diff above mechanically possible — lives in adjacent primers and in an existing engineering note.

Traces (Primer 11). When an eval fails, the trace is where you go to understand why. A trace is the full recorded record of a run — the prompt, every tool call, every tool result, every intermediate model response, the final answer, and the timings on each. Evals tell you that the agent regressed; traces tell you where in the loop the regression happened. The two are useless without each other, which is why they ship together.

Safety evals (Primer 12). Everything in this primer is about whether the agent does its job. A separate suite — red-team prompts, prompt-injection probes, refusal cases, jailbreak attempts — measures whether the agent stays inside the lines while doing it. Safety evals run on a different cadence (often nightly rather than per-PR) and have different scoring rules (a single failure on a critical safety case blocks the release, even if everything else is green). Treat them as a parallel suite, not as more rows in this one.

Existing engineering note. Agentic Harness, Evals & Traces covers the runtime infrastructure — how to wire a harness, capture traces, route trajectories to evals, and structure the store that holds it all. This primer is the methodology; that note is the plumbing. Read them together when you build.

Evals are the threshold module of this course. Everything before it is what an agent is. Everything after it depends on knowing whether the one you built is working. A demo is a single trajectory you got lucky with; a deployable system is a population of trajectories you have measured. The line between the two is drawn here.
