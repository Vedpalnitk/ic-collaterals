Why planning matters

The canonical agent loop from Primer 5 is incremental. The model looks at the current context, decides on one next action, the harness runs it, the result lands back in the context, and the loop turns again. There is no plan written down. There is no commitment to a sequence. Each step is decided in the moment, with all the prior steps as input.

That works beautifully for a large class of agent tasks — the ones where the next step genuinely depends on what just happened. A support agent answering a question, a coding agent following a stack trace, a research agent chasing a citation. In each case, step three only makes sense after step two lands. There is no plan to write, because the plan would have to be hedged so heavily on what step one returned that you might as well let the model decide step three when it gets there.

Planning is the other case. Planning is when you write the whole sequence first, then execute it. The model produces an ordered list of steps before any tool fires. The harness then walks the list, executing each step, optionally checking the result before moving on.

The distinction is not stylistic. It changes what the model is doing and what the harness has to do. In an incremental loop, the model is a step-chooser; the harness is a thin runner. In a planned execution, the model is a planner-then-step-executor; the harness owns the plan, tracks progress against it, and decides what to do when a step deviates from expectation.

Reach for planning when the task is big enough that the model needs to think across steps, not just within them. A multi-system data migration. A research brief that has to cover five subtopics in a specific order. A code refactor that touches eight files and has to land in a sequence that keeps the build green. In each case, the cost of running step three before realising step one was wrong is high — and the shape of the work is knowable up front. That is the sweet spot for planning.

Reach for incremental when the task is exploratory, branchy, or short. The cost of "wasting" one step in a four-step run is small; the cost of writing a plan you immediately have to revise is large. Most agents in production are mostly incremental, with planning reserved for the harder cases.

Plan-then-execute

The simplest planning pattern is a two-phase split. Phase one: ask the model to produce a numbered plan — a list of steps, each with a name, a brief description, and the tool it expects to call. No tools fire in this phase. The output is structured text. Phase two: the harness walks the plan, executing each step in turn. The model may be re-invoked per step to flesh out arguments or summarise the result, but the shape of the sequence is fixed before any tool runs.

Why bother with phase one at all? Three reasons. First, the model often writes a better plan than it executes — given a clear request and no immediate pressure to act, it considers the whole sequence rather than just the next move. Second, the plan is inspectable. You can show it to a human reviewer before running it. You can log it. You can compare it to what actually happened. Third, the harness gains structure: it knows what step four is supposed to do, so it can detect when step three's output is going to make step four impossible.

Used for: long-horizon tasks with known shape. A KYC case-pack assembly that always has the same six phases. A nightly data pipeline that hits the same systems in the same order. A research brief that follows a fixed outline.

Risk: the plan can be wrong. The model wrote it before it saw any tool outputs, working from its priors. If the first tool call returns something unexpected, the next four steps may be addressing the wrong problem. Plan-then-execute has to be paired with a step-level check or it will cheerfully run a bad plan all the way to completion.

PYTHON 
from anthropic import Anthropic

client = Anthropic()
MODEL = "claude-sonnet-4-6"

def make_plan(goal: str) -> list[dict]:
    """Phase 1: ask the model for a numbered plan as structured JSON."""
    resp = client.messages.create(
        model=MODEL,
        max_tokens=1024,
        system="You are a planner. Produce a numbered plan as a JSON list. "
               "Each item: {step, name, description, tool}. No tools fire yet.",
        messages=[{"role": "user", "content": f"Goal: {goal}"}],
    )
    return parse_plan(resp.content[0].text)

def execute_step(step: dict, prior_results: list) -> dict:
    """Phase 2: ask the model to carry out one step, given the plan and history."""
    resp = client.messages.create(
        model=MODEL,
        max_tokens=1024,
        tools=TOOL_REGISTRY,
        system=f"Execute step {step['step']}: {step['name']}. "
               f"Use tool: {step['tool']}.",
        messages=[{"role": "user",
                   "content": f"Step description: {step['description']}\n"
                              f"Prior results: {prior_results}"}],
    )
    return run_tool_call(resp)

def plan_then_execute(goal: str):
    plan = make_plan(goal)
    results = []
    for step in plan:
        outcome = execute_step(step, results)
        if not step_succeeded(outcome, step):
            return replan_or_escalate(goal, plan, results, step, outcome)
        results.append(outcome)
    return results

Notice the shape. make_plan returns structured data, not prose. execute_step is a thin per-step loop with a tight system prompt. The outer plan_then_execute walks the plan but always has a branch for failure — replan_or_escalate is not optional decoration. Without it, the harness silently pushes through bad plans, which is the failure mode Section 6 is about.

Incremental planning

The other extreme is to write no plan at all. The canonical loop from Primer 5 decides each step on the fly: the model sees the current context, picks one action, the harness runs it, the result lands back in the context, the model picks the next action. There is no commitment past the current step. The "plan," to the extent there is one, lives inside the model's reasoning for this single turn and is forgotten as soon as the next turn starts.

This sounds primitive next to the plan-then-execute pattern, but it has serious advantages. The model never commits to a sequence it might have to walk back. Each step is decided with the maximum available information — every prior tool result is in the context when the next step is chosen. There is no "the plan was wrong" failure mode, because there is no plan to be wrong about.

Used for: exploration, conditional logic, situations the model cannot anticipate before seeing some output. A debugging session where step one is "grep for the error string" — what step two should be depends entirely on what step one finds. A research task that follows citations wherever they lead. A customer-service flow where the user can change topic at any turn.

Incremental planning fails differently. It can wander. Without a written plan to walk against, the loop can spend ten turns chasing a thread that turned out to be a dead end, with no scaffolding to notice the drift. The harness has nothing to compare progress against. Step-level evals help — but the deeper guardrail is keeping the loop short and giving the model an explicit "stop and report" tool for when it's not making progress.

The choice between plan-then-execute and incremental is not binary. Most real agents are mostly incremental, with a planning step injected when the task is big enough to warrant it. A coding agent might run incrementally for small fixes, but produce an explicit plan when the user asks for a multi-file refactor. The harness decides which mode to enter based on the task; the model is asked to plan only when planning earns its keep.

Task graphs (DAGs)

Plan-then-execute treats the plan as a linear list. Real work often has structure the linear view loses: some steps depend on others, some are independent and could run in parallel, some are gates that must finish before anything downstream can start. The data structure that captures this honestly is a directed acyclic graph — a DAG of tasks.

The planner's output is no longer a numbered list but a graph: nodes are tasks, edges are dependencies. The harness performs a topological sort to determine execution order; independent branches run in parallel; the result of each node becomes available to its downstream nodes when they execute. Steps that have no dependency on each other no longer wait in line behind unrelated work.

This matters for two reasons. Wall-clock time: a five-step linear plan where steps two and three are independent doesn't need to take 5× the time of one step. Failure isolation: if step C fails and step D didn't depend on it, D can still run; only the descendants of C are blocked. A linear plan would have to abort the whole sequence.

FIG. 01

A five-node task graph. A fans out into B and C, which run in parallel; both feed D; D feeds E. The harness executes in topological order, parallelising where the graph allows.

DAGs are not free. The planner has to produce a valid graph — no cycles, every dependency named, every output schema matched to the downstream input. The harness has to schedule, run nodes concurrently, collect their results, and reason about partial failure. For a five-node graph this is straightforward; for a fifty-node graph it starts to resemble a workflow engine, and you are better off using one than writing it yourself.

Reach for DAGs when the task has obvious parallelism that a linear plan would waste, or when partial failure recovery matters. Skip them when the work is genuinely sequential — a graph with one path through it is just a linear plan with extra ceremony.

Hierarchical agents

The patterns so far assume one model doing both the planning and the execution. Hierarchical agents split those roles across separate model invocations, each operating at a different "altitude." A planner agent sees the whole goal and produces the structure: the steps, the dependencies, the success criteria. An executor agent sees one step at a time and carries it out, with no context on the larger plan beyond what it needs to do its piece. A reviewer agent sees the executor's output and the planner's intent and decides whether the step succeeded.

The benefit is specialisation by context. The planner's context is small and focused on structure; it does not get cluttered with tool outputs. The executor's context is small and focused on one task; it doesn't have to hold the whole plan in mind. The reviewer's context is the executor's output plus the success criteria. Each agent operates with less noise than a single agent would.

This beats one deeper agent when: the task has a clear hierarchy (a brief that breaks into chapters that break into sections), the steps are independent enough that an executor doesn't need full plan context to do its piece, or you want to introduce a review gate that is structurally separate from the work itself. The separation is the point — a reviewer that lives inside the executor's context is just self-checking, with all the blind spots that implies.

It loses to a single agent when the task is small enough that the coordination overhead exceeds the specialisation benefit. Three model calls to do what one call could have done is not a win. Hierarchical patterns earn their keep on long, structured work — Primer 8 picks this up under multi-agent patterns and goes deeper.

Recovering from a bad plan

This is the single hardest part of any planning system, and the part most builders under-invest in. A plan is a hypothesis. The hypothesis can be wrong — the model wrote it before it saw any tool outputs, working from its priors about the task. When the first real result lands and contradicts the plan, the harness has to do something about it. The default behaviour of a naive planner is to keep executing anyway, because the loop only knows about steps, not about whether the steps still make sense.

The discipline has three parts.

Detect early. Run a step-level eval after every step, not just at the end. The check can be lightweight — did the tool return the shape we expected, does the result satisfy the success criterion the planner wrote for this step, does the model itself flag the result as off. The check has to be cheap enough to run every step and strict enough to actually catch drift. A check that always returns "looks fine" is worse than no check, because it creates the illusion of supervision.

Replan when the plan is wrong, fall back to incremental when replanning fails twice. If a step fails, hand the planner the original goal, the completed steps, and the failure, and ask for a revised plan from this point. If the revised plan also fails, the task is more exploratory than the planner thought — drop back to the incremental loop from Section 3 and let the model decide the next step on the fly. There is no shame in switching modes. The shame is in pretending the original plan was still valid after step two contradicted it.

Never silently push through. When detection fires and recovery fails, the loop must stop and report — not continue with degraded confidence, not "try the next step anyway," not produce a partial result and label it complete. The honest output is "I ran steps one to three, step four failed, here is what I have so far, and here is why I am stopping." That output is useful. A wrong answer presented as a right one is not.

RESILIENCE

A bad plan that keeps executing is worse than no plan at all.

An incremental loop that wanders for ten steps wastes ten tool calls. A bad plan that runs to completion produces an authoritative-looking result that is wrong, often without anyone noticing until it lands downstream. Detection and graceful failure are the price of admission for planning. If you cannot afford them, stay incremental.

The deeper point: planning shifts where errors hide. Incremental loops fail visibly — they wander, they loop, the user sees them flailing. Planned executions fail invisibly — they march through a wrong plan and present a confident result. The first kind of failure is annoying; the second kind is dangerous. Build the detection and recovery before you build the planning, not after.

What this connects to

Planning sits between the per-step craft of the agent loop and the per-system craft of multi-agent design. The pointers worth carrying forward:

  • Primer 5 — The Agent Loop — the incremental loop is the baseline. Planning is what you add when the task outgrows it. Every planning pattern in this primer reduces, in its degenerate case, to the canonical loop from Primer 5.
  • Primer 6 — Memory & State — a plan is itself a kind of state the harness must carry across steps. Where it lives, how it gets updated, what survives a replan — these are memory-design questions the moment you stop running purely incrementally.
  • Primer 8 — Multi-Agent Patterns — hierarchical agents from Section 5 are the next altitude up. Primer 8 takes the planner/executor/reviewer split further: into specialised agents, message buses between them, and the orchestration patterns that hold them together.
  • Primer 11 — Observability & Traces — planning bugs are trace-bugs. The plan is a structure; the execution is a sequence; the gap between them is what your traces have to surface. A planning system without a trace view is a planning system you cannot debug.

Planning is a tool, not a default. Most agent tasks are best served by the incremental loop. Reach for planning when the work is big enough, structured enough, and slow-failing enough that committing a sequence up front earns its keep. Then build the detection and recovery before you ship the planner.
