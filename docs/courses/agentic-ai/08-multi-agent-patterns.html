The temptation, and the trap

The pitch is intuitive and almost always wrong. You have a single agent that struggles with a hard task — researching a market, drafting a long document, reviewing code across many files. The instinct is to split the work: one agent researches, another writes, a third reviews. Each can be smaller, more focused, easier to prompt. Add a coordinator, and you've turned a hard single-agent problem into a clean little org chart. Surely this is more capacity.

It almost never is. What you have actually added is coordination cost — and unlike a human team, where coordination cost is dwarfed by the time it takes humans to do the underlying work, with agents the coordination IS most of the work. Every hand-off is a context serialization. Every dispatch is a new system prompt, new tool definitions, new turns burned getting an agent oriented before it produces its first useful token. The conductor has to read each worker's output and decide what to do next, often using more tokens than the workers used to produce their parts.

The economics are unforgiving. A single agent loop with twelve tool calls might run 40,000 tokens end to end. The same task split across an orchestrator and three workers, each with its own loop, will routinely run 200,000 tokens — five times the cost for a result that is, in most cases, no better and sometimes meaningfully worse. The orchestrator drops context the workers had. The workers each rediscover facts the others already established. The seams between agents leak information and pile on tokens.

If you can do it with one agent, do it with one agent.

This is the default, and it should be a strong one. Reach for a single agent first. Give it the tools, give it the context budget, give it the planning capacity from Primer 7. If it cannot do the work, ask why specifically before reaching for multi-agent. Is it because the task genuinely has parallelism the model cannot exploit alone? Is it because the model's reasoning fails at a particular sub-step and you need a specialist? Or is it because the prompt is bad, the tools are wrong, or the context isn't being managed? Most of the time the answer is the third one. Fix that first.

Multi-agent is real and it has its place. The rest of this primer maps the patterns that work and names the failure modes. But the test before adopting any of them is the same: would a tighter single-agent solution get you there? If yes, do that instead.

Orchestrator + workers

The orchestrator-and-workers pattern is the most defensible multi-agent shape, and the one most production deployments converge on when they need multi-agent at all. One conductor agent decomposes the task, dispatches sub-tasks to specialist workers, integrates their outputs, and decides what to do next. The workers do not talk to each other. They talk only to the orchestrator.

This pattern earns its cost when the work shards cleanly. "Research five competitors" is the canonical example: the orchestrator splits the list, dispatches one worker per competitor with the same research prompt, and the workers run in parallel. There is no shared state to coordinate, no order dependency, and the synthesis step at the end is cheap because each worker returns a structured summary. The wall-clock speedup is real, the token cost overhead is bounded, and the result quality is comparable to a single agent doing the same work serially — sometimes better, because each worker had a tighter context.

One conductor; many specialists. Workers do not talk to each other — only to the orchestrator. Hand-offs in both directions: dispatch down, results up.

Where it fails: when the sub-tasks are not independent. The moment one worker needs another's output to do its job, you are no longer running parallel workers — you are running a pipeline, badly, with a hub in the middle that has to broker every dependency. The orchestrator's context balloons. The serial path through the dependency graph dominates wall-clock time. You would have been better off with hand-off (next section) or a single agent.

The orchestrator itself is the highest-risk part of the pattern. It needs enough context to make good dispatch decisions, which means it tends to accumulate state across the whole job — every worker's output, every retry, every revision. If you are not careful, the orchestrator's context is the union of every worker's context plus the synthesis overhead. Token budgets here can run away. The discipline is to keep the orchestrator's prompt and state lean: decompose, dispatch, integrate, decide — not narrate, repeat, or re-explain.

A useful rule: if the orchestrator's context is larger than what a single agent would have needed for the whole task, you have not split the work, you have copied it.

Hand-off (pipeline)

Hand-off is the linear-chain pattern. Agent A produces an output; that output becomes Agent B's input; B's output becomes C's input. There is no central conductor — each agent's job description includes "when you are done, pass to the next stage." Research, then draft, then review. Triage, then route, then resolve. Transcribe, then translate, then summarize.

This pattern earns its cost when each stage has a genuinely different specialty — different tools, different reasoning style, different success criteria. A research agent is good at searching and grounding claims; it is mediocre at long-form writing. A drafting agent does the inverse. A review agent's job is to find what the first two missed. Each stage can have a tighter system prompt, a smaller tool set, and a clearer success criterion than a single agent trying to be all three.

The defining advantage over orchestrator-and-workers is that the hand-off pattern has no synthesis problem. Each agent's output is the next agent's input — no integration step, no risk of the orchestrator misreading three parallel results, no token tax on a hub that has to hold everything. The cost is that you lose parallelism. Hand-off is fundamentally serial: the second agent cannot start until the first finishes.

Where it fails: when the boundaries between stages are blurry. If the research agent needed to consult the reviewer mid-flight ("is this source acceptable?"), or if the writer wants to send a question back to research ("what was that figure again?"), you have a feedback loop, not a pipeline. Hand-off does not handle feedback loops well — each backward step is an expensive retry through stages you already paid for. If your workflow has real back-edges, either flatten them into a single agent that can revisit its own work, or move to orchestrator-and-workers where the hub can broker the feedback.

The other failure mode is context loss across the seams. Agent A had rich context — search results, intermediate notes, dead ends it ruled out. Agent B receives only A's final output. The reasoning behind that output is gone. If B's downstream decision depends on something only A would have known, B will get it wrong. The fix is to include structured rationale in the hand-off payload — not just the answer but the supporting evidence the next stage might need to interrogate. This costs tokens on the hand-off but saves errors downstream.

Hand-off works best for workflows that are already pipeline-shaped in the business — a content production line, a triage-then-resolve service desk, a transcribe-then-translate stack. It works badly for tasks that are pipeline-shaped only in your head because you wanted them to be.

Debate

Debate is the pattern where two or more agents argue opposing or differing positions on the same question, and a judge agent synthesizes a final answer. The structure exploits a real weakness in single-agent reasoning: a confident model tends to commit to its first hypothesis and rationalize forward, missing the cases where it was wrong. Forcing the model to argue against itself — or against a sibling agent — surfaces the errors the original answer hid.

The technique is most useful for high-stakes calls where being confidently wrong is much worse than being correctly uncertain. A model that says "the contract is enforceable" with one paragraph of reasoning, when the right answer is "probably enforceable but with three named risks," is a danger in production. A debate harness routinely catches that kind of error: one agent argues enforceable, another argues unenforceable, the judge synthesizes the conditions and risks. The output is longer, slower, more expensive — and meaningfully more reliable on the cases that matter.

PYTHON 
from anthropic import Anthropic

client = Anthropic()
MODEL = "claude-sonnet-4-6"

def argue(side: str, question: str, prior: str) -> str:
    system = (
        f"You are arguing the {side} position. "
        "Be specific. Cite the strongest reasoning. "
        "Identify the weakness in the opposing argument."
    )
    user = f"Question: {question}\n\nOpposing argument so far:\n{prior}"
    return client.messages.create(
        model=MODEL, max_tokens=512, system=system,
        messages=[{"role": "user", "content": user}],
    ).content[0].text

def judge(question: str, pro: str, con: str) -> str:
    system = (
        "You are the judge. Read both arguments. "
        "Produce a synthesis: the answer, the conditions, "
        "and any remaining uncertainty. Do not pick a side mechanically."
    )
    user = f"Q: {question}\n\nPRO:\n{pro}\n\nCON:\n{con}"
    return client.messages.create(
        model=MODEL, max_tokens=1024, system=system,
        messages=[{"role": "user", "content": user}],
    ).content[0].text

def debate(question: str, rounds: int = 2) -> str:
    pro, con = "", ""
    for _ in range(rounds):
        pro = argue("PRO",  question, con)
        con = argue("CON",  question, pro)
    return judge(question, pro, con)

Two warnings. First, debate is expensive — roughly four to six times a single-agent answer for the same question, depending on round count. Use it for the calls where you would pay a senior reviewer to check a junior's work, not for routine queries. Second, debate can fail by converging on a plausible-but-wrong synthesis when both arguers share a blind spot. The judge is not a ground-truth oracle. If the two agents are the same model with different system prompts, they share the same training distribution and the same systematic errors. For the highest-stakes cases, vary the model, vary the system prompt, or include a retrieval step that grounds at least one side in external evidence.

Debate is a sharp tool. It is not the default. It is the pattern you reach for after you have identified specific failure modes — overconfidence, omitted risks, missed counter-arguments — that single-agent reasoning will not catch, and after you have decided the cost is worth the lift.

Swarms (use cautiously)

The swarm pattern runs N independent agents on similar or identical tasks and aggregates their results — by majority vote, by weighted score, by taking the union, by picking the best on a rubric. It is the agentic version of ensemble methods in classical ML: more samples, less variance, hopefully a better answer than any single agent would have produced.

Swarms work for a narrow band of problems where (a) the task has high variance across runs, (b) there is a cheap way to score or merge outputs, and (c) the aggregation step adds real signal rather than just averaging noise. Classification with self-consistency, generating multiple candidate solutions for a code task and picking the one that passes the tests, sampling multiple search queries and merging the retrieved set — these are the cases where swarms help.

The trap is that swarms look like parallelism, and parallelism looks free. It isn't. Running ten agents on the same task is ten times the token cost. The wall-clock time may be the same as a single agent if you can run them in parallel, but the bill is real and the carbon footprint is real. Worse, if the aggregation step is sloppy — a naive majority vote on a generative task, for instance — you can spend ten times the tokens to get an answer no better than one.

Most "swarm" architectures in the wild fail this test. They run multiple agents because it sounds impressive, not because the variance-reduction argument holds for the task. Before you build one, ask whether you have a real scoring function — a unit test, a rubric, a structured evaluator — that can pick the best output. If you do not, you are paying N times the cost to roll the dice N times.

When multi-agent is the wrong answer

The default position is that multi-agent is the wrong answer. This is not a contrarian pose — it is what the cost and reliability numbers say. Before you build a multi-agent system, walk through the cheaper alternatives and convince yourself each one is genuinely inadequate.

A deeper single-agent loop. Give the agent more turns. Raise the planning depth from Primer 7. Add a verification step at the end where the same agent checks its own work against a rubric. Most "we need multiple agents" arguments collapse when you allow the single agent twenty turns instead of eight. The loop is doing the work; you were not letting it iterate enough.

Better tools. Many multi-agent designs are compensating for tool gaps. The "researcher agent" exists because the main agent's search tool returns garbage. The "writer agent" exists because the main agent has no structured-output tool. Build better tools — a real retrieval pipeline, a structured-output schema, a domain-specific calculator — and the rationale for splitting agents often evaporates. The work moves into the tool layer where it belongs.

RAG. A surprising number of multi-agent designs are reinventing retrieval. The "domain expert agent" is a poor approximation of a vector index over the domain documents. The "policy enforcement agent" is a worse version of including the policy document in the retrieved context. Use retrieval. It is cheaper, faster, and easier to debug than a sub-agent that exists only to fetch knowledge.

Prompt and context discipline. Many failures blamed on "the single agent can't handle it" are actually context-engineering failures from Primer 2. The agent's context is bloated with stale tool results, irrelevant history, and over-broad system prompt. It is not under-powered. It is buried. Clean the context first.

DISCIPLINE

Multi-agent is the answer when the problem really is multi-agent shaped.

If the work has genuine parallelism that a single sequential loop cannot exploit, or distinct specialties that need different tools and different prompts, or a structured debate-like reliability requirement — then multi-agent earns its keep. Otherwise it is overhead dressed as architecture. Build the single agent first, push it hard, and let the move to multi-agent be the reluctant choice it should be.

One practical heuristic: if you cannot explain, in one sentence, why a single agent cannot do this job, you do not yet have the case for multi-agent. The sentence needs to name a specific constraint — parallelism, specialty, reliability — that a single loop cannot satisfy. "It would be cleaner" is not such a sentence. "It feels more scalable" is not such a sentence. Multi-agent is your last move, not your first.

What this connects to

Multi-agent patterns sit downstream of the loop and the planning primitives, and upstream of the implementation primitives. The pointers worth carrying forward:

  • Primer 5 — The Agent Loop — every "agent" in a multi-agent system is a loop. Multi-agent does not transcend the loop; it composes loops together. Most multi-agent failures are loop failures multiplied by the number of agents.
  • Primer 7 — Planning & Task Decomposition — orchestrator-and-workers is planning made concrete: the orchestrator is doing the decomposition, the workers are executing the steps. If you can do the decomposition inside a single agent's planning step, you usually should.
  • Primer 9 — Sub-Agents, Skills & Harnesses — sub-agents are the implementation primitive for the patterns in this primer. A "worker" in an orchestrator-and-workers system is, in practice, a sub-agent invocation. Primer 9 covers how harnesses give you sub-agents, skills, and the supporting machinery without the bespoke wiring most teams build the first time.

The throughline is restraint. Multi-agent is a real architectural option, but it is one you should approach skeptically. The right question is not "how do I split this across agents?" It is "have I exhausted what one agent can do?" When the answer is genuinely yes, pick the pattern that matches the work, expect the coordination cost, and move on. When the answer is no, go back and use the cheaper tool.
