The vocabulary

The agent ecosystem has overloaded almost every word it uses. "Agent" means a chatbot in one repo and an autonomous research loop in the next. "Skill" means a markdown file in one SDK and an entire microservice in another. Before any of the patterns in this primer are useful, the four words have to be pinned down. The rest of the course returns to these definitions, so it is worth spending a paragraph on each.

A tool is a function. It takes structured arguments, does something — calls an API, runs a query, writes a file — and returns a result the model can read on the next turn. That is the whole definition. A tool has no prompt, no judgment, no memory. It is the executable atom of the system. get_weather(city) is a tool. search_kb(query, top_k) is a tool. So is send_email(to, subject, body), even though it has consequences in the outside world.

A skill is a tool plus its prompt plus its eval. Where a tool is a raw function, a skill is the function packaged with the instructions for when to use it, the schema for how to call it, and the test set that proves it still works. A skill is what you actually ship and reuse. Two teams sharing a "search_kb tool" will use it differently and get different results. Two teams sharing a "search_kb skill" — same tool, same canonical system fragment, same eval set — get the same behavior. The skill, not the tool, is the unit of portability.

A sub-agent is a skill that has its own loop. Most skills are stateless: the model calls them, they return, the conversation continues. A sub-agent is the case where the skill is itself an agent — it takes a goal, runs its own model calls and tool calls until it produces a result, and returns that result to the caller. The boundary is the loop. If the unit runs at most one model call per invocation, it is a skill. If it runs many, it is a sub-agent.

A harness is the code around all of this. The model does not call tools by itself; something has to read the model's tool-use request, dispatch the actual function, capture the result, format it back into a message, and call the model again. That something is the harness. It also owns the things the model is not responsible for: timeouts, retries, budget caps, tracing, auth, error surfaces. The harness is where engineering rigor lives. Most failures attributed to "the model" are actually failures in the harness.

Four primitives. Tool, skill, sub-agent, harness. Everything else in the agent stack — orchestrators, planners, controllers, routers — is some arrangement of these. The rest of this primer is the operational view: how each one is built and what crosses the boundary between them.

Tool registries

The first mistake builders make with tools is treating them as strings. A tool gets defined inline next to where it is used, its name is hardcoded into the prompt, its schema lives in a dict literal three files away from its implementation, and its auth scopes are implicit in whatever credentials the process happens to hold. This works for the demo. It does not survive contact with a second tool, a second team, or a second environment.

The fix is to treat tools as objects in a registry. Each tool is a typed entity with a name, a JSON schema, a callable implementation, a version, an owner, and an auth scope. The registry is the lookup table the harness consults at runtime: given a tool-use request from the model, find the matching entry, check the caller is authorized for it, validate the arguments against the schema, run the implementation, return the result. The model never sees the registry directly — it only sees the subset of tool definitions the harness chose to expose this turn.

The pattern below is deliberately small. Production registries add concurrency, structured errors, observability hooks, and per-tenant filtering. The shape stays the same.

PYTHON 
from dataclasses import dataclass
from typing import Any, Callable

@dataclass(frozen=True)
class Tool:
    name: str
    version: str
    description: str
    schema: dict          # JSON schema for arguments
    scopes: frozenset     # auth scopes required to invoke
    impl: Callable[..., Any]

class ToolRegistry:
    def __init__(self) -> None:
        self._tools: dict[str, Tool] = {}

    def register(self, tool: Tool) -> None:
        key = f"{tool.name}@{tool.version}"
        if key in self._tools:
            raise ValueError(f"duplicate tool: {key}")
        self._tools[key] = tool

    def expose(self, names: list[str], caller_scopes: frozenset) -> list[dict]:
        """Return the tool definitions the model will see this turn."""
        out = []
        for key in names:
            tool = self._tools[key]
            if not tool.scopes.issubset(caller_scopes):
                continue
            out.append({
                "name": tool.name,
                "description": tool.description,
                "input_schema": tool.schema,
            })
        return out

    def invoke(self, name: str, version: str, args: dict, caller_scopes: frozenset) -> Any:
        tool = self._tools[f"{name}@{version}"]
        if not tool.scopes.issubset(caller_scopes):
            raise PermissionError(f"missing scopes for {name}")
        return tool.impl(**args)

Three properties matter and are worth naming. Tools are versioned — search_kb@1.2 and search_kb@2.0 can coexist while callers migrate, and a regression in 2.0 does not break callers pinned to 1.2. Tools are scoped — the registry knows which scopes a caller holds and silently omits tools they cannot use, rather than exposing them and failing on invocation. Exposure is per-turn — the harness decides which tools the model sees this turn, not which tools exist in the system. A registry of forty tools can be presented as four to the model on any given call.

Once tools are objects, the rest of the stack stops fighting them. The harness can log every invocation. The skill package can reference tools by name and version. The eval suite can mock specific tools without monkey-patching the world. None of that is possible when the tool is a string the model is asked to incant.

Skills as the unit of reuse

Raw tools do not compose well. Two agents can call the same search_kb tool and behave completely differently — one because its system prompt told it to search before answering, one because its prompt told it to search only when uncertain. The behavior lives in the prompt, not in the tool. Sharing the tool without the prompt shares almost nothing useful.

The unit that does compose is the skill: the smallest portable bundle of "model knows when to use it, model knows how to call it, the team can prove it still works." Concretely, a skill is four things shipped together:

  • A prompt fragment — the system-prompt instructions that tell the model when this skill is appropriate, what it does, and any pitfalls. Usually one paragraph.
  • A tool (or small set of tools) — registered with a name and version, exposed when this skill is in scope for the turn.
  • A schema — the JSON schema the model sees, plus any output schema the harness validates against before passing results downstream.
  • An eval — a small test set the skill must pass before it ships and before any change to its prompt or tool is accepted. The eval is the skill's specification; the prompt and tool are the implementation.

Once skills are the unit, composition gets simple. An agent is "a system prompt plus a list of skills." A new agent built for a different team is "the same skills, a different system prompt." A regression in a skill is caught by its own eval before it cascades into every agent that uses it. A skill that gets adopted by three teams is upgraded in one place. None of that holds when the unit of reuse is a tool — the prompt that taught the model when to use the tool is what made the tool useful, and that prompt was attached to an agent, not to the tool.

A working test for whether something is a skill: can a second team adopt it without reading your agent code? If yes — they get the prompt fragment, register the tool from your package, and run the eval to confirm it works in their environment — it is a skill. If they have to copy paragraphs out of your system prompt and guess at when the tool fires, it is not a skill yet, no matter what the directory is called.

This is why later primers treat the skill, not the tool and not the agent, as the natural unit of evaluation. The eval lives with the skill, travels with the skill, and gates changes to the skill. Without that coupling, agent quality is something you measure end-to-end and cannot localize when it drops.

Sub-agent dispatch

Most skills resolve in one model call: the model decides to use the skill, the harness invokes a tool, the result comes back, the conversation continues. A sub-agent is the case where resolving the skill itself takes a loop. The model calls a skill named research_company; behind that call, a separate loop runs — its own model, its own system prompt, its own tools — until it produces a synthesized result, which is what the outer model finally sees.

Sub-agents earn their cost in two situations. The first is when the work has its own multi-step reasoning that would clutter the parent's context — a research task that needs to issue six searches, read three documents, and synthesize a paragraph. Running that inside the parent loop pollutes the parent's history with eighteen turns of intermediate state; running it as a sub-agent collapses all of that into a single tool result. The second is when the work needs a different model, a different system prompt, or a different tool set than the parent — a SQL-writing sub-agent with database tools and a precise schema-aware system prompt that the parent agent has no business carrying.

FIG. 01

The sub-agent has its own loop. The parent sees a single skill call; the boundary crossing hides an entire inner conversation.

The crucial property is the boundary. The parent agent does not see the sub-agent's intermediate turns. It sees a single tool result — the synthesized output. The sub-agent's context is private; it can use a different model, hold a different system prompt, and run for fifteen turns without any of that leaking back. From the parent's perspective, calling a sub-agent is no different from calling any other tool.

This is also where sub-agents get expensive and dangerous. A sub-agent that takes thirty seconds to return is a tool call that blocks the parent loop for thirty seconds. A sub-agent that loops indefinitely is a leak. The harness — not the model — is responsible for putting a turn cap, a time cap, and a token cap on every sub-agent invocation, and for surfacing partial results when those caps trigger. Treat the sub-agent as a process you are dispatching, not a function you are calling, and the engineering follows.

Harnesses: the code around the loop

The loop itself is a few lines. Call the model with the current message stack; if it asks for a tool, run the tool, append the result, call the model again; stop when it returns a final answer. Anyone can write that in an afternoon. What separates a demo from a production agent is everything around that loop. That everything is the harness.

A useful frame: the loop owns the conversation; the harness owns the system. The model is responsible for the next decision; the harness is responsible for everything the model is not. When something goes wrong in production — a tool times out, a model returns malformed JSON, a sub-agent leaks tokens, a downstream system rate-limits — the harness either handled it gracefully or the user saw a stack trace. There is rarely a middle ground.

In the loopIn the harness
Choosing the next toolLooking up the tool in the registry and validating args
Composing the natural-language responseFormatting tool results back into messages the model can read
Deciding when the task is doneEnforcing the turn cap, time cap, and token cap
Generating a synthesis from sub-agent resultsDispatching the sub-agent and supervising its budget
Asking the user a clarifying questionRetries, timeouts, structured-output validation
(Nothing else)Tracing, auth, observability, error surfaces, partial results

Notice the asymmetry. The model's responsibilities are short and qualitative. The harness's responsibilities are long, quantitative, and unglamorous. This is why most "the agent failed" post-mortems trace back to harness gaps rather than model gaps — the model produced a reasonable next step on a partial context, and the harness had no plan for what to do when that step took longer than expected, or returned a 503, or used 90% of the token budget on the first turn.

Two harness disciplines matter more than the rest. Every external call is wrapped — tool invocations, model calls, sub-agent dispatches — with a timeout, a retry policy, and a structured error that the loop can read on the next turn rather than crash on. Every loop run has a budget — wall clock, model tokens, tool invocations — and the harness enforces it from outside the model, returning a partial result with a clear cap-hit message rather than letting the loop spin. These two together separate a tutorial agent from one you can let a user touch.

Cross-SDK patterns

The four primitives — tool, skill, sub-agent, harness — are not specific to any single SDK. They appear across every credible agent stack, with different vocabulary attached. Once you can map a new framework onto these four words, learning the framework is mostly learning its terminology and its defaults.

A quick translation, by SDK family:

  • Claude Agent SDK. Tools are @tool-decorated functions registered with the runtime. Skills are markdown-defined packages that bundle a prompt fragment, a tool set, and resources. Sub-agents are explicit — you spawn a child agent with its own system prompt and tool set, and it returns its result to the parent. The harness is the SDK runtime itself, which owns the loop, budgets, and tracing.
  • LangChain / LangGraph. Tools are Tool objects or @tool-decorated functions. Skills don't have a single name — they are implied by the combination of a tool and the prompt template that introduces it. Sub-agents appear as nested graphs or as agents-as-tools. The harness is split between the framework's executor and whatever you wrap around it.
  • OpenAI Assistants / Responses. Tools are function-calling schemas. Skills are not a first-class concept — you assemble them yourself out of system prompts and tool selections. Sub-agents are handled via the Assistants API's nested runs or by composing your own. The harness is largely on you.
  • Home-rolled stacks. Tools are whatever your function-calling adapter accepts. Skills, sub-agents, and the harness are whatever you've built. The risk in home-rolled stacks is not that the primitives are absent — they always emerge — but that they emerge implicitly, get named inconsistently, and end up entangled with each other.
PATTERN

Names differ across SDKs. The four primitives don't.

When a new framework lands on your desk, the first useful question is not "what is its abstraction?" — it is "where does this framework put each of the four primitives, and what does it make easy versus hard?" The frameworks that succeed are the ones that make the four primitives explicit and orthogonal. The ones that struggle conflate them — usually fusing skill and sub-agent, or merging harness concerns into the model's prompt.

The practical consequence is that you can design an agent independent of SDK and port it later. Decide which tools the system needs, which combinations are skills, which skills warrant their own loop as sub-agents, and what the harness has to guarantee. Once those decisions exist, mapping them onto any specific SDK is mechanical. Decide them inside an SDK's vocabulary and you will rewrite them every time you change SDKs.
