What a trace is

A trace is the full record of one run of your agent: every model call, every tool call, every retry, with timings, inputs, outputs, and the parent-child relationships that connect them. It is the receipt the system writes for itself. Without it, an agent is a black box that either produced the right answer or did not; with it, you can replay what happened, where it spent its time, which tool returned the bad value, and which model call hallucinated the wrong city.

It helps to be precise. A trace is not "logs." Logs are unstructured strings emitted in execution order; you grep them. A trace is a structured, hierarchical object: a tree of timed units with named parents. You can ask a trace "show me everything that happened inside the model call that produced this final answer" and it answers in one query, because the structure is in the data. Logs cannot do that without a heroic amount of regex.

For a single-shot LLM call, the trace is almost trivial — one unit of work, one input, one output. For an agent loop, a trace is where complexity lives. Twenty turns, each one calling the model, each model call possibly emitting tool calls, each tool call possibly spawning a sub-agent with its own model calls underneath. A non-trivial run produces hundreds of nested units of work. The trace is how you make sense of them at all.

Treat traces as the unsung hero of agent engineering. Evals get the credit because they give you a number. Traces are what let you fix the number. Most teams under-invest in tracing until the first late-night production incident teaches them, expensively, that they cannot debug what they cannot see. Build the tracing layer early, with the harness, not after the first incident.

An agent without traces is a coin flip you cannot replay.

The rest of this primer is the operational version of that idea: how traces are structured, what to capture, what to keep, and how to turn a trace into a fixture you can ship into your eval suite.

The span model

The unit of a trace is the span. A span represents one bounded piece of work — one model call, one tool execution, one HTTP request, one sub-agent run. Every span has the same minimal shape: a name, a start time, an end time, a parent reference, an ID of its own, a status (ok or error), and a bag of attributes that describe what happened. The model is borrowed almost wholesale from distributed tracing — OpenTelemetry, Jaeger, that lineage — because the problem is exactly the same: many timed units of work, nested, that need to be reconstructed into one story.

Spans nest. The root span is usually the whole agent run — call it agent.run. Inside it sit the per-turn spans (one per iteration of the loop), and inside each of those sit the model call and the tool calls. A sub-agent that the parent spawns becomes its own subtree under the parent's tool-call span. The trace, as a whole, is a tree of spans rooted at agent.run.

That nesting is the load-bearing piece. With it, "show me the tool calls inside the model call that produced the final answer on turn 7" is a tree traversal. Without it — if you only have flat events — you are joining timestamps and hoping nothing else fired in the same millisecond. The tree is what makes a trace queryable.

FIG. 01

A span tree. The root span agent.run contains nested children; tool.search nests one level deeper inside model.call.

Two operational pieces fall out of this. First, every span needs a stable trace ID shared across the whole tree, and its own span ID within it. The trace ID is how you reconstruct one run; the span ID is how children point at their parent. Generate both at the root and propagate them through every call site. If a sub-agent runs in a separate process, the trace ID has to travel with it — through HTTP headers, message queues, whatever — or its spans show up as a separate trace and the story falls apart.

Second, attributes are where the real value lives. Span name and timing tell you what ran and how long it took. Attributes tell you why: the model ID, the prompt fingerprint, the token counts, the tool's arguments, the request body of the HTTP call. A span without attributes is a tombstone. The next section is mostly about which attributes to put on which span.

A trace viewer, drilling down

What a tracing UI actually shows you is a horizontal timeline of span bars, indented to reflect nesting, with a scrubber across the top. You click a bar; it expands; you see its attributes; you see its children; you click one of those; the viewer drills in. The animation below walks one cycle of that drill-down — root span, then the model call inside it, then the tool call inside that, then back out to the next sibling.

Watch the four phases. The scrubber sweeps across the top, the active span lights up at each level, and a small drill indicator appears whenever the viewer is "inside" a nested level. Pause whenever you want to dwell on a step.

Phase 1 — root. The viewer opens on the whole run. agent.run highlights. This is your entry point: total wall-clock time, total cost, the user's original request. You scan it and decide where to look next.

Phase 2 — drill into the model call. You suspect the model picked the wrong tool, so you open the model.call span. Now you see its attributes — the full prompt, the response, the token counts, which tool was selected.

Phase 3 — drill into the tool call. The model called tool.search. You open it. Now you see the exact query that went out and the exact result that came back. This is usually where the bug lives: the tool returned something the model could not interpret, or returned the right thing but slowly.

Phase 4 — zoom back out. You climb back to the parent level and look at the next sibling, executor.act. The trace lets you move both down and across — drilling into children, then stepping to siblings — without losing your place in the tree.

FIG. 02 · ANIMATED

A trace viewer drilling through one run. Root, then model call, then tool call, then back out to the next sibling.

The pattern works because the data is structured. The viewer is doing nothing exotic — it is rendering a tree and letting you focus on one node at a time. The only reason that experience is possible at all is that someone, upstream, instrumented the agent to emit the spans in the first place.

What to capture

The temptation, the first time you wire up tracing, is to capture everything. Every byte that crosses every boundary. It is the wrong instinct, and not just for cost reasons. A trace that captures everything is unreadable; the signal drowns in the noise, and the things that actually matter — the prompt fingerprint, the token count, the error code — are buried inside multi-megabyte payloads no human will ever scroll through.

Think of attributes in three tiers. Always capture: the structural metadata that makes the trace useful at all. Sample: heavy payloads that are sometimes load-bearing, but not on every span. Never capture: anything you would be unable to defend in a privacy review.

Always captureSampleNever capture
Span name, start/end, parent ID Full prompt text (hash always, body on error or eval) Raw PII (names, IDs, account numbers)
Model ID, model version Full tool input/output bodies API keys, OAuth tokens, session cookies
Input/output token counts Retrieved RAG passages Customer message bodies (in regulated contexts)
Tool name, arguments hash, latency HTTP request/response bodies Anything covered by a data-residency rule you'd violate
Error code, error class, retry count Intermediate state snapshots Free-form notes from prior internal incidents

The "always" column is small on purpose. Those eight or ten attributes per span are enough to answer most operational questions — what ran, when, for how long, with what model, with how many tokens, and did it error. Multiply by hundreds of spans per run and millions of runs per month, and you can afford to keep all of it forever.

The "sample" column is where engineering judgement lives. The full prompt body is huge and almost always redundant — same system prompt every turn, same tool definitions — but you need it on the runs where something went wrong. The discipline is to store a content hash on every span (cheap, small, lets you join across runs) and the full body conditionally: on errors, on flagged outcomes, on a random sample for eval-set mining.

The "never" column is non-negotiable. PII, secrets, regulated payloads. The cost of a trace store breach scales with the worst thing in any record; one mishandled customer message is enough to blow up the program. Redact at ingest, not in the viewer. The trace store should never have seen the thing you cannot afford to leak.

One practical move: build redaction into the instrumentation layer, not as a downstream cleanup pass. The span emitter should know that the field named account_number is replaced with its last four digits and the field named api_key is replaced with a tag of the token, before the span object ever leaves the process. Downstream redaction is a race you eventually lose.

Sampling and storage

Capturing every byte of every trace is uneconomic. Even for a mid-sized agent, full-fidelity tracing of every run produces terabytes a month. The question is not whether to sample — it is how to sample in a way that keeps the runs you actually need.

The principle is outcome-aware sampling. Errors are rare and high-value: keep all of them, full fidelity, indefinitely. Successes are common and mostly redundant: keep a representative sample, drop the rest after a short hot window. The math works out because errors are usually a single-digit percentage of traffic and successes are most of the volume.

A working sampling policy has three tiers:

  • Errored runs — keep 100%, full payloads, indefinitely. These are how you debug production and how you mine new eval cases.
  • Flagged runs — runs the agent itself marked as low-confidence, or runs an automated check flagged as anomalous (high latency, high cost, repeated tool calls). Keep 100%, full payloads, 30-90 days.
  • Successful runs — keep span structure and metadata always; keep full bodies on a 1-5% random sample. Drop or roll up the rest after a short hot window.
PYTHON 
import random

# Outcome-aware sampling: every byte for errors, a slice for successes.
SUCCESS_SAMPLE_RATE = 0.02  # 2% of clean runs get full payloads

def should_capture_full_payload(trace) -> bool:
    if trace.has_error:
        return True                        # always, no exceptions
    if trace.was_flagged_low_confidence:
        return True                        # eval candidate
    if trace.latency_ms > LATENCY_P99:
        return True                        # anomaly worth keeping
    if trace.cost_usd > trace.expected_cost * 3:
        return True                        # cost outlier
    return random.random() < SUCCESS_SAMPLE_RATE

# Storage tiering: hot for fast queries, cold for archive.
def storage_tier(trace) -> str:
    age_days = (now() - trace.started_at).days
    if trace.has_error:        return "hot"   if age_days < 90  else "warm"
    if age_days < 7:           return "hot"
    if age_days < 30:          return "warm"
    return "cold"

Tier the storage to match the access pattern. Recent traces — anything from the last week — sit in hot storage with fast indexing, because that is where on-call queries go. Older traces move to warm storage where queries are slower but still possible. Anything past 30-90 days goes to cold storage (object store, parquet, the cheapest tier you have) and is mostly there for compliance and the occasional deep retrospective.

One subtle thing worth flagging. Sample traces, not spans. If you randomly drop spans within a kept trace, the tree becomes unreconstructible — children pointing at parents that no longer exist, gaps in the timeline. Decide at the root span whether to keep the whole tree or none of it. Span-level sampling is the most common mistake teams make when they wire this up.

Debugging from a trace

The most valuable thing a trace gives you is not the immediate debug session — it is the next eval case. The diagnostic loop, when traces are wired up well, is a tight cycle: production bug → trace captures it → trace becomes a fixture → fixture becomes an eval case → fix lands → eval guards the fix forever.

Concretely, here is how it runs. A customer reports that the agent produced the wrong KYC verdict. You pull the trace by ID. You scrub through it — root, then the model call that emitted the verdict, then the tool call that retrieved the underlying screening result. The trace shows you that the screening tool returned a record with a name mismatch the model glossed over. The bug is in the prompt, not the tool: the model was not told to be paranoid about name mismatches.

You fix the prompt. Before you ship, you take the trace's inputs — the original customer record, the original screening result — and add them to your eval suite as a fixture. Now the eval suite checks that the fixed prompt produces the right verdict on this case. The next time someone touches the prompt and accidentally re-introduces the bug, the eval catches it on the CI run. The customer-reported bug becomes a permanent guard.

This is the pattern that compounds. Every production bug that survives more than one turnaround is a bug that should have been an eval. Every trace you save is a potential eval candidate. Traces are how evals get their fixtures; evals are how you stop regressing the same bug twice.

FIXTURE

Every prod bug becomes a trace, then a fixture, then an eval.

The diagnostic loop is short, and it has a defined exit. A bug surfaces; the trace replays it; you turn the trace into a fixture and add it to the eval suite; you fix the underlying cause; the next eval run guards the fix. Skip the fixture step and you will debug the same bug six months from now.

The trace also matters for anomaly detection, the other direction of the loop. Latency spikes, cost outliers, runs that hit the iteration cap, runs where the same tool got called five times in a row — these are signals worth automating. A trace-aware monitoring layer surfaces the anomalies; an engineer drills in; nine times out of ten, the anomaly is a real regression the eval suite did not yet cover, and the trace becomes the next fixture. The wheel keeps turning.

What this connects to

Observability is the connective tissue across the whole course. The pointers worth carrying forward:

  • Primer 5 — The Agent Loop — every tick of the loop is a span. The READ-DECIDE-EMIT-EXECUTE-APPEND cycle from Primer 5 is what the trace tree is recording, one iteration at a time. If the loop is the verb, the trace is the receipt.
  • Primer 10 — Evaluations for Agents — traces and evals are two halves of the same discipline. Evals tell you a regression happened; traces tell you where. Traces are also where eval fixtures come from, as §6 above lays out. Neither ships without the other.
  • Primer 12 — Safety, Guardrails & Prompt Injection — traces are how you catch prompt injections in the wild. A retrieved passage that suddenly contains "ignore prior instructions" shows up as a specific span attribute; an injected instruction shows up as a model behavior change traceable to the offending input. Without the trace, prompt injection is a ghost story; with it, it is a debuggable event.
  • Primer 13 — Cost, Latency, Caching — traces have the token math. Per-span input and output token counts, per-span latency, cache-hit attributes, model IDs — everything the cost and latency primer asks you to optimize is something the trace already records. Tune what you measure; you only measure what you traced.
