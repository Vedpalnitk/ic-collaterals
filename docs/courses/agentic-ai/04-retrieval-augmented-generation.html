What RAG is, what it isn't

Retrieval-Augmented Generation is one sentence: retrieve relevant text from somewhere, paste it into the model's context, ask the model to answer using that text. That is the whole pattern. Every elaborate RAG architecture you have seen — vector databases, embedding pipelines, reranker cascades, query rewriters — is plumbing in service of those three steps.

Holding that shape in mind clears up most of the confusion around it. Three consequences fall out directly.

RAG is a context-engineering pattern, not a model feature. The model is exactly the same model whether you wrap it in retrieval or not. You are not "training" anything. You are not changing weights. You are choosing what text to put in front of an unchanged model at the moment you ask it a question. The art is entirely in choosing well.

RAG doesn't make the model "know" anything new. The model still only sees what is in its context window for this single call. RAG puts the right text in front of the model at the right time so it has the material to answer from. Outside that call, the model has no memory of what was retrieved. The next call starts cold.

If retrieval is wrong, the model will confidently hallucinate using the wrong text. This is the failure mode that surprises teams. A retrieval system that returns plausible-but-irrelevant passages does not produce "I don't know." It produces a fluent, confident answer grounded in the wrong document. The model is doing its job — answer from the context — and the context is misleading. Retrieval quality is the load-bearing dependency. A great model on bad retrievals is worse than an okay model on great retrievals, because the great model is more persuasive about the wrong answer.

One common confusion worth heading off early: RAG is not an alternative to fine-tuning. They are orthogonal axes. Fine-tuning changes how the model speaks — its tone, its preferred format, the conventions of a domain. RAG changes what the model sees — the specific facts available to it for this call. You almost always want RAG first, because most real product needs are "the model needs to know about my documents," not "the model needs to talk in a specific voice." Fine-tuning is for the residual cases after RAG and prompting have done their work (see §6 for the full decision tree).

Held tightly to this shape, the rest of the primer is mechanical. Everything else is just how to retrieve well.

The retrieval pipeline

A production RAG system has five stages. Each does one job, each has its own cost/quality knob, and each is a place to over- or under-invest.

  1. Query. The user's question, possibly rewritten by the model first. Query expansion — turning "claims process" into "claims process, claim filing, FNOL, first notice of loss" — often does more for recall than a better embedding model.
  2. Embed. Turn the query into a dense vector using an embedding model. The same model must have been used to embed the corpus. Mismatched embedding models silently produce nonsense.
  3. Search. Find candidate passages. Two flavors: semantic (vector similarity, captures meaning) and lexical (BM25 or similar, captures exact tokens). Hybrid search runs both and fuses the results.
  4. Rerank. Score the top 50–100 candidates with a stronger model — usually a cross-encoder that reads (query, passage) together rather than embedding them independently. Keep the top 3–10.
  5. Context. Paste the reranked passages into the prompt with the user's question. The LLM answers.

The shape of the pipeline is the same everywhere. The places teams differ — and the places they fail — are in how much they invest in each stage.

The most common over-investment is at embed. Teams pay for the largest, slowest, most expensive embedding model on a corpus of fifty documents. The cost is real; the quality difference, at that scale, is invisible. Pick a competent embedding model and move on; the budget is better spent elsewhere.

The most common under-investments are at search and rerank. Teams ship semantic-only search on corpora full of technical jargon, brand names, product SKUs, or code identifiers — exactly the strings embedding models handle worst — and are surprised when retrieval misses obvious matches. Or they skip rerank entirely and ship the raw top-10 vector hits, which are often only "roughly relevant." Both are unforced errors. Hybrid search closes the jargon gap. A reranker closes the precision gap. Together they are usually a bigger quality jump than any model swap.

Chunking & overlap

You cannot retrieve a million-token document. You retrieve passages from it. Chunking is the upstream step that turns long documents into the small, retrievable units the embedding model and the search index see. Two parameters dominate the design:

  • Chunk size — typically 200–1,000 tokens. Smaller chunks make each retrieval more precise; larger chunks carry more context per hit.
  • Overlap — typically 10–25% of the chunk size. Adjacent chunks share content so that facts straddling a chunk boundary are captured by both neighbors.

Overlap is the parameter that surprises people the first time. Without it, fixed-size chunks tile the document with hard edges. If a sentence — or a fact, or a definition — happens to straddle a boundary, you split it. The first half lands in chunk N; the second half lands in chunk N+1; neither chunk contains the whole fact, and neither will retrieve well against a query that asks about it. Embedding similarity drops sharply for half-facts. The reranker, given a query and a passage with only half the answer, scores it low. The model never sees the material it needed.

Overlap fixes this by trading redundancy for boundary coverage. If adjacent chunks share 20% of their content, any fact shorter than that 20% window is fully contained in at least one chunk no matter where the boundary fell. The cost is that you index more total tokens; the benefit is that the failure mode at the boundary stops happening.

The three regimes have predictable trade-offs. No overlap: cheapest to index, fastest to search, splits facts at boundaries. Moderate overlap (~20%): small redundancy cost, robust against most boundary splits — the default for almost every production system. Heavy overlap (~50%): every fact lives in two or three chunks; retrieval becomes redundant rather than precise, and you start paying real costs at indexing and rerank time. A note on the parameters that matter less than people assume: the exact chunk size, within a wide band. 400 tokens vs 600 tokens almost never changes the headline retrieval metric. Use the natural unit of your corpus — paragraph, section, code block — and stop tuning. Where you do see headline movement is at the boundaries: paragraph-aware chunking, sentence-aware chunking, or code-block-aware chunking usually beats fixed-size chunking, because it puts boundaries where humans put them and overlap stops being a fix for an arbitrary problem.

Embeddings, similarity, and hybrid search

An embedding model maps a piece of text to a fixed-length vector — typically 256 to 3,072 dimensions. The training objective is roughly: text with similar meaning should map to vectors close together; unrelated text should map far apart. "Close" is almost always measured with cosine similarity — the cosine of the angle between the two vectors, ranging from -1 (opposite) through 0 (orthogonal) to 1 (identical direction).

Two things to internalize about embedding models that catch teams out.

First, different models produce different vector spaces. A vector from text-embedding-3-small and a vector from voyage-3 are not comparable — they live in different geometries with different basis vectors. You cannot mix them in a single index, and you cannot swap one for the other without re-embedding the entire corpus. Pick an embedding model deliberately and budget for the migration when you change it.

Second, embeddings encode the meaning the model was trained on. They are very good at the things English-language web text taught them: paraphrase, synonymy, topical similarity. They are worse at things the training data underrepresents: brand names ("Acuity"), product SKUs ("HSV-9012"), code identifiers (getUserAuthToken), rare technical jargon ("vacuolar H+-ATPase"), and acronym/expansion pairs ("KYC" vs "Know Your Customer"). For these strings, two embeddings of the "same" concept can sit far apart in vector space because the model never learned they were the same. Semantic search returns near-misses; the right document goes unretrieved.

This is the gap that lexical search fills. BM25 — the workhorse algorithm behind most full-text indexes for two decades — scores documents by exact and near-exact term matches, weighted by term frequency and inverse document frequency. BM25 has no idea what a word means; it has perfect recall on what a word is. For the SKUs and brand names and code identifiers, BM25 is the right tool.

Hybrid search is the obvious move: run semantic search and lexical search in parallel, get two candidate lists, fuse them into one. The simplest fusion technique — and a stubbornly hard one to beat — is Reciprocal Rank Fusion (RRF): every candidate gets a score of 1 / (rank + k) from each list, the scores are summed across lists, and the candidates are re-sorted by total. Documents that ranked well in either method bubble to the top; documents that ranked well in both bubble higher still. No tuning, no per-query weights, no learning step. Implement it once and it works.

PYTHON 
def hybrid_search(query, k=50):
    # 1. Semantic candidates
    q_vec = embed(query)
    semantic_hits = vector_db.search(q_vec, top_k=k)

    # 2. Lexical candidates (BM25)
    lexical_hits = bm25_index.search(query, top_k=k)

    # 3. Score-fusion (reciprocal rank fusion is the simplest)
    fused = {}
    for rank, hit in enumerate(semantic_hits):
        fused[hit.id] = fused.get(hit.id, 0) + 1.0 / (rank + 60)
    for rank, hit in enumerate(lexical_hits):
        fused[hit.id] = fused.get(hit.id, 0) + 1.0 / (rank + 60)

    return sorted(fused.items(), key=lambda x: -x[1])[:k]

The constant 60 in rank + 60 is the RRF smoothing factor; it stops rank-1 from dominating and is approximately the value the original paper landed on. You will almost never tune it. The output of hybrid_search is the input to the next stage — the reranker — which takes these 50 fused candidates and picks the 5 that actually deserve to be in the prompt.

Reranking: the under-appreciated step

The output of the search stage — semantic, lexical, or hybrid — is a list of 50 to 100 candidates that are "roughly relevant." The relevance signal is computed independently for each candidate: each was embedded once, scored against the query embedding once, and ranked. Nothing in that pipeline looked at the query and the candidate together. That is the gap a reranker closes.

A reranker reads the (query, passage) pair jointly. A cross-encoder — the canonical architecture — concatenates them into a single input and produces a single relevance score. Because the model sees both sides at once, it can pick up signals an independent embedding cannot: the passage uses the same entity as the query, the question word matches a definitional clause in the passage, the passage is structurally an answer rather than a topical neighbor. Cross-encoders are slower than bi-encoders by 10–100×, but you only run them on the top-50 candidates, not the top-1M corpus. The cost is bounded.

The framing that helps most: retrieval is recall; rerank is precision. The search stage's job is to make sure the right passage is somewhere in the top 50, even if it is at rank 47. The rerank stage's job is to move it from rank 47 to rank 1 and to discard the 45 irrelevant passages above it. Different stages, different objectives, different budgets.

QUALITY

Retrieve 50, rerank to 5, ship 5.

The single biggest jump in RAG quality usually comes from adding a reranker, not from a better embedding model or a smarter chunking strategy. Reranking is precision; retrieval is recall. Treat them as separate stages with separate budgets.

The economic case is more favorable than teams expect. Suppose your search stage returns 50 candidates per query and your reranker charges a small per-pair fee. Fifty rerank calls per user query is cheap — comfortably under a cent at typical 2026 prices. The expensive thing in the pipeline is the LLM call that produces the final answer, not the rerank that selects the passages. Spending a tenth of a cent on rerank to make the LLM call ten cents more accurate is a trade nobody should hesitate over.

There is one practical wrinkle: rerankers add latency. A typical reranker call adds 100–300ms to the end-to-end response time. For interactive use this is usually fine; for very latency-sensitive applications (autocomplete, sub-second voice), it may not be. The mitigation is to run rerank in parallel with whatever else you can parallelize, and to keep the rerank candidate count small enough that the latency is bounded.

The thing teams discover, six months in, is that the reranker is also the easiest stage to swap. Embedding-model changes force a corpus re-embed; reranker changes do not. Build the seam, run the experiments, ship the winner.

RAG vs long-context vs fine-tune

RAG is not the only way to get knowledge into a model. There are three options, and the choice between them is one of the more useful decision trees in this stack.

Long-context means paste the whole corpus into the prompt and let the model find what matters. With 200K-to-1M-token windows now routine, this option went from impossible to viable for a real class of problems. Use it when:

  • The corpus is small — comfortably under the model's context window with room for the system prompt and the response.
  • The query genuinely needs the whole corpus, not specific passages. "Summarize this contract," "compare these three policies," "find inconsistencies across this dossier" — these are long-context shapes.
  • Cost isn't a hard constraint and you want zero retrieval complexity. No vector DB, no embedding pipeline, no rerank — just the model and the prompt.

RAG means retrieve relevant passages at query time and put only those in the prompt. Use it when:

  • The corpus is too big for long-context, or the same model serves many corpora (every customer's documents, every team's wiki).
  • The corpus changes faster than you'd want to redeploy a model — daily docs, weekly policies, a knowledge base with constant edits.
  • You need verifiable citations. RAG hands you the source passages naturally; you can show users exactly which documents the answer was grounded in.

Fine-tuning means change the model's weights so it answers differently for everyone. Use it when:

  • You need the model to speak a specific way — a tone, a format, a domain vocabulary — that prompting and few-shot examples couldn't reliably produce.
  • RAG plus prompting can't get you there after honest effort. ("Honest effort" is doing the work in §1–§5 of this primer, not "I prompted it for a day.")
  • You can afford the eval and regression burden of a tuned model — every deploy is a new model that could regress on something the base model handled fine.
ApproachBest forWorst at
Long-contextSmall, stable, query-the-whole corpusLarge or changing corpora; cost at scale
RAGLarge or changing knowledge; citationsCross-document reasoning; style transfer
Fine-tuneSpecific voice, format, or domain languageAdding facts; regression risk on every deploy

The rule that survives every project: try RAG first. Long-context is the right answer if your corpus fits and the query genuinely wants all of it. Fine-tuning is the right answer only when prompting and RAG have hit their ceiling and the residual gap is about how the model speaks, not what it knows. Most teams that reach for fine-tuning early haven't yet done the work that would have made RAG good enough.

What this connects to

RAG is a tool that sits inside a larger system. The other primers in the course pick up the threads.

  • Primer 2 — Prompting & Context Engineering — RAG is context engineering. The retrieved passages are part of the context budget, alongside the system prompt, tool definitions, and conversation history. Everything Primer 2 said about ordering things stable-to-volatile, capping output, and caching the parts that don't change applies to the passages you paste into the prompt.
  • Primer 5 — The Agent Loop — once retrieval becomes a tool the agent calls (rather than a step in a fixed pipeline), RAG joins the loop. The model decides when to retrieve, what query to retrieve with, and whether to retrieve again. This is the shift from "RAG pipeline" to "agent that retrieves," and it changes both the design and the evals.
  • Primer 6 — Memory & State — RAG over the conversation history itself is one form of agent memory: index past turns, retrieve the relevant ones when the context window can no longer hold everything. The patterns from this primer transplant directly.
  • Primer 9 — Sub-Agents, Skills & Harnesses — at scale, retrieval often becomes its own sub-agent with its own evals: query rewriter, retriever, reranker, judge. The decomposition matters because the failure modes are different at each stage and you want to debug them independently.

Most teams' first agent failure is bad retrieval, not bad models. The model gets blamed because it is what the user sees, but the model only ever did what the retrieved context told it to. Get RAG right, then worry about the rest.
