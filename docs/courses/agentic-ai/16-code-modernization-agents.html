Why code modernization is the canonical agentic engineering app

Every large bank carries millions of lines of code that no one wants to write any more, and that no one currently on the payroll wrote in the first place. COBOL on the mainframe still books the deposits. SAS still runs the risk batch overnight. PL/SQL still hosts the pricing rules. Java 1.4 still answers the channel calls. The estate is enormous, the language skills are scarce, and the cost of doing nothing keeps compounding as the people who understood the original code retire faster than the workload shrinks.

This is the canonical application of agentic engineering. Three properties make it so. First, volume: the work is measured in millions of lines, not thousands, and any per-module improvement multiplies. Second, narrow output: a modernization task produces exactly two artefacts — the modernized module and a test suite that verifies behavior parity with the legacy one. There is no open-ended creative surface; the output shape is fixed. Third, verifiable outcome: behavior parity is a binary, runnable property. You either pass the legacy test suite against the new code or you do not. There is no judgement call hiding behind the answer.

The same three properties recurred in Primer 15. KYC and surveillance work because they are high-volume, narrow-output, audit-checkable. Code modernization works because it is high-volume, narrow-output, test-checkable. The patterns are siblings. The engineering one is in some ways easier — a passing test suite is a cleaner ground truth than an analyst's confirm-click — but it carries the same discipline: the agent does the gather, the human ratifies the result, and the verification artefact is the contract between them.

The same pattern proves out across the rest of the engineering modernization surface: schema migrations, framework upgrades, build-system rewrites, API-version transitions, log-format normalisations. Anything where a piece of code has to be expressed in a new vocabulary while keeping its behavior, and where the new behavior can be tested against the old, is in scope. The bank starts with COBOL because the budget is biggest there; the same harness handles a Spring 2 to Spring 6 migration the following quarter.

The translation pattern

The unit of work is not "translate this file." It is translate this module and ship the tests that prove the translation works. The agent reads the legacy module, infers its behavior, produces equivalent code in the target language, and produces a test suite that exercises the behavior in both languages. Two outputs, not one. This is the move that distinguishes a real modernization agent from a clever transpiler.

The flow has three stages. The agent first reads — it pulls the legacy module, any data structures it references, the calling conventions of the modules around it, and whatever runbook or documentation exists. It infers what the module is for: this calculates daily interest accrual, this normalizes a customer-record field, this posts a settlement instruction. The inference is grounded by sample inputs and outputs harvested from production logs where they exist, or constructed from the module's own internal structure where they do not.

The agent then translates. The new code is written in the target language — Java for the COBOL batch, Python for the SAS analytics, whichever modern stack the bank has standardised on. The translation is not line-by-line; it is behavior-preserving. Idioms move from COBOL's verbose verb forms to Java's expression syntax, from SAS's procedural data steps to Python's pandas operations. What is held invariant is what the module does, not how it looks.

The agent also generates a test suite. This is the part that makes the pattern work, and the part most early modernization efforts skip. The test suite is written in the target language's test framework, but it is structured to be runnable against both implementations: each test case is a triplet of inputs, expected outputs, and a label naming the behavior under test. The same triplets that drive the JUnit suite against the new Java module also drive a harness that exercises the legacy COBOL module on the mainframe with the same inputs, and asserts byte-equal outputs. The tests are the contract between old and new.

The shape of the output is what enforces discipline. A modernization agent that produces only code can hallucinate the behavior and ship a plausible-looking module that quietly drops a rounding rule. A modernization agent that produces code and a parity-test suite cannot — the missing rounding rule shows up as a failing test the first time the suite is run against the legacy module. The two outputs are not redundant; they are the front and back of the same contract.

Test generation is the load-bearing part

Without generated tests, you have an unverified translation. With tests that run against both the old and the new code, you have something you can actually ship. This is the load-bearing claim of the whole pattern, and it is the place where most modernization attempts collapse. A translation that no one can verify is a liability dressed as an asset — it looks like progress, but it commits the firm to a piece of code whose behavior nobody has actually checked.

The test-generation step has its own shape. The agent enumerates behaviors from three sources: the legacy module's own branches and edge cases, the production logs that show what the module is actually called with, and the runbook or specification text that names invariants the module is supposed to preserve. Each behavior becomes a labelled triplet — given this input, the module produces this output, because of this rule. The triplets are the corpus; the corpus drives both the new test suite and the parity harness.

The parity harness is the piece that makes the discipline real. It takes a triplet, runs the input through the legacy implementation (over a mainframe RPC, a SAS-runner sidecar, whatever wraps the old runtime), runs the same input through the new implementation, and asserts that the outputs match. The new code passes only if every triplet passes on both sides. This is what test-generation buys you that hand-written tests never could: you do not need a human to specify "expected output" in advance, because the legacy module is the specification, and the harness queries it directly.

def verify(corpus: list[Triplet]) -> list[Triplet]:
    failures = []
    for t in corpus:
        t.legacy_out = run_legacy(t.inp)
        t.new_out    = run_new(t.inp)
        if t.legacy_out != t.new_out:

fails = verify(harvest_corpus("ACCRUE-INTEREST"))
assert not fails, f"{len(fails)} parity failures — translation is not behavior-preserving"

Two properties of the sketch are worth naming. The legacy module is treated as a black-box oracle — the harness never assumes it can read or modify the COBOL, only that it can call it with inputs and receive outputs. This is what makes the pattern survive contact with mainframes that nobody is allowed to touch. And the corpus is the agent's output, not a human-written test plan — the agent's job is to find the behaviors that matter, and the parity harness's job is to confirm the new code reproduces them.

When the suite is incomplete, the translation is unverified, and the modernization team knows it. Coverage of behaviors — not coverage of lines — is the metric. A 90% line-covered translation that misses the leap-year branch is worse than a 60% line-covered translation that catches every branch the agent identified, because the second one is honest about what it does not yet know.

The human at the seam

Senior engineers do not write the modernized code. They review the agent's output. Their checkpoint is exactly two questions: does this preserve behavior? and is the test coverage real? The engineer who used to translate the module by hand now spends the morning reading three modernization PRs the agent has prepared overnight, accepting two and rejecting one for thin coverage of a known edge case. The job shape changes from author to auditor, and the work the human does is the work the agent cannot — judging whether the test corpus actually represents the domain.

This is the same human-at-the-seam pattern as Primer 15's case-pack review. The agent prepares a complete deliverable — code, tests, parity-run results, a short note explaining the inferred behavior. The human reads the deliverable, not the model's reasoning trace, and the harness has constrained the deliverable to a fixed shape. The review is fast because the shape is fixed and because the parity numbers are right there on the PR. The override discipline is the same: every rejected translation gets a structured reason that feeds back into the eval suite.

The reviewer's leverage is in the second column. The agent can write a thousand lines of Java overnight; the human cannot read a thousand lines of Java overnight and should not try. What the human is uniquely good at is reading the behavior list and noticing what is missing — that the agent did not enumerate the quarter-end accrual case, or that the runbook flagged a 1997 emergency patch that the trace never touched. The agent is the typist; the human is the domain conscience.

Working with verification

Verification is the hinge. Run the test suite against the legacy module and against the new module on identical inputs. Any divergence is a translation bug. Not "an interesting modelling choice," not "a slight behavioral drift," not "a refactor that improves the design" — a bug. Escalate to the human. The modernization agent does not get to argue with the oracle, because the oracle is the production system that is currently booking trades, and the production system has already won.

The discipline this enforces is severe and correct. It forbids the agent from rationalizing — from explaining that the divergence is because the legacy module had a bug and the new code is "better." Maybe it is, and maybe the legacy "bug" is a regulatory adjustment that survived three audits. Either way, the call is the human's, not the agent's. The agent flags the divergence with the failing triplet and the trace; the human decides whether to fix the new code, ratify the divergence, or open a ticket against the legacy module that will live for another five years.

VERIFICATION

Run the new tests against the old code. If they don't pass, the translation is wrong.

The same parity-test suite that proves the new module is correct also proves that the agent understood the old module in the first place. Failing tests on the legacy side mean the agent's behavior inference was wrong, not that the legacy code was wrong. Fix the inference, regenerate the corpus, re-run.

The infrastructure cost of verification is real and is part of the deployment plan. Spinning up a parity harness against a mainframe requires sandboxes, data masking, and rate-limited RPC into the old runtime. The bank that wants to modernize at scale invests in that harness once and pays the cost across every module it touches afterwards. Without it, modernization is theatre.

Trace-driven modernization

When verification fails, the trace shows where the translation diverged from the legacy behavior. This is where Primer 11's discipline — the structured, queryable record of what the agent saw, what it inferred, and what it produced — pays off. Without a trace, a failing parity run tells you that the new code is wrong; it does not tell you why. With a trace, you can walk back from the failing triplet to the inference step that produced the wrong behavior, see what the agent assumed, and patch the prompt, the corpus, or the harness rather than re-running and hoping.

The flow is concrete. A parity test fails on a leap-year input. The reviewer opens the trace for that translation. They see the agent's behavior-enumeration step, scan it, and find that "leap year" is not in the list. They see the source documents the agent consulted — runbook excerpts, sample inputs from logs — and observe that the log sample covered three years, none of which was a leap year. The fix is upstream of the model: extend the log-harvest window to span at least one leap year. Re-run. The behavior shows up in the corpus. The translation now passes.

This is the loop that makes the deployment improve. Every parity failure is a data point about where the inference machinery is brittle. The override log from Primer 15 has its analogue here in the divergence log: a structured record of which legacy modules the agent misread and why. Over a quarter, the divergence log tells the modernization team which parts of the harness need investment — better log sampling, richer runbook ingestion, a structured-comment parser for the COBOL source — and which parts are already pulling their weight. The agent gets better not by retraining but by improving the inputs it sees and the corpus it generates.

The trace is also what the regulator will read, eventually. A modernization PR that ships into a mainframe replacement project is a regulated change. Five years from now, when someone asks how the bank verified that the new Python risk engine matched the SAS one it replaced, the answer is: here is the parity corpus, here are the runs, here is the trace of the agent that produced both, and here are the signatures of the engineers who approved each merge. Modernization without traceability is unauditable; modernization with it is just engineering.

What this connects to

Back to the engineering spine. The modernization agent is a loop (Primer 5 — The Agent Loop) that calls a few tools — fetch the legacy module, run the harness, post the PR — under the discipline of a fixed output shape. Inside a real deployment it is usually a sub-agent (Primer 9 — Sub-Agents, Skills, Harnesses): one node in a larger pipeline whose surrounding orchestrator hands it a module to modernize and reads back a PR. The trace work (Primer 11 — Observability & Traces) is what turns the divergence log into a feedback loop the team can act on.

Sister application. The human-at-the-seam pattern that runs this primer is the same pattern that runs Primer 15 — Agentic KYC & Surveillance. There, the agent gathers, the human judges, and the case pack is the contract. Here, the agent translates, the human ratifies, and the parity test suite is the contract. Two different application domains, the same operating-model shape. That is the point of the pattern — once a team has built the seam once, every subsequent agentic deployment plugs into the same review discipline.

Building an agent is easy. Building an agent function is hard. By now the difference should be concrete: it is the test suite, the case pack, the evidence chain, the override log, the trace, the seam.
