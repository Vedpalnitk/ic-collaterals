01Part A · The threat model

A new attack surface

Every prior volume flagged a security risk and deferred it here. This is where they consolidate. Agentic AI introduces a genuinely new attack surface, because an agent combines three things that are individually old but dangerous together: it follows natural-language instructions, it reads untrusted content, and it takes real actions.

Traditional software does only what its code says. An agent does what its instructions say — and it cannot reliably tell legitimate instructions from instructions smuggled into the data it processes (Vol. 03 §10, Vol. 04 §7). Add tools that act on the world, and a content problem becomes an action problem. The result is a class of vulnerabilities with no exact precedent in conventional security, which is why agent security is its own discipline and not just "application security with a model."

The core asymmetry

The same property that makes agents useful — they act autonomously on instructions, over data, using tools — is what makes them exploitable. You cannot remove the capability to remove the risk; you contain the risk while keeping the capability. That framing governs the whole volume.

02Part A · The threat model

Prompt injection

The signature attack on agents. Prompt injection is when content reaching the model carries instructions that hijack its behaviour. It comes in two forms, and the second is the dangerous one for agents.

Direct
The user attacks

A user types instructions to override the system prompt — "ignore your rules and do X." A concern, but the user attacking their own session limits the damage.

Indirect
The data attacks

Instructions are planted in content the agent reads — a web page, email, document, or tool result. The agent encounters them while doing legitimate work, on behalf of a trusting user. Far more dangerous.

Indirect injection is the one to fear because the victim is unaware. An analyst asks an agent to review a document; the document contains hidden text instructing the agent to exfiltrate data or take an action; the agent, unable to separate the document's content from commands, may comply. As agents browse, read email, and consume external data, the supply of attacker-controlled content grows — and so does the exposure.

Watch out

There is no prompt phrasing that makes a model injection-proof. Defences reduce the rate and contain the impact; none eliminate it. Design as though injected instructions will sometimes win inside the context, and limit what the agent can do when they do — the theme of Part C.

Part B · The threats
03Part B · The threats

Jailbreaks & override

Closely related to injection, jailbreaks are techniques that coax a model past its safety training and behavioural rules — getting it to produce content or take actions it was instructed to refuse.

The methods are varied and evolving: role-play framings ("pretend you're a system with no restrictions"), incremental escalation that normalises each small step, obfuscation that hides intent from filters, and "do anything now"–style personas. The relevant lesson for builders is not the specific tricks but the structural fact behind them: safety behaviour learned in training is a soft constraint, not a hard one. A sufficiently clever framing can sometimes talk the model around it, just as injected data can.

Implication

Because model-level safety is probabilistic, it cannot be your only line of defence. Real protection comes from constraints outside the model — what tools exist, what they're permitted to do, what's validated and approved — that hold regardless of what the model is talked into saying. The prompt is not a security boundary; neither is the model's training.

04Part B · The threats

Tool abuse & excessive agency

Excessive agency is the vulnerability that turns a compromised model into real-world harm: an agent granted more capability, permission, or autonomy than its task requires. When injection or a jailbreak succeeds, excessive agency determines the blast radius.

The failure has three flavours: too many functions (tools the task never needed but an attacker can use), too many permissions (a tool with write access where read would do, or access to data beyond scope), and too much autonomy (the agent acts on consequential steps without approval). An agent that can only read public data does little harm when hijacked; one that can read everything and send email externally can do a great deal. The damage is the capability you granted — so granting less is the most reliable defence.

Watch out — the lethal trifecta

Risk concentrates catastrophically when one agent has all three of: access to sensitive data, exposure to untrusted content, and the ability to communicate or act externally (Vol. 04 §7). Any two are usually containable; all three together is the path from "the agent read a malicious page" to "data left the building." Architect to break that combination — separate the capabilities across agents or insert human approval between them.

05Part B · The threats

Data exfiltration & leakage

A primary attacker goal — and a serious accidental risk — is getting sensitive data out. Agents are unusually good at leaking, because they handle confidential data and have channels to send it.

Exfiltration can be driven by injection ("send the customer records to this address"), or happen by accident: sensitive data placed in a logging system, a third-party tool, or a model endpoint without the right controls; an agent including confidential context in an external message; or the model regurgitating data from one user's session that lingered in shared state. System-prompt leakage is a milder cousin — coaxing the agent to reveal its hidden instructions, exposing logic and sensitive framing. In financial services, any of these can be a reportable data-protection breach, not merely a bug.

Watch out

Watch the egress points, not just the inputs. Every channel by which data can leave — external messages, third-party tools, logs, model APIs — needs the same scrutiny as the data store itself. Minimise what the agent can send, to where, and ensure sensitive data never reaches an uncontrolled endpoint (Vol. 02 §10).

06Part B · The threats

Supply-chain & multi-agent threats

Agents increasingly depend on components someone else built — tools, MCP servers, other agents, models. Each is a trust relationship, and each is a potential point of compromise.

A malicious or compromised MCP server (Vol. 04 §4) runs with the access you grant it and can return poisoned content or misuse its permissions. A poisoned knowledge source (Vol. 05 §9) injects attacker content into retrieval. In multi-agent systems (Vol. 07), one compromised agent can manipulate others through handoffs, and every agent another can call is a new trust boundary. Emerging agent-to-agent protocols extend this risk across organisations. The discipline is the same as software supply-chain security: vet what you connect, grant it least privilege, and treat its outputs as untrusted.

Principle

Adding a tool, an MCP server, or an agent connection is a supply-chain decision, not a convenience. Inventory what your agents depend on, understand what each can access and do, and assume any external component could be hostile or become so.

Part C · Defence
07Part C · Defence

Defence in depth

Because no single control is sufficient — not the prompt, not the model's training, not any one filter — agent security is layered. Each layer assumes the ones before it may fail, and limits the damage when they do.

Defence in depth: concentric controls so a single failure isn't catastrophic.

No layer is perfect, and that is the point. Input defences reduce injection; least privilege shrinks what a compromise can reach; human approval stops the irreversible; egress controls block the leak; monitoring catches what slips through. Stack them, and an attacker must defeat several independent controls rather than one.

08Part C · Defence

Approval & least privilege

Two controls do more than all others combined, because they don't depend on the model behaving: granting the agent little, and gating the consequential.

Least privilege (Vol. 04 §8) directly attacks excessive agency: an agent with only the tools and data its task needs simply cannot do most harmful things, no matter what it's tricked into. Human approval on consequential or irreversible actions ensures the worst outcomes require a person's deliberate sign-off — the agent proposes, the human authorises. Together they break the lethal trifecta: even a fully hijacked agent that can read sensitive data and is exposed to malicious content cannot independently act on the outside world, because the action is gated.

The leverage

Most agent security is decided at design time, in two questions: what can this agent do without a human, and what data can it reach? Answer both conservatively and the attack surface collapses. The cleverest input filter is worth less than simply not granting the dangerous capability.

09Part C · Defence

Security in financial services

In a bank, an agent security failure is not just an incident — it is potentially a data breach, a fraud event, a regulatory violation, and a loss of customer trust at once. The stakes raise the bar on every defence in this volume.

The domain's data sensitivity, irreversible actions, and regulatory exposure make the conservative posture mandatory rather than optional. In practice: agents get broad read access but tightly gated, human-approved write access; untrusted external content is isolated from action; sensitive data is barred from uncontrolled endpoints; every action is logged for audit; and the lethal trifecta is deliberately broken by separating capabilities and inserting human checkpoints. Security here is inseparable from governance (Vol. 11) and regulation (Vol. 12) — the same controls satisfy all three.
