01Part A · What a model is

From prediction to intelligence

A large language model does one thing: given a stretch of text, it predicts the next small piece of text. Everything an agent does — reason, plan, draft, decide — is built on top of that single, almost absurdly simple operation, repeated.

The model is shown the text so far and produces a probability for every possible next token (the unit it works in — roughly a word-fragment; §2). It picks one, appends it, and runs again with the slightly longer text. This is autoregressive generation: predict, append, repeat, until it predicts a "stop". A paragraph is just this loop run a few hundred times.

Generation is next-token prediction, one token at a time, over the growing text.

Why does something so mechanical produce apparent intelligence? Because predicting the next token well, across essentially all the text humans have written, forces the model to internalise an enormous amount: grammar, facts, styles, reasoning patterns, the structure of code, the shape of an argument. To reliably continue "the capital of France is ___", it must have absorbed that fact. To continue a proof or a piece of code, it must have absorbed the patterns of valid reasoning. Competence at prediction, at sufficient scale, becomes competence at the underlying tasks.

Definition

A large language model (LLM) is a neural network trained to predict the next token in text, at a scale (billions of parameters, trillions of tokens of training data) where doing so well requires — and produces — broad linguistic, factual, and reasoning ability.

Watch out

The model is optimised to produce plausible text, not true text. Those usually coincide — but not always, and the model has no internal sense of which is which. This single fact explains hallucination (§7), why grounding matters (§6), and why a confident answer is not a correct one.

02Part A · What a model is

Tokens & the context window

Models don't see words or letters — they see tokens. Understanding tokens explains a surprising amount of model behaviour, cost, and limits.

What a token is

A token is a chunk of text, usually a common word-fragment. Frequent words are a single token; rarer words split into several. In English, one token averages about four characters, so roughly 100 tokens ≈ 75 words. The model converts your text into a sequence of these tokens, works entirely in token-space, and converts back to text on the way out.

Tokenisation: rare or long words split into multiple sub-word tokens.

This explains several quirks. Models are priced per token (in and out), so cost tracks tokens, not words. They can struggle to count letters in a word or do digit-by-digit arithmetic, because they never see individual letters or digits — only tokens. And different languages tokenise less efficiently, making them more expensive to process.

The context window

The context window is the maximum number of tokens the model can consider at once — its entire working memory for a single request. Everything competes for that budget: the system instructions, the conversation, any retrieved documents, tool results, and the answer being generated. When the window is full, something must be dropped or summarised.

Windows have grown from a few thousand tokens to hundreds of thousands or more. But bigger is not automatically better. Models attend less reliably to information buried in the middle of a very long context ("lost in the middle"), and stuffing in irrelevant text can degrade answers. The skill — the subject of memory and retrieval in Vol. 05 — is getting the right tokens into the window, not the most.

Takeaway

Think in tokens. Cost, speed, context limits, and several odd failure modes all become predictable once you picture the model reading and writing a stream of word-fragments through a fixed-size window.

03Part A · What a model is

Inside the model

You never need the mathematics, but a mental picture of what is happening between input and output makes the model's strengths and failures far less mysterious. Three ideas carry almost all the intuition: embeddings, attention, and parameters.

Embeddings: meaning as numbers

Each token is turned into a long list of numbers — a vector — that encodes its meaning. The remarkable property is that tokens with similar meanings get similar vectors, so relationships become geometry: "king" is near "queen", "London" relates to "England" as "Paris" relates to "France". This is also what makes semantic search and RAG possible (Vol. 05). The model reasons by manipulating these meaning-vectors, not the letters.

Attention: weighing what matters

The core mechanism of the transformer architecture (the design behind essentially all modern LLMs) is attention: as it processes each token, the model decides which earlier tokens are relevant and weights them accordingly. In "the alert the analyst escalated was a false positive", attention is how the model connects "false positive" back to "alert" rather than "analyst". Stacked across many layers, this lets the model build up context-sensitive understanding of the whole passage.

Parameters: the learned knowledge

Parameters are the model's adjustable internal weights — the numbers tuned during training. A frontier model has hundreds of billions of them, and collectively they encode everything it "knows" and every pattern it can produce. More parameters generally means more capability, but also more compute, cost, and latency — a tension we return to in §9.

Mental model

Picture the model as a vast, fixed function: tokens in as meaning-vectors, many layers of attention weighing context, parameters shaping every step, and a probability over the next token out. It is not looking anything up and not running a program you wrote — it is computing a learned statistical continuation. That is both its power and the reason it can be confidently wrong.

Watch out

"The model knows X" is a convenient shorthand, not a fact about storage. Knowledge is smeared across billions of parameters as patterns, not filed as retrievable records. That is why a model can't reliably cite its source, why it can blend two facts into a plausible-but-wrong third, and why you ground it with real data (§6) when accuracy matters.

Part B · How models are shaped
04Part B · How models are shaped

Training: pretrain → post-train

A usable model is built in stages. The first gives it raw knowledge and fluency; the later ones make it helpful, safe, and aligned to how you want it to behave. Knowing which stage produces which trait tells you what you can and cannot fix with prompting versus training.

The model you use is a base model that has been instruction-tuned and preference-tuned.

Pretraining — where knowledge comes from

The model is trained to predict the next token across an enormous corpus of text and code. This is where it absorbs facts, language, and reasoning patterns — and it is staggeringly expensive, done rarely. The output is a base model: full of knowledge but unhelpful, because it only knows how to continue text, not to answer a request.

Post-training — where behaviour comes from

Two further stages turn the base model into an assistant. Instruction tuning (supervised fine-tuning) trains it on examples of good responses to instructions, so it learns to answer rather than merely continue. Preference tuning (e.g. RLHF — reinforcement learning from human feedback) then nudges it toward responses humans prefer: more helpful, honest, and harmless. Most of a model's "personality", tone, refusals, and helpfulness come from this stage — not from its raw knowledge.

Knowledge cutoff

Because pretraining data ends at a fixed date, the model has a knowledge cutoff — it knows nothing that happened after it, no matter how confidently it answers. This is structural, not a bug, and it is the core reason agents need tools and retrieval (§6) for anything current.

The practical upshot: knowledge lives in pretraining; behaviour lives in post-training. If a model lacks a fact, no amount of prompting reliably adds it — you ground it with data. If a model behaves wrongly (tone, format, refusing too much), that is often addressable with prompting or, at most, fine-tuning. Matching the fix to the layer is half of working with models well.

05Part B · How models are shaped

Reasoning models

The biggest capability shift of 2024–25 was not bigger models but models that think before answering. Understanding what reasoning models do — and what they cost — is essential to building agents that are both capable and affordable.

A standard model produces its answer in one forward pass; it commits to the first token immediately. A reasoning model is trained to first generate an extended internal chain of thought — working through the problem step by step — before producing its visible answer. It spends more computation at the moment of answering (called inference-time or test-time compute) rather than relying solely on what was baked in during training.

The effect is large on tasks that genuinely require multi-step deliberation: mathematics, coding, logical puzzles, careful analysis. Giving the model room to "show its working" — and training it to do so well, often via reinforcement learning on problems with checkable answers — measurably raises accuracy where a snap answer would fail.

Reach for a reasoning model
When the task needs deliberation

Multi-step analysis, math, code, planning, anything where a wrong first instinct is likely and a careful one pays off. The agent's planning brain (Vol. 06) often wants one.

Reach for a standard model
When speed and volume matter

Simple extraction, classification, short drafting, high-throughput steps. Reasoning here just adds latency and cost for no accuracy gain.

Watch out — the cost is real

Thinking is tokens, and tokens are time and money. A reasoning model can be several times slower and pricier per call. In an agent that loops dozens of times, choosing reasoning-vs-standard per step — not once for the whole system — is one of the highest-leverage cost decisions you will make.

06Part B · How models are shaped

Adapting a model to your needs

An off-the-shelf model doesn't know your policies, your data, or your house style. There are three levers to close that gap — and choosing the right one for the right problem saves enormous time and cost.

Three levers, cheapest first. Try them in this order; most needs are met by the first two.
  • Prompting — change what you tell the model: its role, rules, examples, and output format. No training, instant, free to iterate. The first and usually sufficient lever for behaviour. (The whole of Vol. 03.)
  • RAG (retrieval) — fetch relevant information and put it in the context at query time, so the model answers from your data and current facts rather than its frozen training. The right lever for knowledge it lacks. (The whole of Vol. 05.)
  • Fine-tuning — train the base model further on your own examples to adjust its weights. The right lever for deep, consistent style or specialised behaviour that prompting can't reliably hold — and the wrong lever for adding fresh facts, which it does unreliably. The most expensive and slowest to iterate.
Decision rule

Need knowledge the model lacks? → RAG. Need different behaviour, tone, or format? → prompting first, fine-tuning only if prompting genuinely can't hold it. Most teams reach for fine-tuning far too early; in practice, prompting plus RAG solves the large majority of real adaptation problems at a fraction of the cost.

07Part B · How models are shaped

Why models hallucinate

A hallucination is a confident, fluent, and false output — an invented fact, a fake citation, a plausible-but-wrong number. It is not a malfunction. It is the direct consequence of how the model works, which is exactly why it cannot be fully eliminated, only managed.

The mechanism

Recall §1: the model predicts plausible continuations, and has no internal ground truth. When it knows an answer, the plausible continuation is the true one. When it doesn't, the most plausible-sounding continuation is still produced — fluently and confidently — because the model has no separate signal that says "I don't actually know this." It is, in effect, always guessing; usually the guess is right, and the failure is that a wrong guess looks identical to a right one.

Several things make it worse: questions about niche or recent topics (thin or post-cutoff training data), requests for specific figures or citations (easy to fabricate convincingly), and long agent loops where one hallucinated fact gets treated as established and compounds.

What actually reduces it

Strongest lever
Grounding (RAG + tools)

Give the model the real source and ask it to answer only from that. A model citing a document in front of it fabricates far less than one answering from memory.

Strong lever
Permission to abstain

Explicitly allow "I don't know" and ask for citations. Much hallucination comes from the model feeling obliged to produce an answer at all costs.

Helps
Verification

Check claims with a tool, a second model, or a human before acting on them — especially for figures, names, and anything consequential.

Helps
Lower temperature

Reduces creative wandering for factual tasks (temperature is explained in §8). A partial measure, not a cure.

Watch out

There is no setting that makes a model truthful. Fluency is not reliability, and confidence is not correctness. In any regulated or high-stakes context, the only safe posture is to assume the model can be wrong and to design grounding, verification, and human review around it — the throughline into §10 and the governance volumes.

08Part C · Using models well

Capabilities & limits

Using a model well starts with a calibrated sense of what it is genuinely good at, where it is unreliable, and the fact that it is not deterministic. Most disappointing agent behaviour traces back to asking the model to do something on the wrong side of this line.

Reliable strengthsUnreliable / risky
Language — summarising, rewriting, translating, drafting, tone.Exact arithmetic — multi-digit math; it pattern-matches, doesn't calculate. Give it a calculator tool.
Extraction & classification — pulling fields, tagging, routing.Current / niche facts — anything past the cutoff or thinly represented. Ground it.
Pattern & code work — generating and explaining code, transforming structured text.Precise recall of sources — exact quotes, citations, figures. Easily fabricated.
Reasoning over provided text — analysing, comparing, explaining what's in front of it.Consistency — the same prompt can give different answers (see below).

Non-determinism and temperature

A model is not a calculator: the same prompt can produce different outputs on different runs. This is by design. After scoring the next token, the model samples from the distribution, and a setting called temperature controls how adventurous that sampling is. Low temperature (near 0) makes it pick the highest-probability token almost every time — more consistent, more focused, better for factual and structured tasks. Higher temperature introduces variety — useful for brainstorming and creative work, riskier for anything needing one right answer.

Why this matters for agents

Non-determinism is why agents need evaluation and guardrails (Vol. 01 §9): you cannot prove an agent correct by running it once, because the next run may differ. It is also why low temperature is the default for tool-using, decision-adjacent steps where you want the same input to behave the same way.

09Part C · Using models well

Choosing a model

There is no single "best" model — only the right model for a given step, balanced across a handful of axes. Treating model choice as one global decision is a common and costly mistake.

The axes that trade off

Axis
Capability

How well it reasons and follows complex instructions. Bigger / reasoning-tuned models lead — at a price.

Axis
Cost & latency

Price per token and speed per call. In a looping agent, both multiply fast.

Axis
Context window

How much it can take in at once — matters for long documents and rich agent state.

Axis
Modality

Text only, or also images, audio, structured tool use. Match to the task's inputs.

Axis
Deployment

Hosted API vs. self-hosted / open-weight — a control, cost, and data-residency choice (key in §10).

The pattern
Route, don't standardise

Send easy, high-volume steps to small fast models; reserve frontier models for the hard ones. Mixing models per step is normal and economical.

The model landscape, in shape not brand

Models cluster into tiers: frontier models (most capable, most expensive — for the hardest reasoning), mid-tier workhorses (the balanced default for most steps), and small/fast models (cheap and quick — for classification, extraction, and high-throughput work). They come as closed hosted APIs (simplest, most capable, least control over data location) or open-weight models you can run yourself (most control, more engineering). Specific names change constantly; the tiers and trade-offs do not — design around the shape.

Takeaway

Pick models per step against capability, cost, latency, context, and deployment constraints — then revisit, because the frontier moves monthly. Architect so that swapping a model is a config change, not a rebuild; the model you choose today will not be the best choice next quarter.

10Part C · Using models well

Models in financial services

Everything above meets a harder reality in a bank. The model's two defining traits — that it can be confidently wrong, and that it processes whatever data you feed it — collide directly with confidentiality duties and regulatory expectations. The constraints don't forbid models; they dictate how and where they're used.

The four constraints that shape every deployment

Constraint
Data confidentiality

Customer and transaction data cannot flow to uncontrolled endpoints. This drives choices toward private, contractually-bound, or self-hosted deployments and strict data-handling — often the deciding factor in §9's deployment axis.

Constraint
Hallucination intolerance

A fabricated fact in a credit decision, a SAR, or customer-facing advice is a compliance event, not a glitch. Hence grounding and human review are mandatory wherever output informs a regulated decision.

Constraint
Explainability & audit

Decisions affecting customers must be explainable and logged. A smear-of-parameters answer with no traceable basis is hard to defend; grounded, cited, recorded output is far more so.

Constraint
Model risk governance

AI/ML in decisioning falls under model-risk regimes requiring validation, monitoring, and documentation — covered in Vol. 12. The model is governed infrastructure, not a free utility.

The resulting pattern

These constraints produce a consistent, defensible posture: use models where they are strong and the stakes are bounded — summarising, drafting, extracting, triaging, and preparing — with humans owning the decide and file steps, all output grounded in real data and the whole flow logged. The same model that should never autonomously close an account or file a report is genuinely transformative at compressing the hours of evidence-gathering and drafting that precede those human decisions.

Provenance note. Productivity and adoption figures circulating for LLMs in banking and financial crime are drawn from 2025–26 secondary sources and vary widely by definition. Treat them as directional only and verify against primaries before client use; sourced domain detail lives in Vols. 13–14.
11Part C · Using models well

The engine, not the agent

This volume opened the box that Vol. 01 left closed. The payoff is a precise understanding of where the model ends and the system begins — and why the model is necessary but never sufficient.

The model is the engine: it sets the ceiling on reasoning quality, but on its own it has no memory between calls, no access to your data, no ability to act, and a built-in tendency to sound certain whether or not it is right. Capability you can actually deploy comes from what Vol. 01 called the other five components — instructions, tools, memory, orchestration, and guardrails — built around the engine to supply exactly what it lacks.

The synthesis

A model gives you fluent reasoning over whatever is in its context, sampled probabilistically, frozen at its cutoff, with no inherent sense of truth. An agent adds current data (RAG/tools), persistence (memory), action (tools), repeatability (orchestration and low temperature), and safety (grounding, verification, human review). Every later volume is, in this sense, about compensating for a specific limitation you now understand from the inside.

Where this goes next

With the engine understood, the series turns to driving it well: Vol. 03 — Prompting & Context Engineering (the cheapest, highest-leverage lever from §6), then tools, memory and reasoning patterns (Vols. 04–06), multi-agent systems, evaluation and security (Vols. 07–10), governance and regulation (Vols. 11–12), and the financial-crime and enterprise applications where it pays off (Vols. 13–15).

Glossary

The working vocabulary of this volume.

Attention
The transformer mechanism by which a model weighs which earlier tokens are relevant when processing each new one.
Autoregressive generation
Producing text one token at a time, each prediction conditioned on all tokens so far.
Base model
A pretrained model before instruction- and preference-tuning — knowledgeable but not yet a helpful assistant.
Context window
The maximum number of tokens a model can consider in one request; its working memory.
Embedding
A vector of numbers representing a token's or text's meaning, such that similar meanings sit close together.
Fine-tuning
Further training of a model's weights on custom examples to adjust style or specialised behaviour.
Hallucination
A confident but false model output; a structural consequence of predicting plausible rather than true text.
Inference-time compute
Computation spent while answering (as opposed to during training); reasoning models spend more of it.
Instruction tuning (SFT)
Supervised fine-tuning on instruction–response examples so a base model learns to answer requests.
Knowledge cutoff
The date the model's training data ends; it knows nothing later without tools or retrieval.
Large language model (LLM)
A large neural network trained to predict the next token, yielding broad language and reasoning ability.
Parameters
A model's learned internal weights — billions of numbers encoding its knowledge and patterns.
Pretraining
The expensive first stage: next-token prediction over a massive corpus, where knowledge is acquired.
Preference tuning (RLHF)
Aligning a model toward human-preferred responses for helpfulness, honesty, and harmlessness.
Reasoning model
A model trained to produce extended internal reasoning before its answer, improving multi-step accuracy at higher cost.
Temperature
A setting controlling randomness in token sampling; low = focused and consistent, high = varied and creative.
Token
The sub-word unit a model processes; cost, context limits, and several quirks are measured in tokens.
Transformer
The neural-network architecture, built on attention, behind essentially all modern LLMs.
