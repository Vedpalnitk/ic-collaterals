01Part A · Why it's hard

The reliability problem

Conventional software is deterministic: the same input gives the same output, and a passing test stays passed. Agents are neither. They are probabilistic, multi-step, and open-ended — which makes "does it work?" a genuinely hard question and evaluation the discipline that answers it.

Two properties from earlier volumes make this acute. Non-determinism (Vol. 02 §8): the same prompt can yield different outputs, so one successful run proves nothing about the next. Compounding (Vol. 01 §9): across a long loop, small errors accumulate, so a system that's 95% reliable per step can be far less reliable end-to-end. Together they mean you cannot eyeball your way to confidence — you have to measure, repeatedly, against many cases.

The mindset shift

Stop asking "did it work?" (a single run) and start asking "what fraction of representative cases does it handle, and is that fraction holding or drifting?" Reliability is a measured rate, not a binary — and a property of the system you build, not the model you chose.

02Part A · Why it's hard

What "good" means

Before you can measure an agent, you have to define success — and for open-ended tasks that is harder than it sounds. There is rarely one right answer, so you evaluate along several dimensions.

DimensionThe question
Task successDid it achieve the goal — close the case correctly, extract the right fields, reach the right disposition?
GroundednessWere claims supported by real evidence, or fabricated (Vol. 02 §7, Vol. 05 §8)?
Trajectory qualityDid it take a sensible path — the right tools, no wasted or destructive steps?
Safety & complianceDid it stay within policy, refuse what it should, and escalate when required?
Cost & latencyDid it reach the outcome efficiently, within budget?

Different uses weight these differently — a customer-facing agent prizes safety and groundedness; a back-office one may prize cost and throughput. The act of defining what "good" means for your agent, explicitly, is half the value of evaluation.

Part B · Measuring
03Part B · Measuring

Evals & golden datasets

An eval is a repeatable test of an agent against a set of cases with known-good outcomes. A golden dataset is that curated set of representative inputs and their expected results. Together they turn "seems fine" into a number you can track across every change.

Building the golden set is the work that matters. It should span the real range of inputs — common cases, the tricky edge cases you care about, and known past failures — so the score reflects reality, not a flattering sample. Each change to the agent (a new prompt, a swapped model, a reworked tool) is then run against the whole set, and you watch whether the score improves or regresses. This is exactly the prompt-iteration discipline of Vol. 03 §7, scaled to the whole agent.

The payoff

A golden dataset is what lets you change an agent with confidence. Without it, every "improvement" is a gamble — you fix one case and silently break three. With it, you ship changes the way good software teams ship code: tested against a known baseline.

04Part B · Measuring

LLM-as-judge

Many agent outputs — a narrative, an analysis, a recommendation — have no single correct string to match against. LLM-as-judge uses a model to score them against a rubric, making open-ended quality measurable at scale.

You give a judge model the input, the agent's output, and clear scoring criteria ("is every claim supported by the cited evidence? is the disposition justified? 1–5"), and it produces a score and rationale. This scales human-like judgement across thousands of cases that no team could grade by hand. It is powerful and now standard — but it is itself a model, with all the same fallibility.

Watch out

The judge can be wrong, biased toward verbosity or its own style, and gameable. Calibrate it against human judgement on a sample, use precise rubrics, and reserve human review for the highest-stakes evaluations. An LLM judge is a scalable approximation of human grading, not a replacement for it.

05Part B · Measuring

Trajectory & component evals

For agents, the final answer is only half the story. The path matters too: an agent can reach a right answer through a reckless or lucky route that will fail next time. So you evaluate both outcomes and trajectories, and you evaluate components in isolation.

Trajectory evaluation checks the sequence of steps — did it choose appropriate tools, avoid wasteful loops, take no destructive action, escalate when it should? Component evaluation tests pieces separately: retrieval quality apart from generation (Vol. 05 §8), each tool's reliability, a sub-agent on its own. Isolating components tells you where a system fails, which whole-system scores alone can't: a poor end result might be a retrieval problem, a reasoning problem, or a tool problem, and you fix each differently.

Why both

Outcome evals tell you whether the agent is good; trajectory and component evals tell you why it's good or bad and what to fix. Mature teams run all three, because debugging a probabilistic multi-step system needs the diagnostic resolution.

Part C · Operating
06Part C · Operating

Observability & tracing

You cannot debug, audit, or trust what you cannot see. Observability is the instrumentation that records what an agent actually did — every reasoning step, tool call, result, and handoff — so a run can be understood and replayed.

Evaluation and observability form one loop: test before deploy, observe in production, feed failures back.

A good trace captures the full story of a run in order — inputs, each model decision, each tool call and its result, timing, and cost. When something goes wrong, you replay the trace and see exactly where. For agents this is not just an engineering aid; it is the raw material of the audit trail a regulated deployment requires (Vols. 11–12), so observability and compliance are built by the same instrumentation.

07Part C · Operating

Monitoring & drift

An agent that passed its evals at launch will not stay good on its own. The world changes around it, and quality drifts — so production monitoring is a permanent obligation, not a launch checklist.

Several things shift over time. Inputs change as customer behaviour and fraud patterns evolve (data drift), so cases the agent never saw start arriving. Underlying models get updated, subtly altering behaviour. New edge cases and failure modes surface only at scale. Monitoring watches live quality signals — success rates, escalation rates, cost, latency, user corrections — and alerts when they move, so degradation is caught early rather than discovered in an incident.

Watch out

In financial crime, drift is not just a quality issue but a coverage risk: an agent tuned to last year's typologies may quietly miss this year's. Monitoring must track whether the agent still catches what it should, and the golden dataset must be refreshed as new patterns emerge — an evaluation set is a living asset.

08Part C · Operating

Eval-driven development

The mature practice ties it all together: eval-driven development treats evaluation the way good software treats automated testing. The agent isn't "done" when it demos — it's done when it passes its evals, and it stays done only while it keeps passing them.

In practice this means writing evals alongside (or before) building, running them on every change, gating deployment on the score, and feeding every production failure back into the golden set so the same mistake can't recur unnoticed. It reframes agent development from "prompt until the demo looks good" — how most pilots stall — into a measured engineering loop with a baseline you defend. This is the single biggest predictor of whether an agent reaches reliable production.

The throughline

Vol. 01 §9 named evaluation, observability, and guardrails as the three defences against unreliability. This volume operationalised the first two into a development discipline. Evaluation is not a phase; it is how you build and run agents you can trust.

09Part C · Operating

Evaluation in regulated FS

In financial services, evaluation and observability are not optional engineering hygiene — they are regulatory requirements wearing technical clothes. Model-risk regimes (Vol. 12) demand exactly what this volume describes.

The mapping is direct. Validation before deployment is an eval against a representative, documented dataset. Ongoing monitoring is drift detection and live quality signals. Auditability is the trace of every decision and the context behind it. Documentation is the record of what was tested, how it scored, and what changed. An institution that builds rigorous evaluation and observability is, largely, building its model-risk evidence at the same time — which is why these capabilities are foundational, not a later add-on, for any agent that touches a regulated decision.

Glossary

Data drift
Gradual change in the inputs an agent sees over time, degrading performance tuned to older patterns.
Eval
A repeatable test of an agent against cases with known-good outcomes.
Eval-driven development
Treating evaluation like automated testing: write evals, gate changes on them, feed failures back.
Golden dataset
A curated, representative set of inputs and expected outcomes used as the evaluation baseline.
Groundedness
The degree to which output is supported by real evidence rather than fabricated.
LLM-as-judge
Using a model to score open-ended outputs against a rubric, scaling human-like judgement.
Observability
Instrumentation that records what an agent did, enabling debugging, replay, and audit.
Trace
An ordered record of a single run — inputs, decisions, tool calls, results, timing, cost.
Trajectory evaluation
Assessing the path an agent took, not just its final output.
