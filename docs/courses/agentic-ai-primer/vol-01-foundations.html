01Part A · What an agent is

What is agentic AI?

An AI agent is a system that pursues a goal by deciding its own next steps, taking actions in the world through tools, observing the results, and repeating — until the goal is met or it gives up. The model is no longer just answering; it is operating.

For most of the last few years, the dominant pattern was the assistant: you write a prompt, the model writes a reply, the turn ends. The intelligence sat entirely inside one request-and-response. Agentic AI breaks that frame. Given a goal rather than a single instruction, an agent plans a route to it, calls tools to gather information or change something, reads what came back, and decides what to do next — looping autonomously across many steps without a human in every one.

The shift is less about the model getting smarter and more about what the model is allowed to do between your messages. A chatbot that can only talk is bounded by its training. An agent that can search, read files, query a database, call an API, and run code is bounded by its tools — a far larger surface.

Definition

Agentic AI — software in which a language model acts as the reasoning core of a loop that perceives, plans, acts through tools, and adapts over multiple steps to achieve a goal, with limited or no human intervention per step.

Assistant vs. agent

The clearest way to feel the difference is side by side.

Assistant (single-turn)
"Summarise this transaction report."

One input, one output. The model uses only what is in front of it. It cannot fetch the underlying transactions, cross-check a watchlist, or act on its conclusion. The work begins and ends in one reply.

Agent (multi-step)
"Investigate this alert and recommend a disposition."

The agent pulls the customer profile, queries recent transactions, checks sanctions and adverse-media sources, reasons about the pattern, drafts a rationale, and proposes close-or-escalate — looping through tools as needed before returning.

The five properties that make something agentic

Not every system that calls an LLM is an agent. "Agentic" is a spectrum, not a label, but five properties together are what move a system along it. Most production "agents" in 2026 have the first three strongly and the last two partially.

Property 01
Goal-directed

Works toward an outcome you specify ("resolve this case"), not a single instruction ("write this sentence"). It owns the gap between intent and result.

Property 02
Autonomous

Chooses its own next step. You don't script the sequence of tool calls; the model decides what to do based on what it has seen so far.

Property 03
Tool-using

Acts on the world beyond text — searching, reading, writing, calling APIs, running code. Tools are how the agent reaches outside its own context.

Property 04
Stateful

Carries information across steps and sometimes across sessions — what it has tried, what it learned, what the user prefers. Memory turns a sequence of calls into a coherent effort.

Property 05
Adaptive

Changes course when a step fails or returns something unexpected, rather than blindly continuing. This is the property that is hardest to get right — and the source of most failures.

Takeaway

An agent is a loop with a model at the centre and tools at the edges. Everything in this primer is an elaboration of that one sentence: how the loop runs (§2–3), how the model decides inside it (§4), how it reaches the world (§5), what it remembers (§6), how far it is trusted (§7), how many of them you run (§8), and why they break (§9).

02Part A · What an agent is

The agent loop

Every agent, however sophisticated, runs the same core cycle: perceive → reason → act → observe, with remember threaded throughout. The loop repeats until the goal is reached, a stop condition fires, or a budget runs out.

The agent loop. Each pass through is one "step"; a single task may take dozens.

Walking the loop

  • Perceive — the model reads the current state: the original goal, the conversation so far, and any new information (a tool result, a user reply, a sensor reading).
  • Reason / Plan — it decides what to do next. Sometimes that is a small plan ("first find the customer, then check transactions"); sometimes just the single next action.
  • Act — it emits a tool call: a structured request to run a search, query a database, send a message, write a file. (Covered in depth in §5.)
  • Observe — the tool runs and returns a result, which is fed back into the model's context as new input.
  • Remember — relevant facts are retained so later steps can build on earlier ones rather than starting cold.

The loop ends when the model judges the goal complete and produces a final answer, when it hits a guardrail (a step limit, a cost cap, a forbidden action), or when it asks a human for a decision it isn't allowed to make alone.

Watch out

The loop is also where things go wrong. A bad observation poisons the next reasoning step; a model that never decides it's "done" loops forever; small errors compound across steps. The longer the loop runs unattended, the more these matter — which is exactly why §9 (reliability) and §13 (governance) exist.

03Part A · What an agent is

Anatomy of an agent

If §2 is the verb, this is the noun. An agent is assembled from six parts. Understanding them individually is what lets you reason about cost, risk, and failure later.

ComponentWhat it isWhy it matters
ModelThe LLM at the centre — the reasoning core that interprets, plans, and chooses actions.Sets the ceiling on reasoning quality. Bigger / reasoning-tuned models plan better but cost more and run slower.
InstructionsThe system prompt: the agent's role, goals, rules, tone, and what it must never do.The cheapest, highest-leverage control you have. Most behaviour problems are instruction problems first.
ToolsThe functions the agent can call to reach outside itself — search, database, API, code execution.Defines what the agent can actually do. Also its largest risk surface (§5).
MemoryShort-term (the context window) plus optional long-term stores it can write to and read from.Turns disconnected steps into a coherent effort; lets the agent learn across sessions (§6).
OrchestrationThe control flow around the model: the loop runner, routing, stop conditions, retries, handoffs.Where determinism lives. The model is probabilistic; orchestration is where you put the guarantees.
GuardrailsLimits and checks: allowed actions, approval gates, output validation, budgets, kill switches.The difference between an agent you can deploy and one you can only demo (§13).
Mental model

Think of the model as a brilliant new analyst, the instructions as their job description, the tools as their system access, memory as their notebook, orchestration as the workflow they sit inside, and guardrails as the policies that say what they can do without a manager's sign-off. A capable analyst with no access does nothing; full access with no policy is a liability.

A crucial implication: most of the engineering in an agent is not the model. The model is largely fixed — you select it. The work, and most of the value and risk, lives in the other five components. That is good news: it means agentic systems are something you design and govern, not a black box you simply switch on.

04Part B · How an agent works

How an agent reasons

Inside the "reason" step of the loop, the model isn't magic — it follows recognisable reasoning patterns. Knowing them lets you pick the right one for a task and diagnose why an agent is wandering.

Chain-of-thought: think before answering

The foundation. Rather than jumping straight to an answer, the model is prompted to work through intermediate steps in natural language first. "Let's think step by step" is the cliché, but the effect is real: spreading reasoning across more tokens measurably improves accuracy on multi-step problems. Modern reasoning models do this internally by default, producing a private train of thought before their visible answer.

ReAct: reason and act, interleaved

The pattern most production agents use. ReAct (Reason + Act) alternates between a thought and an action: the model reasons about what it needs, takes one tool action, observes the result, reasons again. It is the agent loop of §2 made explicit in the model's own output. Its strength is grounding — each action is informed by real observations, not just the model's assumptions.

Plan-and-execute: decide the route first

For complex tasks, the agent first drafts a multi-step plan, then executes the steps (sometimes delegating each to a sub-agent), re-planning if reality diverges. This reduces aimless wandering on long tasks and makes the agent's intent inspectable before it acts — valuable when actions are costly or irreversible.

Reflection: critique and retry

The agent reviews its own draft output or trajectory, identifies flaws, and revises. A "generator" produces, a "critic" (often the same model in a different role) evaluates against the goal, and the loop repeats. Reflection is how agents catch their own mistakes — at the cost of extra time and tokens per attempt.

Use when
ReAct

Tasks needing live information and incremental decisions — investigation, research, troubleshooting. The default.

Use when
Plan-and-execute

Long, structured tasks with clear sub-goals, or where you want to review the plan before any action runs.

Use when
Reflection

Quality-critical output where a second pass pays for itself — drafting a regulatory narrative, writing code.

Cost note
All of them spend tokens

More reasoning = more tokens = more latency and money. Match the pattern to the stakes; don't reflect on a one-line lookup.

Watch out

Reasoning text is not a guarantee of correct reasoning. A model can produce a fluent, confident chain of thought that is wrong at step two and carries the error to the conclusion. The visible reasoning helps you audit a decision — it does not certify it.

05Part B · How an agent works

Tools & function calling

Tools are how an agent reaches the world. Function calling is the mechanism: the model, instead of replying in prose, emits a structured request to run a named function with specific arguments. This single capability is what separates a chatbot from an agent.

How a tool call actually flows

It is worth seeing the full round-trip once, because nearly every subtlety of agent behaviour — and every security concern — lives in it.

Trace · one tool call, end to end
Setup
The runtime tells the model which tools exist, each with a name, description, and a schema for its arguments — e.g. get_transactions(account_id, days).
Reason
"To assess this alert I need the last 90 days of activity for account A-4471."
Emit call
The model outputs, instead of prose: get_transactions(account_id="A-4471", days=90)
Execute
The runtime — not the model — runs the real function against the real system and gets back data. The model never touches the database directly.
Return
The result is inserted back into the model's context as a new observation: [312 transactions, total £1.84m, 7 to high-risk jurisdictions…]
Continue
"Seven transfers to high-risk jurisdictions clustered in week three — I should check whether these counterparties are sanctioned."
→ next tool call, and the loop continues.

Two points are easy to miss and important. First, the model only proposes; the runtime disposes. The model emits a request; your code decides whether to run it, with what permissions, and whether a human must approve first. That seam is where governance lives. Second, the tool's result re-enters the context as trusted input — which is the root of the biggest security risk below.

MCP: a standard plug for tools

Historically every tool was wired to every agent by hand. The Model Context Protocol (MCP), introduced by Anthropic in late 2024 and now broadly adopted, standardises that wiring: a tool or data source is exposed once as an MCP "server", and any MCP-aware agent can use it without bespoke integration. Think of it as USB for agent tools — one connector instead of a custom cable per device. It is a large part of why agent ecosystems matured quickly through 2025–26.

Watch out — tools are the attack surface

Because tool results flow back into the context as input the model trusts, a malicious result can carry instructions. A web page, email, or document the agent reads can contain hidden text that hijacks it — prompt injection. Combined with broad tool permissions, this is how an agent can be tricked into exfiltrating data or taking unauthorised actions. The defences — least-privilege tools, human approval on consequential actions, treating all tool output as untrusted — are core to §13 and the entire subject of Vol. 10 (Agent Security).

Takeaway

An agent's capability and its risk are the same thing: its tools. Designing the tool set — what exists, what arguments it accepts, what permissions it carries, and what requires sign-off — is the highest-leverage safety decision in the whole system.

06Part B · How an agent works

Memory & knowledge

A model has no memory of its own between calls — each request starts blank. Everything it "knows" in the moment is whatever sits in its context window. Memory is the engineering that decides what goes in that window, and what persists beyond it.

The context window: working memory

The context window is the finite span of text the model can attend to in a single request — the goal, instructions, conversation, retrieved documents, and tool results, all competing for the same budget (measured in tokens). It behaves like RAM: fast, in front of the model, but limited and wiped between sessions. When it fills, older content must be summarised or dropped, and quality can degrade if the relevant detail is buried among irrelevant text ("context rot"). Windows have grown enormously, but more context is not automatically better — what matters is the right context.

Types of memory

Short-term
Working / session

The current conversation and task state, held in context. Lost when the session ends unless deliberately saved.

Long-term · facts
Semantic

Durable knowledge — policies, customer preferences, prior conclusions — stored externally and retrieved when relevant.

Long-term · events
Episodic

A record of what happened before: past cases, prior interactions, what was tried and how it went. Lets the agent learn across runs.

RAG: giving the model knowledge it wasn't trained on

A model's training data is frozen and general; it doesn't know your policies, your customers, or yesterday's transactions. Retrieval-Augmented Generation (RAG) closes that gap: relevant documents are fetched from a knowledge store and inserted into the context at query time, so the model answers from your data rather than from memory alone. The store is usually a vector database — text is converted into numerical embeddings so that "find me passages similar in meaning to this question" becomes a fast similarity search.

Classic RAG: retrieve relevant passages, add them to the context, answer from them.

Agentic RAG: retrieval as a decision, not a step

Classic RAG retrieves once, up front. Agentic RAG hands retrieval to the agent as a tool: the agent decides whether it needs to look something up, what to search for, can reformulate a query that returned nothing useful, search multiple sources, and keep going until it has enough. Retrieval becomes part of the reasoning loop rather than a fixed pre-step — more capable on hard questions, at the cost of more calls and more variability.

Takeaway

"What does the model know right now?" always reduces to "what is in its context right now?" Memory and RAG are the disciplines of getting the right information into that finite window at the right moment — and keeping stale or malicious content out of it.

07Part B · How an agent works

The autonomy spectrum

"Agent" is not binary. How much an AI system is allowed to decide and act on its own runs along a spectrum — and choosing the right rung for a given task is one of the most consequential design and governance decisions you will make.

A practical five-rung autonomy scale. Different tasks within one system can sit at different rungs.

The vocabulary that matters most here is the human's posture. Human-in-the-loop means a person must approve each consequential action before it happens (L1–L2). Human-on-the-loop means the agent acts on its own but a person monitors and can intervene (L3). Human-out-of-the-loop means no routine oversight (L4) — appropriate only where actions are reversible, low-stakes, or tightly bounded.

Choosing the rung

Autonomy should track two things: how reversible the action is, and how costly a mistake would be. Drafting a summary a human will read is safe at high autonomy — the human is the gate. Closing a customer's account, releasing a payment, or filing a regulatory report is not: those belong at L1–L2 regardless of how good the model is, because the failure is expensive and hard to undo.

Watch out

The temptation is to set autonomy by model capability ("it's smart enough now"). Set it by consequence instead. A more capable model raises the quality of decisions; it does not change who should be accountable for an irreversible one.

08Part B · How an agent works

Multi-agent orchestration

When a task is too broad for one agent, the answer is often several specialised agents working together. Orchestration is how they coordinate — who does what, in what order, and who decides.

The common patterns

Three building-block patterns. Real systems combine them — a supervisor whose workers are pipelines, and so on.
  • Supervisor (orchestrator–worker) — a lead agent decomposes the goal and delegates to specialist sub-agents (a "research" agent, a "screening" agent), then synthesises. The most common production pattern.
  • Sequential pipeline — agents run in a fixed order, each refining the last's output. Predictable and easy to govern; less flexible.
  • Parallel + merge — several agents tackle parts simultaneously and a merger combines them. Fast, good for breadth, but coordination and conflicting outputs need handling.

The control plane

Whatever the pattern, something has to manage routing, shared state, message-passing, and handoffs (one agent passing control and context to another). That coordination layer is the control plane. It is where you enforce order, prevent two agents from clobbering each other's work, and keep a coherent record of what the system as a whole did — essential for both reliability and audit.

Watch out — agent sprawl

More agents is not more capability. Each added agent multiplies cost, latency, coordination overhead, and the number of places a failure or injection can enter. Many "multi-agent" problems are better solved by one well-equipped agent with good tools. Reach for multiple agents when genuinely distinct skills or parallelism justify it — not by default.

09Part B · How an agent works

Reliability & why agents fail

A demo that works once is easy; an agent that works the thousandth time, unattended, is the hard part. Because agents are probabilistic and run many steps, errors don't just occur — they compound. Reliability is the discipline of measuring, containing, and observing that.

Why long loops are fragile

If each step in a task is 95% reliable, a ten-step task is only about 60% reliable end-to-end (0.9510), and a twenty-step task barely better than a coin flip. This compounding is the central reliability challenge of agentic systems: the very autonomy that makes agents useful also gives small errors room to accumulate before any human notices.

The common failure modes

Failure modeWhat happens
HallucinationThe model states something false with confidence — a fabricated fact, citation, or tool argument — and acts on it.
Error compoundingA wrong observation or assumption early in the loop quietly corrupts every step after it.
Tool misuseRight intent, wrong call — bad arguments, the wrong tool, or a destructive action taken when a read-only one was meant.
Looping / no-stopThe agent never decides it is finished, retrying or circling until a budget cuts it off.
Context overflowRelevant detail is pushed out of the window or buried, and the model loses the thread.
Prompt injectionUntrusted content read via a tool hijacks the agent's instructions (see §5).

The three defences

Measure
Evaluations

Test the agent against curated cases with known-good outcomes ("golden" datasets), score trajectories, and use a model to judge open-ended output (LLM-as-judge). Evals turn "seems fine" into a number you can track as you change things.

See
Observability

Trace every step — inputs, reasoning, each tool call and result — so a failure can be replayed and diagnosed. You cannot debug, audit, or trust what you cannot see.

Contain
Guardrails

Step and cost budgets, allow-lists of actions, output validation, and human-approval gates on consequential steps. Guardrails assume the model will err and limit the blast radius when it does.

Takeaway

Reliability is not a property of the model; it is a property of the system you build around it. The mature posture is to assume every step can fail, then design evaluation, observability, and guardrails so that failures are caught, contained, and cheap. This is the bridge from Part B's mechanics to Part C's enterprise reality — and the heart of later volumes on evaluation (Vol. 09) and security (Vol. 10).

10Part C · In the enterprise

Evolving trends

The field moves fast, but the direction of travel is stable enough to plan around. These are the shifts shaping agentic AI through 2026 and beyond — and what each one means for how you should build.

Trend 01
Reasoning models as default

Models that "think" before answering — spending compute on internal deliberation — have become standard for agentic work, materially improving multi-step planning. The implication: better agents, but higher latency and cost per step to budget for.

Trend 02
Standard protocols (MCP, A2A)

MCP for tool/data connection and emerging agent-to-agent protocols are turning bespoke integrations into plug-in ecosystems. Building on standards now avoids rebuilding later.

Trend 03
Computer & browser use

Agents that operate a screen — clicking, typing, navigating apps with no API — are unlocking the long tail of systems that can't be integrated cleanly. Powerful, and a fresh security surface.

Trend 04
Eval-driven development

Teams increasingly treat evaluation suites the way software treats tests: the agent isn't "done" when it demos, but when it passes its evals. Reliability is becoming an engineering practice, not a hope.

Trend 05
Smaller, specialised models

Not every step needs the largest model. Routing simple steps to cheaper, faster, sometimes fine-tuned small models cuts cost without losing much quality.

Trend 06
Ambient & background agents

Agents are moving from chat windows to running quietly in the background — triggered by events, monitoring queues, doing work before anyone asks. The interface recedes; the governance need grows.

Watch out

The capability curve is steep, but the reliability and governance curve lags it. The recurring 2025–26 lesson across the industry is that the hard problem is rarely getting an agent to work once — it is getting it to work safely and predictably at scale. Plan budgets and timelines around the second curve, not the first.

11Part C · In the enterprise

Agentic AI in financial services

Financial services is among the most fertile and most fraught grounds for agentic AI: process-heavy and data-rich, which suits agents — and heavily regulated with low tolerance for error, which constrains them. Nowhere is this sharper than in financial crime, where the same technology is both defence and threat.

Where agents add value

Defence · investigations
Alert triage & case work

Agents gather the evidence around an AML or fraud alert — profile, transactions, counterparties, screening hits — assemble a structured picture, and draft a recommended disposition for an analyst to confirm. The repetitive evidence-gathering compresses; the judgement stays human.

Defence · onboarding
KYC / CDD automation

Document checks, data extraction, sanctions and PEP screening, and risk-rating drafting orchestrated end to end, with exceptions routed to humans. Faster onboarding, consistent files.

Defence · reporting
Narrative & SAR drafting

Agents draft the investigation narrative or suspicious-activity report from the case record — a first draft for the analyst to review and own, not a filing made autonomously.

Threat · the other side
Agentic fraud

The same autonomy serves attackers: agents that open accounts at machine speed across many institutions, run hundreds of scam conversations at once, and pass automated checks. Agentic defence is partly a response to agentic offence.

Why the constraints bite harder here

Three features of the domain set the rules. Regulation demands explainability, auditability, and human accountability for decisions affecting customers. Irreversibility is common — released payments, filed reports, and closed accounts are costly to undo, which (per §7) caps autonomy. And model risk regimes treat AI/ML in decisioning as governed infrastructure subject to validation and oversight. The result is a clear pattern: agents that prepare, gather, and draft at high autonomy, with humans retaining the decide and file steps.

Provenance note. Market-sizing and productivity figures circulating for agentic AI in financial crime (e.g. directional claims of large KYC/AML productivity uplifts, multi-agent adoption surges, and forward adoption percentages) are drawn from secondary-source 2025–26 estimates and vary widely by definition and source. Treat them as directional signals only; verify against primary sources before quoting in any client-facing deliverable. The domain volumes (13–14) of this series carry the sourced detail.
12Part C · In the enterprise

Impact on the organization

Agentic AI does not simply make existing roles faster; it reshapes what work is left for people and what the org chart looks like. The clearest way to picture it is a change in the shape of the workforce.

Agents absorb routine volume; human effort shifts up into judgement, exception-handling, and oversight.

As agents take on the high-volume, repeatable base of work, the people who did that work don't simply disappear — the centre of gravity moves up. The growth is in judgement, exception-handling, and oversight: deciding the hard cases the agent escalates, supervising what the agent does, and owning the outcomes. New roles appear alongside it.

New role
Agent designers

People who define an agent's instructions, tools, and guardrails — translating a process and its rules into a safe, working system.

New role
Evaluators / AI QA

People who build the eval suites and monitor agents in production — the quality function for probabilistic systems.

New role
Exception handlers

Domain experts who own the cases agents escalate — fewer in number, higher in skill, with more authority per decision.

Watch out

The hard part of organisational adoption is rarely the technology — it is trust, role redesign, and accountability. Where the routine base shrinks, the remaining work is harder and the stakes per decision are higher. Reskilling and clear ownership matter more than raw model capability; this is the subject of Vol. 15.

13Part C · In the enterprise

Governance & operating model

Everything in Parts A and B converges here. An agent you can deploy — as opposed to demo — is one wrapped in governance: clear control over what it may do, who is accountable, and how you would know if it went wrong. This is the discipline that turns capability into something an enterprise can responsibly run.

The control questions

Before any agent goes live, four questions need answers. They map directly onto the components from §3 and the autonomy scale from §7.

QuestionWhat it governs
What can it do?The tool set and permissions — least privilege, read-only by default, consequential actions gated.
How far can it decide alone?Its rung on the autonomy spectrum, set by reversibility and cost of error — not by model capability.
Who is accountable?A named human owner for the agent's outcomes. Autonomy never transfers accountability to the system.
How would we know it failed?Observability, evals, monitoring, and an intervention path — including the ability to stop it.

The regulatory backdrop

In regulated industries, governance is not optional. Model-risk management regimes (in US banking, the SR 11-7 lineage and its successors) require AI/ML used in decisions to be validated, documented, and monitored. The EU AI Act imposes obligations — risk management, human oversight, logging, transparency — on high-risk uses, with several creditworthiness and decisioning applications squarely in scope. The practical effect is that explainability, audit trails, and human oversight become design requirements, not afterthoughts. (Vol. 12 covers this in depth.)

The agentic operating model

At organisational scale, the pattern that works is centralised platform, federated use: a common, governed foundation — model access, the tool/MCP layer, shared guardrails, evaluation and observability tooling, and policy — on top of which individual teams build their own agents. The centre owns safety, standards, and the things that are dangerous to reinvent per team; the edges own the domain knowledge and the specific use cases. This balances speed of adoption against consistency of control.

The deployable-agent test

An agent is ready for production when you can answer, for every consequential thing it can do: who approved it, who is accountable, how it is bounded, and how you would catch and stop a failure. If any answer is missing, the agent is a demo, not a deployment — regardless of how impressive it looks.

Glossary

The working vocabulary of this volume, in plain terms.

Agent
A system in which a model runs a perceive–reason–act–observe loop, using tools to pursue a goal over multiple steps with limited human intervention.
Agentic RAG
Retrieval handed to the agent as a tool, so it decides whether, what, and how often to look something up — rather than retrieving once up front.
Autonomy spectrum
The range from human-does-everything (L0) to self-directed (L4); the chosen rung should track an action's reversibility and cost of error.
Chain-of-thought
Prompting or training a model to produce intermediate reasoning steps before its answer, improving accuracy on multi-step problems.
Context window
The finite span of text a model can attend to in one request — its working memory, shared by instructions, conversation, retrieved data, and tool results.
Control plane
The coordination layer in a multi-agent system that manages routing, shared state, handoffs, and order.
Embedding
A numerical representation of text that captures meaning, enabling similarity search in a vector database.
Eval (evaluation)
A repeatable test of an agent against known-good cases, used to measure and track quality as the system changes.
Function calling
The mechanism by which a model emits a structured request to run a named tool with arguments, instead of replying in prose.
Guardrail
A limit or check — allowed actions, budgets, approval gates, output validation — that contains an agent's behaviour and failures.
Handoff
One agent passing control and relevant context to another.
Hallucination
A confident but false output from a model — a fabricated fact, citation, or argument.
Human-in / on / out-of-the-loop
Whether a human approves each action (in), monitors and can intervene (on), or does neither routinely (out).
MCP (Model Context Protocol)
An open standard for connecting models to tools and data sources — "USB for agent tools."
Orchestration
The control flow around a model or set of agents: the loop runner, routing, stop conditions, retries, and handoffs.
Prompt injection
An attack in which untrusted content the agent reads via a tool contains instructions that hijack its behaviour.
RAG (Retrieval-Augmented Generation)
Fetching relevant documents and inserting them into the context so the model answers from your data, not just its training.
ReAct
A reasoning pattern interleaving thoughts and actions — reason, act, observe, repeat — the basis of most production agents.
Reasoning model
A model that performs internal deliberation before its visible answer, improving multi-step planning at higher cost and latency.
Reflection
An agent critiquing and revising its own output or trajectory before finalising.
Token
The unit of text models process; context windows, cost, and latency are all measured in tokens.
Tool
A function an agent can call to act beyond text — search, database query, API call, code execution.
Vector database
A store of embeddings that supports fast similarity search, the usual backbone of RAG.

  1. Anthropic — "Building Effective Agents." Practical patterns for agents vs. workflows, and when to use each. (Foundations / patterns.)
  2. Anthropic — Model Context Protocol (MCP) documentation & introduction. The open standard for tool/data connection. (§5, §10.)
  3. OpenAI — "A Practical Guide to Building Agents." Components, orchestration, and guardrails from a builder's view. (§3, §8, §13.)
  4. Yao et al. — "ReAct: Synergizing Reasoning and Acting in Language Models." The originating ReAct paper. (§4.)
  5. Wei et al. — "Chain-of-Thought Prompting Elicits Reasoning in Large Language Models." (§4.)
  6. Lewis et al. — "Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks." The foundational RAG paper. (§6.)
  7. Shinn et al. — "Reflexion: Language Agents with Verbal Reinforcement Learning." Reflection as a mechanism. (§4, §9.)
  8. OWASP — Top 10 for LLM Applications (incl. prompt injection). The standard security reference. (§5, §9.)
  9. NIST — AI Risk Management Framework (AI RMF). A widely used governance scaffold. (§13.)
  10. US Federal Reserve / OCC — SR 11-7 Model Risk Management guidance (and successors). (§11, §13.)
  11. European Union — Artificial Intelligence Act (Regulation (EU) 2024/1689). High-risk obligations relevant to financial decisioning. (§11, §13.)
Provenance reminder. Where this volume references market-sizing or productivity figures for agentic AI in financial crime, those are directional secondary-source estimates and are deliberately not stated as precise here. Verify against primary sources before reuse in client-facing material. Detailed, sourced domain figures live in Vols. 13–14.
