01Part A · The landscape

What a framework gives you

You can build an agent with nothing but a model API and your own code — the loop of Vol. 01 is only a few lines. So what do agent frameworks actually buy you? Mostly: the parts that are tedious, error-prone, and the same every time.

A framework provides the scaffolding around the model: the loop runner, tool-calling plumbing, state and memory management, multi-agent orchestration, retries and error handling, and increasingly hooks for tracing and evaluation. It lets you describe what the agent should do and handles much of how the loop executes. The value is leverage and consistency; the cost is a layer of abstraction between you and the model, which can obscure behaviour and complicate debugging.

Framing

A framework is not capability — the model is. A framework is productivity and structure. The right question is never "which framework is most powerful?" but "which removes the most undifferentiated work for my use case without hiding what I need to control?"

02Part A · The landscape

The framework landscape

The ecosystem is crowded and moves monthly, so memorising tools is futile. Understand it by shape instead — the categories endure even as specific names rise and fall.

CategoryWhat it is & examples
Graph / workflow orchestrationExpress the agent as a controllable graph of steps and state — maximum control over flow. (e.g. LangGraph.)
Role / crew abstractionsDefine agents by role and let them collaborate at a higher level — fast to start, less low-level control. (e.g. CrewAI, AutoGen.)
Provider SDKsFirst-party agent toolkits from the model providers, tightly integrated with their models and tool/MCP support. (e.g. OpenAI Agents SDK, Anthropic's tooling.)
Supporting infrastructureNot frameworks but the stack around them — vector DBs, evaluation/observability platforms, model gateways (§5).
Watch out

Specific tools, versions, and capabilities change far faster than this primer's cycle. Treat any named framework as illustrative of its category, not a current recommendation — and re-check the landscape at decision time. Architect so the framework is replaceable.

03Part A · The landscape

High-level vs. low-level

The deepest distinction among frameworks is how much they abstract away — and it is a genuine trade-off, not a quality ranking.

High-level
Describe agents, get behaviour

Define roles and goals; the framework handles the loop, delegation, and coordination. Fast to prototype, great for standard patterns. Less visibility and control when behaviour must be precise — a liability in regulated work.

Low-level
Control the flow explicitly

You define the graph, the state, and the transitions. More code and more to own, but full control and transparency — which is what auditability and reliability demand.

The pattern many teams settle on: prototype quickly with a high-level abstraction to validate the use case, then move to a lower-level, more controllable foundation for anything going to production in a regulated environment — where "I'm not sure why it did that" is not an acceptable answer.

Part B · Decisions
04Part B · Decisions

Build vs. buy

Beyond which framework lies a bigger question: how much to build versus buy at all. The answer differs by layer, and the right strategy is usually a mix.

LayerLean buyLean build
The modelAlmost always buy (use a provider).Self-host open-weight only for strong data-residency or cost reasons.
Framework / orchestrationUse an established one to start.Build/own the critical path where control and audit matter.
Tools & integrationsReuse MCP servers where trusted.Build your own for proprietary systems and sensitive data.
The use-case agent—Build — this is your differentiation and domain knowledge.
The principle

Buy the commodity, build the differentiator. The model and generic plumbing are commodities to buy or adopt; your domain logic, your data integrations, and your control and governance layer are where you build, because that is where your advantage and your accountability live.

05Part B · Decisions

The supporting stack

A production agent needs more than a framework. Several supporting components, drawn from earlier volumes, together form the real stack — and underinvesting in them is the most common reason pilots fail to reach production.

Knowledge
Vector DB & retrieval

The memory and RAG layer (Vol. 05) — where grounding and currency come from.

Quality
Evaluation & observability

Eval suites, tracing, and monitoring (Vol. 09) — how you know it works and keep knowing.

Access
Model gateway

A layer that routes to models, enforces limits and policy, and lets you swap models as a config change (Vol. 02 §9).

Control
Guardrails & security

Permission enforcement, approval gates, input/output validation (Vols. 04, 10) — the control plane that makes it deployable.

None of this is the "agent" itself, yet collectively it is most of the engineering. A framework that gives you the loop but leaves you to assemble knowledge, evaluation, access, and control has solved the easy 20%.

06Part B · Decisions

Deployment & operations

An agent in production is a long-running, stateful, externally-acting service — operationally unlike a stateless web app, and that difference shapes how you run it.

Several properties demand attention. Cost and latency are variable and can spike with a single runaway loop, so budgets, rate limits, and timeouts are operational necessities, not nice-to-haves. State must persist reliably across an agent's many steps and survive restarts. Non-determinism (Vol. 02 §8) means the same input can behave differently, so monitoring and replay matter more than in conventional software. And because agents act, a deployment needs a tested way to pause, intervene, and stop them — a real kill switch, not a hope.

Watch out

The leap from a working prototype to a reliable production service is larger for agents than for ordinary software, precisely because of cost variability, statefulness, non-determinism, and real-world side effects. Plan for this gap; it is where most agent initiatives stall.

Part C · Selecting
07Part C · Selecting

Choosing for the enterprise

Enterprise selection criteria differ sharply from a hackathon's. Speed-to-first-demo matters far less than control, transparency, security, and the ability to evolve.

Criterion
Control & transparency

Can you see and govern exactly what the agent does? Opaque magic fails audit.

Criterion
Security & data handling

Does it support least privilege, on-prem/private deployment, and your data-residency needs?

Criterion
Observability & eval

Does it integrate with tracing and evaluation, or leave you blind?

Criterion
Maturity & support

Production track record, stability, and a path for support — not a fast-moving experiment.

Criterion
Lock-in & portability

How hard to swap the model or framework later? Favour replaceable parts.

Criterion
Standards support

MCP and open interfaces, so tools and agents compose rather than entangle.

The strategy

Standardise the platform centrally — shared framework, model gateway, eval/observability, guardrails — and let teams build use-case agents on top. This is the "centralised platform, federated use" operating model (Vol. 11), realised as a technology stack.

08Part C · Selecting

Frameworks in regulated FS

In financial services, the framework and stack are part of the control environment, and three requirements dominate the choice — all of which push toward control over convenience.

Auditability: the stack must record what every agent did — decisions, tool calls, handoffs, the context shown — in a form a regulator or internal audit can reconstruct (Vols. 09, 12). Data control: confidentiality and residency rules often force private or self-hosted deployment and strict data handling (Vol. 02 §10), constraining which platforms are even viable. Model risk: the whole system is governed infrastructure subject to validation and monitoring (Vol. 12), so transparency and evaluability are non-negotiable. Together these explain why regulated institutions tend toward more controllable, lower-level foundations they can fully see and own.

Where this goes next

Glossary

Agent framework
Software providing the scaffolding around a model — loop, tool plumbing, state, orchestration — so you describe behaviour rather than build it from scratch.
Build vs. buy
The decision, taken per layer, between adopting existing components and building your own.
Graph orchestration
Expressing an agent as a controllable graph of steps and state for maximum flow control.
High-level vs. low-level
Whether a framework abstracts the loop away (fast, less control) or exposes it (more code, full control).
Model gateway
A layer routing requests to models while enforcing limits and policy and enabling model swaps.
Provider SDK
A first-party agent toolkit from a model provider, tightly integrated with its models.
Supporting stack
The components around the framework — vector DB, eval/observability, gateway, guardrails — that production needs.
