01Part A · When & why

One agent or many?

The instinct, on seeing a complex problem, is to throw more agents at it. Resist it. The first and most important decision in multi-agent design is whether you need multiple agents at all — and the honest answer is usually "not yet."

A single agent with a good set of tools (Vol. 04) and the right reasoning pattern (Vol. 06) handles a remarkable range of tasks. Multiple agents add real costs: more model calls, more latency, coordination overhead, and more places for failure or prompt injection to enter. They earn their keep only when a problem has genuinely distinct sub-skills, benefits from parallelism, or grows too large for one agent's context and focus.

The test

Reach for multiple agents when you can name the specialists and why each needs to be separate — "a screening agent, a transaction-analysis agent, a narrative-drafting agent, because each needs different tools and a different focus." If the answer is "it feels complex," a single well-equipped agent is almost always the better, cheaper, more debuggable choice.

02Part A · When & why

The orchestration patterns

When you do go multi-agent, a handful of patterns cover almost every case. Vol. 01 sketched three; here is the fuller set, with the trade-offs that decide between them.

Most common
Supervisor (orchestrator–worker)

A lead agent decomposes the goal, delegates to specialist workers, and synthesises their results. Flexible and powerful; the lead is a single point of coordination and failure.

Predictable
Sequential pipeline

Agents run in a fixed order, each refining the last's output. Easy to govern and reason about; inflexible when the task needs branching.

Fast
Parallel + merge

Several agents work simultaneously on parts; a merger combines them. Good for breadth and speed; needs careful handling of conflicting outputs.

Layered
Hierarchical

Supervisors of supervisors for large problems. Scales scope but multiplies coordination cost and latency — use sparingly.

Real systems combine these: a supervisor whose workers are pipelines, a pipeline with a parallel stage in the middle. The patterns are building blocks, not rival architectures — start with the simplest that fits and compose upward only as needed.

Part B · Coordination
03Part B · Coordination

The control plane & state

Whatever the pattern, something must manage the agents: route work, hold shared state, sequence steps, and keep a coherent record. That coordination layer is the control plane, and it is where the determinism and the auditability of a multi-agent system live.

The control plane answers the questions the agents themselves can't be trusted to: which agent runs next, what each is allowed to do, what state is shared versus private, how conflicts are resolved, and when the whole thing stops. Crucially, it is also where you record what the system did — not just each agent in isolation — which is essential for both debugging and the audit trail a regulated deployment requires.

Why it matters

Agents are probabilistic; the control plane is where you put the guarantees. Order, permissions, limits, and logging belong in this deterministic layer, not in the agents' prompts. A multi-agent system without a real control plane is a collection of agents hoping to cooperate.

04Part B · Coordination

Handoffs & communication

The hard part of multi-agent systems is rarely the individual agents — it is the seams between them. A handoff is one agent passing control and context to another, and how you design these seams decides whether the system is coherent or chaotic.

Two recurring problems. Context loss: the receiving agent needs enough of what came before to act sensibly, but not the whole bloated history (Vol. 03 §8) — so handoffs require deliberate summarisation of what matters. Telephone-game drift: each handoff is a chance to distort or lose information, and errors compound across a chain just as they do across steps (Vol. 01 §9). Clear, structured handoff contracts — exactly what each agent receives and returns — contain both.

Watch out

Free-form agent-to-agent chatter is expensive and lossy. Structured handoffs (defined inputs and outputs, like the structured output of Vol. 03 §5) beat open conversation: cheaper, more reliable, and far easier to debug and audit when something goes wrong.

05Part B · Coordination

Agent-to-agent protocols

Just as MCP standardised the agent-to-tool boundary (Vol. 04 §4), emerging agent-to-agent (A2A) protocols aim to standardise how agents discover and communicate with each other — including agents built by different teams or vendors.

The vision is interoperability: an agent advertises its capabilities, and other agents can delegate to it over a common protocol without bespoke integration. This is early and evolving through 2025–26, but the direction is clear and mirrors how software moved from point-to-point integrations to standard interfaces. For an enterprise, the value is the same as MCP's: standard boundaries are where shared controls — identity, permissions, logging — can be placed once.

Watch out

Every agent another agent can call is a new trust boundary and a new attack surface. Cross-organisation agent communication amplifies the prompt-injection and excessive-agency risks of Vol. 04 §7 — an agent you delegate to can be compromised, or can be the compromise. Treat A2A links as supply-chain decisions (Vol. 10 §6).

Part C · Risks
06Part C · Risks

Agent sprawl & cost

The defining failure mode of multi-agent enthusiasm is sprawl: more and more agents, each plausible on its own, collectively producing a system no one fully understands, costs too much, and runs too slowly.

The economics are unforgiving. Each agent is model calls; a supervisor coordinating five workers, each looping several times, can be dozens of calls — and dozens of times the cost and latency of a single agent — for a task that one good agent might have handled. Beyond cost, sprawl erodes comprehensibility: ownership blurs, failures become hard to localise, and the audit story gets murky. Capability does not scale with agent count; past a point it declines as coordination overhead dominates.

Discipline

Treat each agent as a cost and a liability that must justify itself, not a free capability. Periodically ask whether two agents should be one, whether a "specialist" is really just a tool, and whether the parallelism is worth the coordination. The best multi-agent system is the smallest one that does the job.

07Part C · Risks

Failure & debugging

When a multi-agent system goes wrong, the question "which agent caused this?" is often genuinely hard to answer — and that difficulty is itself the central operational risk.

Failures take multi-agent-specific forms: an error in an early agent propagating silently through handoffs; two agents working at cross purposes or undoing each other; a coordination deadlock where each waits on the other; cost or latency blowups from runaway delegation. Diagnosing them requires system-level observability (Vol. 09): a trace that spans all agents and the control plane, showing every handoff, decision, and tool call in order — not just per-agent logs that miss the interactions where the failure actually lives.

Watch out

You cannot debug, govern, or trust a multi-agent system you can't fully trace. If adding agents outpaces your ability to observe them end-to-end, you have built something you can demo but not safely run — the gap Vol. 01 §13 warned against, multiplied.

08Part C · Risks

Multi-agent in financial crime

Financial-crime workflows are a natural fit for the supervisor pattern, because the work genuinely decomposes into distinct specialisms — which is exactly the test from §1.

A realistic configuration (expanded in Vols. 13–14): a lead investigation agent coordinates a screening specialist (sanctions, PEP, adverse media), a transaction-analysis specialist (patterns, typologies, network links), and a narrative specialist (drafting the disposition or SAR from the assembled evidence) — each with its own tools and focus, the lead synthesising and routing the consequential decision to a human. The separation also helps governance: distinct agents with distinct data access make least privilege (Vol. 04 §8) cleaner to enforce and the audit trail clearer.

Watch out

In a regulated setting the multi-agent system's complexity is itself a model-risk and auditability concern (Vol. 11–12). Every handoff and decision must be reconstructable, accountability for the outcome stays with a named human, and the system's behaviour must be evaluable as a whole — not just agent by agent. Complexity you can't explain is complexity you can't deploy.

Glossary

An emerging standard for agents to discover and communicate with each other.
Control plane
The coordination layer managing routing, shared state, sequencing, limits, and logging across agents.
Handoff
One agent passing control and relevant context to another.
Hierarchical pattern
Supervisors coordinating other supervisors, for large multi-stage problems.
Parallel + merge
Agents working simultaneously on parts of a task whose results are then combined.
Sequential pipeline
Agents running in fixed order, each refining the previous output.
Sprawl
The proliferation of agents beyond what the task needs, raising cost and eroding comprehensibility.
Supervisor (orchestrator–worker)
A lead agent that decomposes a goal, delegates to specialists, and synthesises results.
System-level observability
Tracing that spans all agents and the control plane, capturing their interactions.
