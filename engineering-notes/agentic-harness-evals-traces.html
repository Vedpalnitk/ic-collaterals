Chapter I · Foundations

What is an agentic harness?

A definition. Five responsibilities of the harness. Why the model alone is not the system.

§ 01 · Foundations

The model is the engine. The harness is everything else.

A large language model, on its own, is a function: text in, text out. It doesn't take actions, it doesn't iterate, it doesn't recover from mistakes. Turning an LLM into something that can do useful multi-step work in production is the job of a harness.

If the model is the engine, the harness is the chassis, the steering, the brakes, the instrument panel — everything that lets a competent driver actually take it down a road safely.

Concretely, a harness owns five things:

  • The loop. The repeated cycle of reasoning, acting, observing, and continuing. The model produces one thought at a time; the harness keeps the loop going until the goal is reached, the budget is spent, or something breaks.
  • Tool access. A defined set of operations the agent can call — read a database, send an email, write to a file system, query a vector store. Tools are how the agent touches the world.
  • State. What the agent remembers between steps — what it sees, what it forgets, what is persisted across sessions, what is scoped to a single run.
  • Control flow. When to stop. When to retry. When to escalate to a human. When to defer to a fallback. The hard-coded logic that surrounds the soft-coded reasoning.
  • Observability. Every step the agent takes is logged, replayable, and scoreable. Without this, you have a black box you cannot debug, evaluate, or defend in front of a regulator.

The model is the engine. The harness is the chassis, the steering, the brakes, the instrument panel. A mental model

The temptation when starting out is to skip the harness — wire a model directly to a chat interface, give it a couple of tools, and call it an agent. This works for demos. It does not survive contact with reality.

Production-grade systems consistently fail at one of three places: the loop runs away (cost or time), tools fail in unexpected ways (and the agent doesn't recover), or nobody can figure out why the agent did what it did (and the team can't iterate). All three are harness problems, not model problems.

Note

What the harness is not.

The harness is not the agent's intelligence. The model is. The harness is the rules of the road — what's allowed, what's not, what to do when something goes sideways. Confusing the two leads to over-engineered control flow that fights the model, or under-engineered systems that crash on the first edge case.

Chapter II · Foundations

The agent loop, conceptually.

Prompt → reason → act → observe → repeat. When the loop ends — and why everything depends on it.

§ 02 · Foundations

Every agent is the same loop, dressed differently.

Underneath the framework branding — ReAct, Plan-and-Execute, OpenAI Assistants, Anthropic's tool-use loop — the fundamental shape of every agentic system is the same. Once you see it, every "agent framework" reads as a variation on a single theme.

The shape:

Pseudocodestate = initial_state(user_request)

while not done(state):
    thought, action = model.next_step(state)
    state.append(thought)

    if action.kind == "final_answer":
        return action.payload

    observation = tool.execute(action)
    state.append(observation)

    if exceeded_budget(state):
        return escalate(state)

raise BudgetExhausted(state)

That is it. Three primitives: the model produces a thought and an action, the harness executes the action, the result feeds back into the next step. The loop ends one of four ways: the model emits a final answer, the agent runs out of budget (steps, time, dollars), an error trips a fallback, or an external stop condition fires.

The model owns the reasoning. The harness owns the loop.

This separation matters. The model is non-deterministic — give it the same input twice and it might produce different outputs, especially in long-horizon agentic runs. The surrounding logic must be deterministic: when the loop terminates, when retries happen, when a tool result is sanitized before being shown to the model.

If you let the model drive the loop control (e.g., "decide for yourself when to stop"), you end up with agents that won't stop, agents that loop on themselves, or agents that quietly give up. The control layer must be code, not prompt.

The model is non-deterministic. The harness must not be. A principle

What gets fed back to the model

Not everything that comes out of a tool should be shown to the model verbatim. A search API might return ten kilobytes of HTML; the agent doesn't need ten kilobytes — it needs the relevant chunk. A database query might return ten thousand rows; the model can't reason about ten thousand rows.

The harness sits between the world and the model and decides what the model sees. This is one of the highest-leverage parts of an agentic system to get right, and one of the easiest to get wrong by skipping.

In practice

The context is a state, not a transcript.

A common early mistake is to feed the model the entire transcript of every step, every observation, every reasoning trace. This works at small scale and breaks at any real size. Mature harnesses summarize, prune, and reshape the state that goes back into the model — keeping only what is necessary for the next step.

Chapter III · Evals

Why evals matter.

The demo-to-production gap, and why evals are the only artifact in your stack that survives a model swap.

§ 03 · Evals

You cannot ship what you cannot measure.

Demo-to-production is a wide gap in agentic systems — Gartner's 2024 research put the GenAI project-abandonment rate at over thirty percent by end of 2025.¹ The teams that close the gap are not the ones with better models. They are the ones with better evals.

Three properties of LLM systems make evaluation non-optional:

  • Models are non-deterministic. Identical prompts can produce different outputs. A single happy-path test means nothing.
  • Models drift. The frontier model you build against today will be deprecated, retrained, or augmented in months. Behavior shifts even when the API stays the same.
  • Edge cases dominate. The eighty percent you saw in the demo is not the hundred percent you ship.

An eval is a measurable scorecard for an agent: a set of test cases with expected behaviors, run continuously, scored automatically where possible and by humans where necessary. It is the only artifact in your stack that survives a model swap. The prompt changes; the harness changes; the tools change; the eval set stays — and grows.

Build evals, not POCs. A discipline

Five kinds of evals — what to use and when

  • Pass / fail. The agent produces a deterministic output you can compare exactly. Good for: "did it call this tool?", "did it return this value?", "did it route to this branch?" Cheap, fast, brittle.
  • Rubric. A checklist of criteria scored independently. Good for long-form output: "was the summary factually accurate? Was it concise? Did it cover the three main points?" More work to set up; more signal.
  • LLM-as-judge. A separate model scores the output against the rubric. Scalable, surprisingly good, vulnerable to model-specific quirks. Always validate with human eval on a sample.
  • Human eval. A subject-matter expert scores a sampled batch. Ground truth. Expensive. Use it to calibrate the cheaper evals.
  • Regression. Compare the current output against a historical baseline. Catches "the model got worse" silently. Cheap if you have the historical traces.

The right portfolio for any non-trivial agent is usually all five, weighted toward what is cheapest and most automatable. Pass/fail and regression cover the floor; LLM-as-judge handles the bulk; human eval is the calibration anchor.

A rule of thumb

If you cannot put a number on it, you cannot ship it.

For every claim you make about your agent — "it answers correctly 90% of the time", "it doesn't hallucinate citations", "it never exceeds the policy bounds" — there should be an eval that produces a number, run every time the model or the harness changes. Claims without evals are vibes.

Chapter IV · Evals

Writing your first eval.

The anatomy of a test case. A minimal eval harness in twenty lines of Python. Where to grow from there.

§ 04 · Evals

An eval case is data + an expectation + a scorer.

Every eval framework — Braintrust, LangSmith, Promptfoo, OpenAI Evals, Anthropic's built-in evals — ships the same three primitives. Understanding the primitives lets you read any of them, and gives you a way to build your own when none of them fit.

The three primitives
Pythonfrom dataclasses import dataclass
from typing import Callable, Any

@dataclass
class EvalCase:
    name: str           # human-readable identifier
    input: dict         # what you feed the agent
    expected: dict      # what you expect (output, tool calls, behavior)
    scorer: Callable    # function that produces a score
    tags: list[str]     # for slicing dashboards by capability

A minimal eval runner — twenty lines, no dependencies:

Pythondef run_eval(agent, cases: list[EvalCase]) -> dict:
    results = []
    for case in cases:
        try:
            actual = agent.run(case.input)
            score = case.scorer(case.expected, actual)
            results.append({
                "case": case.name,
                "tags": case.tags,
                "score": score,
                "actual": actual,
                "passed": score >= 1.0,
            })
        except Exception as exc:
            results.append({
                "case": case.name,
                "tags": case.tags,
                "score": 0.0,
                "actual": None,
                "error": str(exc),
                "passed": False,
            })
    return summarise(results)

A first scorer

The scorer is where most of the design work lives. A pass/fail scorer for tool routing:

Pythondef called_expected_tool(expected: dict, actual: dict) -> float:
    """Score 1.0 if the agent called the tool we expected, 0.0 otherwise."""
    expected_tool = expected["tool"]
    actual_tools = [step.tool for step in actual["trace"]
                    if step.kind == "tool_call"]
    return 1.0 if expected_tool in actual_tools else 0.0

And a rubric scorer using an LLM-as-judge:

PythonRUBRIC_PROMPT = """
You are evaluating an agent's output. Score each criterion 0-1.

Output: {actual}
Expected criteria: {criteria}

Return JSON: {{"factual": 0-1, "complete": 0-1, "concise": 0-1}}
"""

def rubric_judge(expected: dict, actual: dict) -> float:
    judge_model = anthropic.Client()
    response = judge_model.messages.create(
        model="claude-haiku-4-5-20251001",
        messages=[{"role": "user",
                   "content": RUBRIC_PROMPT.format(
                       actual=actual["text"],
                       criteria=expected["criteria"])}],
    )
    scores = json.loads(response.content[0].text)
    return mean(scores.values())

This is sufficient to get started. Once you have a few dozen cases running on every change, you'll feel the difference immediately — broken changes are caught before they ship, regressions become visible, and "did the new model help?" is a number not an opinion.

Where to grow

The next steps from a working eval harness, in roughly the right order:

  1. Parallelize. Eval cases are embarrassingly parallel. Run them concurrently.
  2. Persist. Store results and the full agent trace for every run, indexed by case + commit + model version.
  3. Tag. Categorize cases by capability (tool-use, reasoning, refusal, formatting) so dashboards slice meaningfully.
  4. Gate. No deploy if a critical-tag case is failing. No model swap if the regression set drops below threshold.
  5. Adopt a tool. Once you understand what you need, Braintrust or LangSmith save you a lot of the next steps for free.

Important

Capture failing production runs as eval cases.

The most valuable eval cases are not the ones you imagine ahead of time — they are the ones that show up in production and surprise you. Every failure, every escalation, every user complaint becomes a candidate eval case. Operationalize the move from "trace of a real failure" → "regression test in the eval set." This is how the eval portfolio becomes a real asset.

Chapter V · Traces

Why traces matter.

An agent run is a tree, not a line. Without traces you have a black box that returns an answer and tells you nothing about how it got there.

§ 05 · Traces

If evals are the scorecard, traces are the X-ray.

In a single-shot LLM application, debugging is easy: you have the prompt, you have the response. In an agentic application, you have a tree. The model reasoned, the harness called a tool, the tool returned, the model reasoned again, called another tool, returned, made a decision, and so on. A single user request can span ten to a hundred model calls and just as many tool invocations.

Without traces, you see only the final answer. The agent reports: "the booking was confirmed." You do not see that it tried three flights, got rate-limited on the second, timed out on the third, and fell through to the first — which turned out to be the wrong day.

Things you cannot do without traces:

  • Debug. A user complaint about output X has no thread back to the steps that produced X.
  • Measure cost. You can sum total tokens, but you cannot tell which step of which workflow is the expensive one.
  • Reproduce. "It worked on my machine" is a phrase that already had a bad reputation; in agentic systems it is fatal.
  • Convert failures to evals. The trace of a failing production run is what you turn into a new eval case. No trace, no case.
  • Defend the system. A regulator, a customer, a board member asks "why did the agent do X?" — the answer is in the trace, or there is no answer.

The shape of a trace

A trace is a structured record of every step the agent took. It is a tree (sometimes a DAG): the top-level span is "user request"; nested below are model calls, tool executions, retries, sub-agent invocations, scoring runs. Each span has attributes — model name, token count, latency, cost, output snippet — and parent/child relationships that preserve the structure.

If you have used distributed tracing for microservices (Jaeger, Zipkin, Honeycomb), this is exactly the same model. The vocabulary is portable; the concepts transfer directly.

Operationally

The trace is the debugger.

The single most productive workflow change for agentic teams is: every time someone reports a bad output, the first thing they do is open the trace. Not the logs, not the prompt, not the model. The trace. If your team can't find a trace for a given run in under thirty seconds, your tracing setup is the bottleneck.

Chapter VI · Traces

Instrumenting traces.

OpenTelemetry, spans, attributes. What to capture, what to sample, what every production agent stack ends up looking like.

§ 06 · Traces

OpenTelemetry is the lingua franca. Spans are the unit.

The current standard for distributed tracing is OpenTelemetry (OTel) — the same observability layer used for microservices. The OTel concept of a span maps cleanly to an agent step. Every major LLM SDK now ships with OTel hooks; every major observability vendor ingests OTel traces.

The data model in 30 seconds
  • Trace. One end-to-end agent run, identified by a trace ID.
  • Span. One unit of work inside the trace: an LLM call, a tool execution, a scoring run.
  • Attributes. Structured key/value data attached to each span — model name, token counts, latency, prompt fragment, response snippet.
  • Parent/child. Spans nest. A model call that triggers a tool call has the tool span as its child.

Minimal manual instrumentation

Twenty lines of Python that wrap an LLM call into an OTel span:

Pythonfrom opentelemetry import trace
tracer = trace.get_tracer("agent")

def call_llm(prompt: str, model: str = "claude-opus-4-7") -> str:
    with tracer.start_as_current_span("llm.call") as span:
        span.set_attribute("llm.model", model)
        span.set_attribute("llm.prompt_length", len(prompt))
        span.set_attribute("llm.prompt_preview", prompt[:200])

        response = llm_client.complete(prompt=prompt, model=model)

        span.set_attribute("llm.response_length", len(response.text))
        span.set_attribute("llm.input_tokens", response.usage.input_tokens)
        span.set_attribute("llm.output_tokens", response.usage.output_tokens)
        span.set_attribute("llm.latency_ms", response.latency_ms)
        return response.text

Do that around every LLM call, every tool call, every scoring run, and you have a complete trace. Nested calls — a model call that triggers a tool that triggers a sub-call — nest automatically because OTel maintains the current span on the context.

Automatic instrumentation

You generally do not want to write this by hand. Most LLM SDKs ship OTel instrumentation built in:

Python# Anthropic SDK
from anthropic import Anthropic
from openinference.instrumentation.anthropic import AnthropicInstrumentor

AnthropicInstrumentor().instrument()

client = Anthropic()
# Every client.messages.create(...) is now traced automatically.

The same exists for OpenAI, LangChain, LlamaIndex, OpenAI Assistants. Enable it once and every call is captured.

What attributes to set

OpenTelemetry has published a draft of semantic conventions for GenAI — a standard set of attribute names so traces are interoperable across vendors. Use them. The important ones:

  • gen_ai.system — provider (anthropic, openai, etc.)
  • gen_ai.request.model — model name
  • gen_ai.usage.input_tokens, .output_tokens
  • gen_ai.response.finish_reasons — why generation stopped
  • gen_ai.prompt and gen_ai.completion — prompts and responses (consider sampling for cost and privacy)

What to sample and what to capture

Capturing the full prompt and response on every call gets expensive in storage and presents a privacy concern. A reasonable default in production:

  • Always capture metadata — model, tokens, latency, cost. Cheap. High signal.
  • Sample prompts and responses — capture 1-10% of full payloads. Adjust for sensitive flows.
  • Always capture failures. If an eval fails or a tool errors, the full payload goes to storage regardless of sample rate.
  • Capture everything in dev. Storage is cheap; debugging time is expensive.

Sample the metadata never; sample the payload always. Tracing rule of thumb

Cross-cutting

Connect traces to evals.

Every eval run should produce a trace; every production trace that fails should be capturable as an eval case. The data model on both sides — span IDs, attributes — is the same. The teams that wire this together early get the feedback loop in chapter VII for free; the teams that don't end up rebuilding it twice.

Chapter VII · Practice

The feedback loop.

From a failing trace to a new eval to a green deploy. The discipline that turns "we have AI" into "we ship AI."

§ 07 · Practice

Trace fails → eval added → fix → green eval → ship.

This is where the discipline shows up. Every team that has successfully scaled an agentic system runs some version of this loop. Every team that has stalled is the team that did not.

The cycle, step by step:

  1. A user reports a bad output, or an eval starts failing in CI, or a production monitor flags an anomaly.
  2. An engineer opens the trace for the affected run. They find the step where the agent went wrong — a misrouted tool call, a hallucinated citation, a missed escalation, an unsanitized observation that confused the next model call.
  3. The engineer captures the failing input as a new eval case in the regression set. Tag it. Attach the trace.
  4. They fix the root cause — adjust the prompt, refine the harness, sanitize the tool output, add a guardrail.
  5. The new eval case runs and passes. The regression set as a whole still passes.
  6. Ship.

The eval set grows monotonically. Every fix adds a regression test. Every model swap is gated on the eval set passing. After six months, the team has an eval portfolio that captures every weird production case they have ever seen — which is exactly the asset you want when the next frontier model arrives and you need to decide whether to switch.

The eval set is a durable record of every mistake your agent has ever made — and is therefore the most valuable asset you build. Why this loop matters

What goes wrong without the loop

The failure mode is recognizable. The team ships an agent. It works in demo. Production users find edge cases. The team plays whack-a-mole: fix one bug, another appears. There is no regression set, so old bugs come back. Six months in, nobody is sure whether the latest fix made things better or worse. Confidence in the system erodes. The agent gets quietly retired.

Versus: the team that builds the eval portfolio early. Every weird production case is captured. Every fix is a regression test. The team can answer "is this version better than the last version" with a number, not a feeling. The agent gets better over time, not worse.

Tools that close the loop

The platforms that do this best — Braintrust, LangSmith, Arize, Langfuse — all share a feature: turn this trace into an eval case, one click. If your stack does not support this gesture cheaply, the loop will be slow, and the team will skip it under pressure. Make the gesture cheap.

A useful litmus test

From production trace to regression test in under 5 minutes.

If a junior engineer can go from "user reported a bad output" to "the failing case is now an eval that will run on every change" in under five minutes, your loop is healthy. If it takes a day, your loop is broken — and your agent will not get reliably better over time.

Chapter VIII · Practice

A reference architecture.

What a production agentic system looks like end-to-end, with evals and traces threaded through every layer.

§ 08 · Practice

Evals and traces are not layers — they cross the stack.

The shape that real production agentic systems converge on is recognizable: a request comes in, an orchestrator runs the agent loop, the agent calls models and tools, results come back, the user gets an answer. What is less obvious until you have built one: the eval pipeline and the trace pipeline are orthogonal to the stack. They cut across every layer.

What this diagram is meant to argue

If you put the eval and trace pipelines on the left — as a layer in the stack — you get a system where observability and evaluation are an afterthought, scoped to one team or one component. If you put them on the right — as cross-cutting infrastructure — you get a system where every layer emits traces and every output is a candidate eval, and the discipline lives in the architecture rather than in someone's head.

The trace pipeline and the eval pipeline are organizationally different (one is "observability"; the other is "quality"), but technically they share a model — a span, attributes, a trace ID. Most real teams end up with the same vendor or the same in-house tooling for both.

The minimum production setup

  • Tracing: OTel auto-instrumentation on every LLM SDK + one trace backend (LangSmith, Arize, Langfuse, or Honeycomb). Sample 100% of failures, 1-10% of successes.
  • Evals: A regression set running on every commit, gating deploys. A small rubric-based eval set running on every model change. A "convert this trace into a case" workflow that takes thirty seconds.
  • Dashboards: Per-capability eval pass-rates over time. Cost-per-request. Latency P50/P95/P99. Error rate per tool.
  • Alerting: Page when the regression set drops below threshold. Page when error rate on a critical tool spikes.

That is the floor. It is also the ceiling for many teams — because once this is in place, building the next agentic capability becomes a normal engineering exercise rather than a research project.

Glossary, tools & reading.

A working glossary, a recommended toolset, and a reading list to take this further.

Glossary

The vocabulary used in this guide.

Agent
An LLM-based system that takes actions over multiple steps to achieve a goal. Distinguished from a single-shot LLM application by having state, tool access, and a control loop.
Harness
The runtime infrastructure around an LLM that turns it into an agent — the loop, tool access, state, control flow, and observability.
Loop
The repeated cycle of reasoning, acting, and observing that defines every agentic system. The model produces one step at a time; the harness keeps the loop going.
Tool
An external operation an agent can invoke. Concretely, a function the harness exposes to the model with a description, a schema, and an implementation.
Eval
A measurable test case for an agent's behavior. An eval consists of an input, an expected output or behavior, and a scorer that produces a number.
Eval portfolio / eval set
The full collection of eval cases for an agent, accumulated over time. Functions as the regression suite for the system and is the most durable artifact across model changes.
Scorer
The function that decides whether an eval case passes. Ranges from a simple equality check to a multi-criterion LLM-as-judge rubric.
LLM-as-judge
An evaluation technique where a (typically smaller, cheaper) model scores the output of the primary model against a rubric. Scales human-style judgment but requires periodic human calibration.
Trace
A structured record of every step an agent took during a single run, captured as a tree of spans with attributes and parent/child relationships.
Span
A single unit of work within a trace — typically a model call, a tool execution, or a scoring run. The OpenTelemetry concept of a span maps cleanly to an agent step.
OpenTelemetry (OTel)
The vendor-neutral standard for distributed tracing. The same data model that powers microservices observability now powers LLM observability.
Control plane
The orchestration layer that owns the agent loop, the routing between agents, the budget enforcement, and the escalation paths.
Feedback loop (in this guide)
The discipline of converting production failures into eval cases, which become regression tests, which gate future deploys.
