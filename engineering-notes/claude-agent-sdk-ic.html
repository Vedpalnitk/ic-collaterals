Chapter I · Definition

What it is.

The runtime that powers Claude Code, packaged as a library you can embed in your own applications.

§ 01 · Definition

The model is the engine. The SDK is everything else.

The Claude Agent SDK is the runtime that powers Claude Code — packaged as a library you can embed in your own applications. It is the loop, the tools, the permissions, and the plumbing. It is what stands between a single API call and a self-directed program that opens files, runs commands, and decides what to do next.

Anthropic SDK lets you talk to Claude once. The Agent SDK lets Claude drive a loop with tools. The one-line mental model

It ships in Python (pip install claude-agent-sdk) and TypeScript (npm install @anthropic-ai/claude-agent-sdk). Both are thin bindings over the same agent runtime.

A note on lineage

Originally released as the Claude Code SDK.

Renamed in late 2025 once people started building research agents, ops agents, and customer-facing bots with it. Same code, broader name.

Chapter II · Architecture

Three layers, stacked.

Each layer assumes the one below it and adds a coordination concern. You can stop at any layer that solves your problem.

§ 02 · Architecture

The stack.

The Anthropic-built agentic surface is three layers. The Agent SDK sits in the middle: not the raw model, not the multi-agent protocol — the harness that turns a single model into something that can do work.

FIG. 01
L3 A2A — inter-agent Wire protocol so independent agents can discover and call each other over HTTP + JSON-RPC. Agnostic to who built them.
L2 Agent SDK — the loop Runtime that lets Claude call tools, dispatch subagents, manage context, and follow a plan to completion.
L1 Anthropic SDK — the model Raw client. One prompt in, one completion out. Streaming, caching, tool-use schemas. No autonomy.

Chapter III · Architecture

What it constitutes of.

Five layers stacked on top of the raw Claude API. Together they replace the harness you would otherwise build yourself.

§ 03 · Architecture

The anatomy, in five parts.

When you import the SDK, here is what shows up. Each part replaces something you would otherwise build yourself — and most of it you would build badly the first time.

i

The agent loop.

Call Claude → parse tool calls → execute them → feed results back → repeat until the model says "done." Handles context-window management, automatic compaction, retries, and termination. This is the core.

ii

Built-in tools.

Ready-made primitives the model can call. Same implementations Claude Code uses.

ReadWriteEdit BashGlobGrep WebFetchWebSearchTodoWrite
iii

Extension surfaces.

Where your agent diverges from Claude Code. Define your own tools, attach MCP servers, dispatch subagents with isolated context, install skills that load on demand, and hook into lifecycle events.

Custom toolsMCP SubagentsSkills HooksSlash commands
iv

Runtime services.

The plumbing you would re-invent: a granular permission system, session pause/resume/fork, automatic prompt caching, settings hierarchy (global → project → local), and environment-variable inheritance.

v

Deployment helpers.

Not a framework, but enough surface area to wrap the runtime in a CLI, a web service, a Slack bot, or — yes — an A2A server. The loop stays the same; only the I/O changes.

Inheritance

You inherit a mature toolkit; you do not rebuild it.

Every tool, hook, and skill you see in Claude Code is available to your Agent SDK app. That's the practical advantage of the SDK being the same runtime as the product — the parts that have already been polished are yours for free.

Chapter IV · Indications

When to reach for it.

Indications and counter-indications, side by side. Not every problem needs an agentic loop.

§ 04 · Indications

Use vs. don't use.

The Agent SDK is a heavy abstraction. When the problem fits, it saves you weeks of harness-building. When it doesn't, you are carrying weight you don't need.

Use the Agent SDK when —

  • You want an agentic loop, not a single-shot completion
  • The agent needs side effects: files, shell, APIs
  • Tools need to compose — multi-step plans, branching, chains
  • You would rather inherit Claude Code's loop than write your own
  • You want subagent dispatch, MCP, hooks, and skills out of the box

Reach for something else when —

  • You need a single prompt → single completion (Anthropic SDK)
  • You are building a chat UI with no tool calls (Anthropic SDK + streaming)
  • You need provider-agnostic code (LangChain, LlamaIndex)
  • You are doing pure RAG with no agentic behavior (Anthropic SDK + vectors)
  • The runtime must be in a language Anthropic does not ship a binding for

Chapter V · Practice

A minimal Agent SDK app.

Fourteen lines of Python — fully agentic. Same primitives Claude Code uses internally.

§ 05 · Practice

A working example.

The shape of every Agent SDK program is the same: pick the tools you want available, hand the model a goal, let the loop run. The example below reads a codebase, finds every TODO comment, and summarises them in priority order.

todo_summarizer.py Python · Agent SDK ~ 14 lines 
# pip install claude-agent-sdkfrom claude_agent_sdk import query, ClaudeAgentOptionsimport asyncioasync def main():    options = ClaudeAgentOptions(        system_prompt="You are a code archaeologist. Be terse.",        allowed_tools=["Read", "Grep", "Glob"],   # read-only — safe        permission_mode="acceptEdits",        cwd="./my-project",    )    prompt = "Find every TODO/FIXME in this repo. " \             "Group by file, rank by apparent severity, return markdown."    async for message in query(prompt=prompt, options=options):        print(message)asyncio.run(main())

Behind those fourteen lines: an entire agent loop. Claude inspects the repo with Glob, narrows with Grep, opens individual files with Read, reasons about severity, and emits a markdown report — all without you writing the control flow.

What this does

A working agent, not a toy.

Reads a directory, finds every TODO comment in the codebase, and summarises them in priority order. Fully agentic — multi-step, tool-using, with proper permission scoping. Same primitives Claude Code uses internally.

Chapter VII · Composition

How it composes.

The Agent SDK builds individual agents. A2A lets agents call each other. Together, you get a fleet.

§ 07 · Composition

Three patterns for composition.

Agent SDK and A2A are not competitors. They compose. There are three common patterns for putting them together — pick by where you want the orchestration to live.

SDK agent, exposed via A2A

Wrap a Claude Agent SDK app inside an AgentExecutor. Now other agents discover it through its Agent Card and call it like any other A2A service.

SDK agent that consumes A2A

Add a custom tool that calls out to specialist A2A agents. Claude now orchestrates a fleet — picking the right specialist for each subtask.

Pure A2A, no SDK

Just the wire protocol — good for understanding the contract, but not what you would ship to production for any non-trivial agent.

Orthogonal layers

Not competitors — collaborators.

Agent SDK gives you the runtime for one agent. A2A gives you a way for agents to find and call each other. You can use both, either, or neither. The interesting systems use both.
